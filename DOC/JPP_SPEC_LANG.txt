================================================================================
                  SPECIFICATIONS DU LANGAGE DE PROGRAMMATION J++
================================================================================

Document de reference pour l'integration dans le systeme d'aide DEVENV.HLP.
Dans ce document, J++ designe Microsoft Visual J++ 1.0/1.1 et 6.0, et non
Java moderne, J#, C++/CLI, ni un langage nomme "Java++" par un autre editeur.

Visual J++ est une implementation historique du langage Java accompagnee du
compilateur JVC, de la Microsoft Virtual Machine for Java (MSJVM), de
bibliotheques Microsoft et, dans Visual J++ 6.0, des Windows Foundation
Classes (WFC). Son noyau syntaxique correspond essentiellement a Java 1.1.

IMPORTANT : J++ est abandonne. La MSJVM, les WFC et les outils Visual J++ ne
doivent pas servir a de nouveaux logiciels. Cette specification documente le
langage pour la lecture, la maintenance et la migration de sources anciens.


================================================================================
1. PRESENTATION ET PERIMETRE
================================================================================

Visual J++ fut livre par Microsoft avec Visual Studio 97 puis Visual Studio
6.0. Il permettait de produire du bytecode Java execute par la MSJVM, et de
creer notamment des applications console, des applets, des applications
Windows WFC et des composants interoperables avec COM/Win32.

Le terme J++ recouvre trois couches qu'il faut distinguer :

  1. le langage Java 1.1 : classes, interfaces, exceptions, threads, etc.;
  2. les extensions du compilateur Microsoft : delegates, directives en
     commentaire, compilation conditionnelle et services d'interoperabilite;
  3. les bibliotheques Microsoft : com.ms.*, WFC, J/Direct et acces COM.

Une classe WFC n'est donc pas une construction syntaxique du langage. De meme,
un programme peut etre syntaxiquement Java tout en etant non portable parce
qu'il importe com.ms.wfc.* ou appelle Win32.

J++ reste anterieur a Java 2 et ne comprend pas les ajouts ulterieurs :
assert, strictfp, generics, annotations, enum, foreach, varargs, autoboxing,
imports statiques, lambdas, methodes default, modules, records, text blocks,
classes sealed, pattern matching et expressions switch.

1.1 Resume comparatif Java / J++

Le tableau suivant compare Java standard a Microsoft Visual J++ dans son
environnement historique. "Java" designe ici le langage et les API standard,
sans bibliotheque native ajoutee par un fournisseur.

  Fonctionnalite                    Java                  J++
  --------------------------------  --------------------  ----------------------
  Portable                          Oui                   Non
  Conforme aux specifications Java Oui                   Partiellement
  API Windows integree              Non                   Oui
  COM / ActiveX integre             Non                   Oui
  Registre Windows integre          Non                   Oui
  Delegates                         Non (a l'epoque)      Oui
  Events integres au modele MS      Non                   Oui
  Proprietes de composants          Non (*)               Oui (*)
  Plates-formes                     Toutes                Windows
  Etat actuel                       Developpement actif    Abandonne

(*) La ligne "Proprietes" demande une precision. Visual J++ et son IDE
exposent les proprietes de JavaBeans, AFC, WFC et COM comme des membres de
composants. J++ ne reconnait toutefois pas la declaration automatique C# :

  public int Age { get; set; }      // C#, pas Visual J++ 6.0

Dans une classe Java ou J++, une propriete JavaBean est implementee par des
methodes telles que getAge() et setAge(). Voir la section 15.3. Ainsi, "Oui"
dans la colonne J++ qualifie le modele composant et son integration aux outils,
pas une grammaire get/set identique a celle de C#.

Java peut appeler Windows, COM, ActiveX ou le registre par JNI, JNA ou une
autre bibliotheque externe. La valeur "Non" signifie que ces services ne font
pas partie du langage ni de l'API Java standard. A l'inverse, Visual J++ les
integre directement par la MSJVM, J/Direct et les paquets com.ms.*.

"Toutes" signifie que les implementations Java existent sur de nombreuses
plates-formes et que le code respectant les API standard est concu pour etre
portable. Cela ne garantit pas qu'un programme Java utilisant du code natif ou
une API propre a un systeme fonctionne sans adaptation partout.


================================================================================
2. UNITES DE COMPILATION ET PROGRAMME MINIMAL
================================================================================

Avec Microsoft Visual J++, un fichier source porte normalement l'extension
.java. Certains clones, dialectes ou outils tiers appeles J++ utilisent
cependant l'extension .jpp afin de distinguer leurs sources du Java standard.
Cette extension .jpp n'est pas celle du compilateur Microsoft JVC et son emploi
ne garantit pas que le contenu respecte la syntaxe Visual J++ 6.0. Il faut donc
identifier le compilateur vise avant de conclure qu'un fichier .jpp ou .java
appartient au dialecte decrit dans ce document.

Une unite de compilation contient, dans cet ordre :

  [ declaration de package ]
  { declaration d'import }
  { declaration de classe ou d'interface }

Exemple console :

  public class Bonjour {
      public static void main(String[] args) {
          System.out.println("Bonjour depuis J++");
      }
  }

Le point d'entree est une methode public static void main(String[] args).
Le nom du fichier d'une classe publique correspond normalement au nom de la
classe : Bonjour.java.

Exemple avec package :

  package exemple.outils;
  import java.util.Vector;

  public class Demo {
      public static void main(String[] args) {
          Vector v = new Vector();
          v.addElement("un");
      }
  }

Syntaxe abstraite :

  unite          ::= [ package nomQualifie ; ] { importDecl } { typeDecl }
  importDecl     ::= import nomQualifie [ .* ] ;
  typeDecl       ::= classDecl | interfaceDecl | ;
  nomQualifie    ::= identificateur { . identificateur }


================================================================================
3. ELEMENTS LEXICAUX
================================================================================

3.1 Identificateurs

Les identificateurs sont sensibles a la casse. Une lettre, '_' ou '$' peut
commencer un identificateur; les positions suivantes acceptent aussi les
chiffres. Les lettres Unicode reconnues dependent de la version de la MSJVM et
du compilateur. Les noms commencant par '$' restent possibles mais sont a
eviter dans le code applicatif.

3.2 Mots reserves du noyau Java 1.1

  abstract    boolean      break       byte        case
  catch       char         class       const       continue
  default     do           double      else        extends
  final       finally      float       for         goto
  if          implements   import      instanceof  int
  interface   long         native      new         package
  private     protected    public      return      short
  static      super        switch      synchronized this
  throw       throws       transient   try         void
  volatile    while

const et goto sont reserves mais n'ont pas d'instruction associee. true, false
et null sont des litteraux et non des identificateurs.

Les extensions Microsoft peuvent en outre reconnaitre des mots contextuels,
notamment delegate et multicast, lorsque les extensions sont activees. Il est
prudent de ne pas les employer comme identificateurs.

3.3 Commentaires

  // commentaire jusqu'a la fin de ligne
  /* commentaire sur une ou plusieurs lignes */
  /** commentaire de documentation */

Les commentaires de bloc ne s'imbriquent pas. Certaines lignes commencant par
/** @... */ ont un sens special pour les outils Microsoft; voir section 17.

3.4 Litteraux

  entiers decimaux       0  12  2147483647  12L
  entiers octaux         077
  entiers hexadecimaux   0xFF  0XCAFE
  reels                  1.0  .5  6.02e23  3.14F  2.0D
  caracteres             'A'  '\n'  '\u00e9'
  chaines                "texte"  "ligne 1\nligne 2"
  booleens               true  false
  reference nulle        null

Echappements : \b, \t, \n, \f, \r, \", \', \\, sequences octales et
Unicode \uXXXX. Une chaine ne traverse pas directement une fin de ligne.

3.5 Formation des tokens et separateurs

Avant l'analyse grammaticale, le compilateur transforme le flux de caracteres
en tokens. Les espaces, tabulations, fins de ligne et commentaires separent les
tokens mais n'ont autrement pas de signification. Une fin de ligne ne termine
pas une instruction : le point-virgule ou la structure grammaticale le fait.

Le compilateur choisit normalement le token valide le plus long. Ainsi, >>= est
un seul operateur, tandis que > > = represente trois tokens. Deux mots doivent
etre separes s'ils formeraient autrement un identificateur unique.

  separateurs     (  )  {  }  [  ]  ;  ,  .
  operateurs      =  >  <  !  ~  ?  :  ==  <=  >=  !=  &&  ||
                   ++  --  +  -  *  /  &  |  ^  %  <<  >>  >>>
                   +=  -=  *=  /=  &=  |=  ^=  %=  <<=  >>=  >>>=

Le caractere '.' sert a la fois dans un reel, dans un nom qualifie et dans un
acces a un membre. Le contexte permet par exemple de distinguer 1.0, java.io et
objet.champ. Les caracteres # et ` n'ont aucun role lexical standard en J++.

3.6 Identificateurs en detail

Forme simplifiee :

  identificateur ::= lettreJava { lettreJava | chiffreJava }

Une lettreJava comprend les lettres Unicode reconnues, '_' et '$'. La valeur
Unicode exacte acceptee depend des tables de caracteres du vieux compilateur
JVC. Aucun mot reserve, ni true, false ou null, ne peut etre un identificateur.

  age, Age, _age, $temp, nombre2     valides
  2nombre, mon-nom, class, true      invalides

Age et age sont deux noms differents. Les conventions, non imposees par la
grammaire, emploient NomDeClasse pour les types, nomDeMethode pour les methodes
et CHAMP_CONSTANT pour les constantes static final.

3.7 Litteraux entiers en detail

Forme simplifiee :

  decimal     ::= 0 | chiffreNonNul { chiffre }
  octal       ::= 0 chiffreOctal { chiffreOctal }
  hexadecimal ::= 0x chiffreHexa { chiffreHexa }
  entier      ::= ( decimal | octal | hexadecimal ) [ l | L ]

Sans suffixe, le type est int si la valeur est representable; avec L ou l, le
type est long. Le suffixe L majuscule est preferable car l minuscule ressemble
au chiffre 1. Java 1.1/J++ n'accepte ni prefixe binaire 0b ni separateurs '_'.

Un zero initial rend un entier octal : 010 vaut huit, et 08 est invalide. Le
signe moins n'appartient pas au litteral; -1 est l'operateur unaire '-' suivi
du litteral 1. Les cas limites -2147483648 et -9223372036854775808L suivent des
regles speciales de compilation liees a cette representation.

3.8 Litteraux flottants en detail

Forme simplifiee :

  flottant ::= chiffres . [ chiffres ] [ exposant ] [ suffixe ] |
               . chiffres [ exposant ] [ suffixe ] |
               chiffres exposant [ suffixe ] |
               chiffres [ exposant ] suffixe
  exposant ::= ( e | E ) [ + | - ] chiffres
  suffixe  ::= f | F | d | D

Sans F/f, un litteral flottant est de type double; D/d le confirme. F/f donne
un float. Les formes hexadecimales flottantes et les separateurs '_' sont
posterieurs a J++. Un litteral non nul trop petit ou trop grand pour son type
est une erreur de compilation plutot qu'une conversion silencieuse en zero ou
en infini.

3.9 Caracteres, chaines et Unicode

Un litteral char contient exactement un caractere ou une sequence
d'echappement entre apostrophes. Un litteral String contient zero ou plusieurs
caracteres/echappements entre guillemets. Les echappements octaux utilisent de
un a trois chiffres avec les limites imposees par la premiere valeur octale.

Les echappements Unicode \uXXXX sont traites tres tot, avant le decoupage des
tokens. Ils peuvent donc introduire un caractere syntaxique, y compris une fin
de ligne; cette particularite exige de la prudence dans les commentaires. Deux
unites char peuvent etre necessaires pour representer un caractere Unicode hors
du plan multilingue de base.

Les litteraux String identiques sont internes par la JVM et peuvent partager
la meme reference. Le programme doit cependant utiliser equals() pour comparer
le contenu et non ==. Il n'existe ni chaine brute, ni interpolation, ni bloc de
texte multilignes dans J++ 6.0.

3.10 Reference de chacun des mots reserves

Les formes ci-dessous utilisent les conventions suivantes :

  [ element ]       element facultatif
  { element }       zero ou plusieurs repetitions
  A | B             choix entre A et B
  expression        expression du type exige par le contexte
  instruction       instruction simple ou bloc { ... }

Un mot reserve n'a pas toujours des "parametres" au sens d'une methode. Les
elements places apres le mot sont ses operandes, noms, types, expressions ou
blocs grammaticaux. Leur role est precise pour chaque entree.

abstract

  abstract class Nom { ... }
  [modificateurs] abstract Type methode(parametres) [throws types] ;

Marque une classe incomplete ou une methode sans corps. Une classe abstract ne
peut pas etre instanciee avec new. Une sous-classe concrete doit implementer
les methodes abstraites accessibles. abstract est incompatible avec final et,
pour une methode, avec private, static, native ou synchronized.

boolean

  boolean nom [= expressionBooleenne] ;
  boolean methode(parametres) { ... }

Declare le type primitif dont les seules valeurs sont true et false. Il n'est
pas implicitement convertible en entier; une condition if/while/for exige une
expression boolean.

break

  break ;
  break etiquette ;

Sans etiquette, termine le switch ou la boucle englobante la plus proche. Avec
une etiquette, termine l'instruction etiquetee correspondante. Il ne prend pas
de valeur et ne quitte pas une methode; return remplit ce role.

byte

  byte nom [= expressionEntiereCompatible] ;

Type entier signe sur 8 bits (-128 a 127). Les calculs promeuvent normalement
byte en int; un cast peut donc etre requis pour reutiliser le resultat comme
byte. Une constante int representable peut etre affectee sans cast.

case

  case expressionConstante : instructions

Introduit une branche de switch. L'operande doit etre une constante entiere
compatible avec le type du selecteur et unique dans ce switch. L'execution
continue dans le case suivant tant qu'un break, return ou throw ne l'arrete.

catch

  catch (TypeException nom) { instructions }

Suit un bloc try et capture une exception assignable au type indique. Le
parametre entre parentheses declare une variable locale contenant l'objet
exception. Plusieurs catch sont permis, du type le plus specifique au plus
general; un catch inaccessible est une erreur.

char

  char nom [= caractereOuEntierCompatible] ;

Type primitif non signe sur 16 bits, representant une unite UTF-16. Ses valeurs
vont de '\u0000' a '\uffff'. Les calculs promeuvent char en int.

class

  [modificateurs] class Nom [extends Super]
      [implements Interface {, Interface}] { membres }

Declare une classe. Nom cree le nouveau type; extends choisit au plus une
superclasse; implements donne zero ou plusieurs contrats d'interface. Le corps
contient champs, methodes, constructeurs, initialisateurs et types imbriques.

const

  aucune syntaxe executable

Mot reserve inutilise. J++ le refuse comme identificateur mais ne permet pas de
declarer une constante avec lui. Employer final, souvent static final.

continue

  continue ;
  continue etiquette ;

Passe a l'iteration suivante d'une boucle. Dans for, l'expression de mise a
jour est executee avant le prochain test. L'etiquette doit designer une boucle
englobante, contrairement a break qui peut viser une instruction etiquetee.

default

  default : instructions

Introduit la branche d'un switch utilisee lorsqu'aucun case ne correspond. Un
seul default est permis; sa position dans le bloc switch n'est pas imposee.
J++ historique ne connait pas les methodes default des interfaces modernes.

do

  do instruction while (expressionBooleenne) ;

Execute d'abord l'instruction, puis repete tant que la condition vaut true. Le
corps s'execute donc au moins une fois. Le point-virgule final est obligatoire.

double

  double nom [= expressionNumerique] ;

Type flottant IEEE 754 double precision sur 64 bits. C'est le type par defaut
des litteraux reels. Il accepte NaN, les infinis et les zeros signes produits
par les calculs, selon les regles flottantes de la JVM.

else

  if (condition) instruction else instruction

Fournit la branche executee quand la condition if vaut false. Un else est
associe au if sans else le plus proche, sauf regroupement explicite par blocs.

extends

  class Fille extends Mere { ... }
  interface Fille extends I1, I2 { ... }

Indique l'heritage. Une classe a une seule superclasse directe; une interface
peut etendre plusieurs interfaces. Les types cites doivent etre accessibles et
ne doivent pas produire un cycle d'heritage.

final

  final class Nom { ... }
  final Type variable = valeur ;
  final Type methode(parametres) { ... }

Interdit respectivement la derivation d'une classe, une seconde affectation a
une variable, ou la redefinition d'une methode. final sur une reference ne rend
pas l'objet immuable. Une variable final doit etre definitivement affectee une
fois avant utilisation.

finally

  try { ... } [catch (...) { ... }] finally { nettoyage }

Declare le bloc execute lors de la sortie normale ou exceptionnelle du try et
de ses catch, y compris apres return, break ou continue. Une sortie brutale de
la JVM peut l'empecher. Un return/throw dans finally masque la sortie initiale
et doit etre evite.

float

  float nom [= expressionCompatible] ;

Type flottant IEEE 754 simple precision sur 32 bits. Un litteral reel doit en
general porter F, par exemple 1.5F; 1.5 est un double et exige un cast.

for

  for ([initialisation] ; [condition] ; [miseAJour]) instruction

initialisation est une declaration locale ou une liste d'expressions;
condition doit etre boolean et vaut true si elle est omise; miseAJour est une
liste d'expressions executees apres chaque iteration. Chacune des trois zones
peut etre vide. Il n'existe pas de for-each en J++ 6.0.

goto

  aucune syntaxe executable

Mot reserve inutilise. Aucune instruction goto n'existe. Les transferts
structures disponibles sont break, continue, return et throw.

if

  if (expressionBooleenne) instruction [else instruction]

Execute la premiere instruction si la condition vaut true, sinon la branche
else eventuelle. La condition doit etre boolean : if (1) est invalide.

implements

  class Nom implements I1 [, I2 ...] { ... }

Liste les interfaces promises par une classe. Une classe concrete doit fournir
les methodes d'interface non deja implementees par une superclasse. implements
n'accepte que des types interface.

import

  import paquet.Type ;
  import paquet.* ;

Rend un type utilisable par son nom simple dans l'unite de compilation. La
forme * importe les types directs d'un paquet, pas ses sous-paquets. Elle ne
charge pas la classe et n'importe ni membres ni paquet recursivement.

instanceof

  expressionReference instanceof TypeReference

Produit true si la valeur non nulle peut etre convertie vers le type indique
sans ClassCastException; null produit false. Les types des deux operandes
doivent etre compatibles a la compilation.

int

  int nom [= expressionEntiere] ;

Type entier signe sur 32 bits. C'est le type usuel des calculs entiers, indices
de tableaux et selecteurs switch. Le debordement utilise les 32 bits de poids
faible sans lever d'exception.

interface

  [public | protected | private | static | abstract]
  interface Nom [extends I1, I2 ...] { membres }

Declare un contrat. Les methodes sont implicitement public abstract et sans
corps; les champs sont public static final et doivent etre initialises. Les
modificateurs possibles dependent du fait que l'interface soit de premier
niveau ou imbriquee.

long

  long nom [= expressionEntiereCompatible] ;

Type entier signe sur 64 bits. Un litteral long utilise L, par exemple 10L.
Les decalages d'un long n'emploient que les 6 bits faibles de leur distance.

native

  [modificateurs] native Type methode(parametres) [throws types] ;

Declare une methode implementee hors du bytecode Java. Elle se termine par ';'
et n'a pas de corps. Dans J++, J/Direct, RNI ou la MSJVM assurent la liaison;
les parametres doivent pouvoir etre convertis selon les regles de marshalling.

new

  new TypeClasse(arguments)
  new TypeElement[dimension] { [dimension] } [initialiseur]
  new TypeInterface(arguments) { corpsClasseAnonyme }

Cree un objet ou un tableau et retourne sa reference. Les arguments choisissent
un constructeur compatible. Les dimensions de tableau sont des int non
negatifs; une taille negative leve NegativeArraySizeException.

package

  package nom.qualifie ;

Doit etre la premiere declaration non commentee de l'unite. Il place les types
du fichier dans le paquet indique et influence leur nom complet et leur acces.
En son absence, les types appartiennent au paquet sans nom.

private

  private declarationDeMembre

Limite l'acces a la classe qui declare le membre. Il s'applique aux membres et
types imbriques, pas a une classe de premier niveau Java 1.1. Une methode
private n'est pas redefinie polymorphiquement dans une sous-classe.

protected

  protected declarationDeMembre

Autorise l'acces depuis le meme paquet et, sous certaines regles de receveur,
depuis les sous-classes situees ailleurs. Il s'applique aux membres et types
imbriques, pas a un type de premier niveau.

public

  public declaration

Rend la declaration accessible partout ou son type englobant est accessible.
Une classe ou interface public de premier niveau reside normalement dans un
fichier portant son nom. public ne contourne pas l'inaccessibilite du contenant.

return

  return ;
  return expression ;

Termine la methode ou le constructeur courant. Une methode void et un
constructeur utilisent la forme sans valeur; une methode non void doit fournir
une expression convertible vers son type retour. finally s'execute avant le
transfert effectif.

short

  short nom [= expressionEntiereCompatible] ;

Type entier signe sur 16 bits (-32768 a 32767). Comme byte et char, il est
promu en int pour la plupart des operations arithmetiques.

static

  static Type champ ;
  static Type methode(parametres) { ... }
  static { initialisationDeClasse }
  static class TypeImbrique { ... }

Associe le membre a la classe et non a chaque objet. Une methode static n'a pas
de this et ne peut acceder directement aux membres d'instance. Un bloc static
s'execute une fois lors de l'initialisation de la classe.

super

  super(arguments)              // premier appel d'un constructeur
  super.membre
  super.methode(arguments)

Designe la partie superclasse de l'objet courant. super(...) appelle un
constructeur direct et doit etre la premiere instruction du constructeur.
super.methode force l'implementation de la superclasse immediate.

switch

  switch (expressionEntiere) {
      { case constante : instructions }
      [ default : instructions ]
  }

Evalue une fois son selecteur char, byte, short ou int, puis saute vers le case
correspondant. L'execution continue ensuite sequentiellement. Ni String, ni
enum, ni expression switch moderne ne sont disponibles en J++ 6.0.

synchronized

  synchronized Type methode(parametres) { ... }
  synchronized (expressionReferenceNonNulle) { instructions }

Acquiert un moniteur avant d'executer le corps et le libere a la sortie. Une
methode d'instance verrouille this; une methode static verrouille l'objet Class.
Le bloc verrouille l'objet calcule et leve NullPointerException s'il vaut null.

this

  this
  this.membre
  this(arguments)

Designe l'instance courante. this(...) appelle un autre constructeur de la
meme classe et doit etre la premiere instruction. this est indisponible dans un
contexte static. Il peut lever une ambiguite entre champ et parametre.

throw

  throw expressionThrowable ;

Evalue puis lance un objet assignable a Throwable. Lancer null provoque
NullPointerException. Une exception verifiee doit etre capturee ou annoncee
par throws. L'instruction ne revient pas normalement.

throws

  Type methode(parametres) throws E1 [, E2 ...] { ... }

Annonce les types d'exceptions verifiees qu'une methode ou un constructeur
peut laisser sortir. La liste ne cree ni ne lance une exception et n'est pas
une liste de parametres; throw effectue le lancement concret.

transient

  transient Type champ ;

Indique au mecanisme standard de serialisation de ne pas sauvegarder ce champ.
Il ne s'applique qu'aux champs. Apres deserialisation standard, le champ reprend
sa valeur par defaut, sauf restauration explicite par la classe.

try

  try { instructions }
      catch (Type nom) { instructions } { catch ... }
      [finally { instructions }]
  try { instructions } finally { instructions }

Delimite une region dont les exceptions sont traitees par catch et/ou dont la
sortie est protegee par finally. Au moins un catch ou un finally est requis.
J++ ne connait ni try-with-resources ni multi-catch.

void

  void methode(parametres) { ... }

Indique qu'une methode ne retourne aucune valeur. void n'est pas un type de
variable, d'argument ou d'element de tableau. Une methode void peut employer
return; mais pas return expression;.

volatile

  volatile Type champ ;

Demande que les acces au champ partage respectent la semantique volatile de la
JVM Microsoft de l'epoque. Le modificateur ne s'applique pas aux variables
locales ni methodes. Il ne rend pas atomique une operation composee comme i++.

while

  while (expressionBooleenne) instruction

Teste la condition avant chaque iteration et execute le corps tant qu'elle
vaut true. Si elle est fausse au premier test, le corps ne s'execute jamais.

3.11 Litteraux reserves true, false et null

true et false sont les deux litteraux boolean. Ils s'emploient comme valeurs et
conditions, sans guillemets ni majuscule : True et FALSE seraient de simples
identificateurs. null est la reference nulle et peut etre converti vers tout
type reference, mais vers aucun type primitif. Dereferencer null leve
NullPointerException.

  boolean termine = false;
  String texte = null;
  if (!termine && texte == null) { ... }

3.12 Extensions lexicales Microsoft : delegate et multicast

Lorsque les extensions J++ sont actives, delegate introduit un type d'appel
fortement type. Sa liste entre parentheses decrit les parametres que la methode
cible doit accepter; le type place avant son nom est le type retourne :

  [modificateurs] delegate TypeRetour Nom(Type p1, Type p2) ;

  delegate void Notification(Object source, int code);

Une instance lie une methode de signature compatible :

  Notification n = new Notification(objet.traiter);
  n.invoke(source, 10);

multicast modifie une declaration de delegate afin que l'instance puisse
representer une chaine de plusieurs methodes compatibles :

  [modificateurs] multicast delegate TypeRetour Nom(parametres) ;

Ces deux mots et la syntaxe de reference liee objet.methode ne font pas partie
de Java 1.1. Leur disponibilite depend des options de JVC et du runtime
Microsoft. Les details du modele evenementiel figurent a la section 16.


================================================================================
4. TYPES ET VALEURS
================================================================================

4.1 Classification generale

J++ reprend le systeme de types de Java 1.1. Il est statique : le type de toute
expression est determine a la compilation. Il est aussi fortement type : une
valeur ne peut etre employee comme un autre type que si une conversion definie
par le langage existe.

Deux familles de types existent :

  types primitifs    boolean, byte, short, int, long, char, float, double
  types reference    classes, interfaces et tableaux

Une variable primitive contient directement une valeur. Une variable reference
contient null ou une reference vers un objet. Les objets eux-memes sont crees
sur le tas gere par la machine virtuelle et leur memoire est recuperee par le
ramasse-miettes.

void n'est ni un type primitif de variable ni un type reference. Il indique
seulement qu'une methode ne retourne aucune valeur :

  void afficher() {
      System.out.println("Bonjour");
  }

J++ 6.0 ne possede ni type unsigned general, ni pointeur Java arithmetique, ni
structure valeur, ni enum, ni generics, ni type nullable primitif. Les types
propres a Win32 et COM sont representes par les classes et conventions des
paquets com.ms.*; ils ne modifient pas le systeme de types du noyau.

4.2 Type boolean

  Domaine             true ou false
  Litteraux           true, false
  Valeur par defaut   false
  Classe enveloppe    java.lang.Boolean

La taille de stockage de boolean n'est pas fixee par le langage. boolean n'est
pas un entier : aucune conversion implicite ou explicite standard n'existe
entre boolean et byte/int. Ces exemples sont invalides :

  boolean b = 1;          // invalide
  int n = (int)true;      // invalide

Les conditions de if, while, do et for exigent une expression boolean :

  boolean pret = true;
  if (pret) { ... }

Les operateurs disponibles sont !, &, |, ^, ==, !=, && et ||. Avec boolean,
& et | evaluent toujours leurs deux operandes; && et || peuvent ne pas evaluer
le second operande.

4.3 Types entiers byte, short, int et long

  Type    Bits  Domaine                                  Defaut  Suffixe
  ------  ----  ---------------------------------------  ------  -------
  byte       8  -128 a 127                                   0  aucun
  short     16  -32768 a 32767                               0  aucun
  int       32  -2147483648 a 2147483647                     0  aucun
  long      64  -9223372036854775808 a 9223372036854775807   0L L ou l

Tous ces types sont signes et utilisent une representation binaire en
complement a deux pour les operations definies par la JVM. int est le type
entier usuel : les litteraux sans suffixe et la plupart des calculs sur byte,
short et char produisent un int.

Exemples :

  byte niveau = 100;
  short port = 8080;
  int population = 1000000;
  long distance = 9876543210L;

Un litteral int constant peut etre affecte directement a byte ou short si sa
valeur est representable :

  byte a = 127;           // valide
  byte b = 128;           // invalide sans cast
  byte c = (byte)128;     // valide, resultat -128

Le debordement entier ne leve pas d'exception. Le resultat conserve les bits
de poids faible :

  int maximum = 2147483647;
  int suivant = maximum + 1;       // -2147483648

La division entiere tronque vers zero. Le reste % a le signe du dividende :

  7 / 3     vaut  2
 -7 / 3     vaut -2
 -7 % 3     vaut -1

Diviser un entier par zero leve ArithmeticException. Le cas
-2147483648 / -1 deborde et conserve -2147483648 sans exception, conformement
aux regles entieres Java.

Classes enveloppes correspondantes :

  byte  -> java.lang.Byte
  short -> java.lang.Short
  int   -> java.lang.Integer
  long  -> java.lang.Long

Il n'existe pas d'autoboxing : Integer i = 1; est invalide en J++ historique.
Il faut ecrire new Integer(1), puis i.intValue() pour recuperer le int.

4.4 Type char

  Taille              16 bits non signes
  Domaine             '\u0000' a '\uffff', soit 0 a 65535
  Valeur par defaut   '\u0000'
  Classe enveloppe    java.lang.Character

char contient une unite de codage Unicode sur 16 bits. Une valeur du plan
multilingue de base tient dans un char. Les caracteres Unicode hors de ce plan,
formalises plus tard sous forme de paires de substituts UTF-16, demandent deux
char et ne doivent pas etre assimiles a une seule valeur char.

  char lettre = 'A';
  char accent = '\u00e9';
  int code = lettre;             // conversion char vers int : 65

char participe aux calculs entiers et est alors promu en int :

  char c = 'A';
  c = (char)(c + 1);             // 'B'; cast necessaire

Une constante int representable peut etre affectee directement :

  char nul = 0;
  char max = 65535;

char n'est pas un entier signe. Le cast (char)-1 produit '\uffff'.

4.5 Types flottants float et double

  Type    Format IEEE 754       Precision approx.  Defaut  Suffixe
  ------  --------------------  ------------------  ------  -------
  float   binaire 32 bits       6 a 9 chiffres      0.0F   F ou f
  double  binaire 64 bits       15 a 17 chiffres    0.0D   D/d ou aucun

float et double representent les nombres finis, les deux zeros signes, les
infinis positif et negatif et NaN (Not a Number). Les constantes utiles sont
notamment Float.NaN, Float.POSITIVE_INFINITY, Double.NaN et
Double.NEGATIVE_INFINITY.

  float taux = 1.5F;
  double pi = 3.141592653589793;
  double infini = 1.0 / 0.0;       // Infinity, aucune exception
  double nan = 0.0 / 0.0;          // NaN

Contrairement a la division entiere, la division flottante par zero ne leve pas
ArithmeticException. NaN n'est egal a aucune valeur, pas meme a lui-meme :

  nan == nan                       // false
  nan != nan                       // true

Les comparaisons <, <=, > et >= avec NaN sont toutes false. Pour tester NaN,
utiliser Float.isNaN(x) ou Double.isNaN(x).

Les calculs flottants sont arrondis vers la valeur representable la plus proche
selon les regles de la JVM. Visual J++ est anterieur au mot strictfp. Selon le
processeur et la version de la MSJVM, des valeurs intermediaires peuvent avoir
une plage d'exposant plus large que le type declare. Le stockage dans un champ,
un tableau ou certains points de conversion force la valeur au format attendu.

4.6 Valeurs par defaut et initialisation

Les champs d'instance, champs static et elements de tableaux recoivent
automatiquement une valeur par defaut :

  Type                         Valeur par defaut
  ---------------------------  -----------------
  boolean                      false
  byte, short, int             0
  long                         0L
  char                         '\u0000'
  float                        0.0F
  double                       0.0D
  tout type reference          null

Exemple :

  class Etat {
      int compteur;             // 0
      boolean termine;          // false
      String message;           // null
  }

Les variables locales ne recoivent pas de valeur par defaut. Le compilateur
verifie leur affectation certaine :

  int x;
  System.out.println(x);        // erreur : x non initialise

Une variable locale peut toutefois ne pas etre affectee sur un chemin qui ne la
lit jamais. L'analyse est faite pour chaque chemin de controle.

4.7 Promotions numeriques

Les promotions determinent le type utilise pendant un calcul.

Promotion unaire :

  byte, short ou char -> int
  int                 -> int
  long                -> long
  float               -> float
  double              -> double

Promotion binaire, appliquee aux deux operandes :

  si l'un est double  -> les deux deviennent double
  sinon, si float     -> les deux deviennent float
  sinon, si long      -> les deux deviennent long
  sinon               -> les deux deviennent int

Exemples :

  byte a = 10, b = 20;
  int somme = a + b;             // resultat int

  int n = 3;
  double d = 2.0;
  double produit = n * d;        // n est converti en double

  long grand = 1L;
  float f = grand + 1.0F;        // grand est converti en float

Les operateurs de decalage suivent une regle differente : chaque operande subit
une promotion unaire; le type du resultat est celui de l'operande gauche. Seuls
les 5 bits faibles de la distance sont utilises pour int, et les 6 bits faibles
pour long.

4.8 Conversions numeriques elargissantes

Une conversion elargissante est implicite :

  byte  -> short -> int -> long -> float -> double
  char  -> int   -> long -> float -> double

Les chemins supplementaires compatibles avec ce graphe sont egalement permis,
par exemple byte vers long ou int vers double.

  byte petit = 12;
  int entier = petit;
  long grand = entier;
  double reel = grand;

Une conversion elargissante ne deborde pas au sens de la plage, mais peut perdre
de la precision lors du passage d'un entier vers float ou double :

  int exact = 16777217;
  float arrondi = exact;         // peut devenir 16777216

boolean ne participe a aucune conversion numerique. Il n'existe pas de
conversion implicite entre char et byte/short, car leurs domaines ne sont pas
inclus l'un dans l'autre.

4.9 Conversions numeriques reductrices et casts

Une conversion reductrice exige normalement un cast :

  TypeCible nom = (TypeCible)expression;

Conversions reductrices possibles :

  double -> float -> long -> int -> short -> byte
  types entiers entre eux, y compris vers et depuis char

Pour une conversion entier vers entier, les bits de poids fort sont supprimes
si necessaire; le resultat est ensuite interprete selon le type cible :

  int n = 300;
  byte b = (byte)n;              // 44
  char c = (char)-1;             // '\uffff'

Pour float/double vers un entier :

  - la partie fractionnaire est tronquee vers zero;
  - NaN devient zero;
  - une valeur trop grande est ramenee au maximum du type intermediaire;
  - une valeur trop petite est ramenee au minimum;
  - une reduction finale vers byte, short ou char conserve les bits utiles.

  int a = (int)3.9;              // 3
  int b = (int)-3.9;             // -3
  int c = (int)Double.NaN;       // 0

Un cast indique que le programmeur accepte une perte possible; il ne controle
pas la plage et ne leve pas d'exception pour un debordement numerique.

4.10 Conversions constantes lors d'une affectation

Une expression constante de type int peut etre affectee sans cast a byte,
short ou char si sa valeur est representable :

  final int DIX = 10;
  byte a = DIX;                  // valide
  short b = 30000;               // valide
  char c = 65;                   // valide, 'A'

  int variable = 10;
  byte d = variable;             // invalide : expression non constante
  byte e = 200;                  // invalide : hors du domaine

Cette facilite s'applique aux constantes calculees a la compilation :

  byte masque = 1 << 3;          // 8, representable

Elle ne transforme pas le type de l'expression arithmetique generale : a + b
reste int lorsque a et b sont byte.

4.11 Types reference : classes

Une variable de type classe contient null ou une reference vers une instance de
cette classe ou d'une sous-classe :

  Point p = new Point(1, 2);
  Object o = p;                  // Point derive de Object
  Point absent = null;

Toutes les classes derivent directement ou indirectement de java.lang.Object.
Object declare notamment equals(), hashCode(), toString(), getClass(), clone(),
finalize(), wait(), notify() et notifyAll(), avec les visibilites et contrats de
Java 1.1.

Le type compile de la variable determine quels membres peuvent etre nommes. Le
type reel de l'objet determine quelle implementation d'une methode d'instance
redefinie est appelee :

  Animal a = new Chien();
  a.crier();                     // appelle Chien.crier()

Les champs et methodes static sont resolus principalement a partir du type
compile; ils ne suivent pas le meme polymorphisme que les methodes d'instance.

4.12 Types reference : interfaces

Une variable interface contient null ou une reference vers un objet dont la
classe implemente cette interface :

  Runnable travail = new MonTravail();
  travail.run();

Une classe peut implementer plusieurs interfaces. Une interface peut etendre
plusieurs interfaces. Le type interface limite les operations visibles a ses
membres et a ceux de ses superinterfaces; un cast est necessaire pour retrouver
des membres propres a la classe concrete.

  if (travail instanceof MonTravail) {
      MonTravail m = (MonTravail)travail;
      m.methodeSpeciale();
  }

Une interface n'est pas instanciable directement. Une classe anonyme peut
toutefois l'implementer dans une expression new.

4.13 Types reference : tableaux

Un type tableau s'ecrit avec une paire [] par dimension :

  int[] vecteur;
  String[][] matrice;
  int ancienStyle[];             // valide mais moins recommande

Creation et initialisation :

  vecteur = new int[10];
  int[] premiers = { 2, 3, 5, 7 };
  String[][] noms = new String[2][3];
  int[][] irregulier = new int[2][];
  irregulier[0] = new int[3];
  irregulier[1] = new int[8];

Chaque tableau est un objet :

  - sa taille fixe est accessible par le champ final length;
  - ses indices vont de 0 a length - 1;
  - un mauvais indice leve ArrayIndexOutOfBoundsException;
  - un acces par une reference null leve NullPointerException;
  - ses elements recoivent leur valeur par defaut;
  - il derive de Object et implemente Cloneable et Serializable selon le
    runtime/bibliotheque disponible.

Les tableaux de types reference sont covariants :

  String[] textes = new String[2];
  Object[] objets = textes;      // conversion permise
  objets[0] = new Integer(1);    // ArrayStoreException a l'execution

Les tableaux primitifs ne sont pas covariants entre eux : int[] n'est ni un
long[] ni un Object[]. Il reste cependant un Object.

Un tableau multidimensionnel est un tableau de tableaux. Ses lignes peuvent
avoir des longueurs differentes; new int[3][4] cree un tableau externe et trois
tableaux internes distincts.

4.14 Chaines String et StringBuffer

java.lang.String est une classe reference speciale dont le langage reconnait
les litteraux et la concatenation par + :

  String nom = "Ada";
  String message = "Bonjour " + nom + " numero " + 1;

Un objet String est immuable. Une operation qui semble le modifier cree une
nouvelle chaine ou emploie un tampon interne :

  String s = "a";
  s = s + "b";                   // nouvelle valeur String "ab"

Les litteraux et expressions constantes String peuvent etre internes. == teste
l'identite des references, pas le contenu :

  String a = new String("abc");
  String b = new String("abc");
  a == b                         // false en general
  a.equals(b)                    // true

StringBuffer est la classe mutable historique pour construire efficacement une
chaine :

  StringBuffer buf = new StringBuffer();
  buf.append("Age=").append(42);
  String resultat = buf.toString();

StringBuilder n'existe pas dans Java 1.1. Une concatenation String peut etre
traduite par le compilateur en operations StringBuffer.

4.15 Classes enveloppes et absence d'autoboxing

Chaque type primitif possede une classe enveloppe :

  boolean -> Boolean       char   -> Character
  byte    -> Byte          short  -> Short
  int     -> Integer       long   -> Long
  float   -> Float         double -> Double

Les enveloppes sont des objets immuables. En J++ historique, les conversions
entre primitif et enveloppe sont toujours explicites :

  Integer objet = new Integer(42);  // boxing manuel
  int valeur = objet.intValue();    // unboxing manuel

  Vector liste = new Vector();
  liste.addElement(new Integer(42));
  int n = ((Integer)liste.elementAt(0)).intValue();

Les methodes parseInt, valueOf et toString disponibles dependent de la version
exacte de java.lang livree avec la MSJVM, mais le principe reste identique.
Integer ne derive pas de int et un cast (Integer) ne peut pas convertir
directement une valeur primitive.

4.16 null et dereferencement

null est le litteral de reference nulle. Il est convertible vers tout type
classe, interface ou tableau :

  String texte = null;
  Runnable action = null;
  int[] valeurs = null;

Il n'est convertible vers aucun type primitif. Toute reference peut etre
comparee a null par == ou !=. L'acces a un membre d'instance, a un element de
tableau ou a length par null leve NullPointerException :

  if (texte != null)
      System.out.println(texte.length());

Appeler une methode static au moyen d'une expression reference est autorise
par l'ancien langage mais deconseille : la resolution est statique et peut ne
pas dereferencer l'objet comme le lecteur s'y attend.

4.17 Egalite, identite et equivalence

Pour les primitifs, == et != comparent les valeurs apres promotion numerique.
boolean ne se compare qu'a boolean. Pour les references, ils comparent
l'identite : les deux operandes designent-ils le meme objet ou sont-ils tous
deux null ?

  Point p = new Point(1, 2);
  Point q = p;
  p == q                         // true

equals() est une methode virtuelle definie par Object. Une classe peut la
redefinir pour comparer un contenu logique. Lorsqu'elle le fait, elle doit
normalement redefinir hashCode() de facon coherente.

Les tableaux ne redefinissent pas equals() pour leur contenu dans l'API
historique : deux tableaux distincts restent inegaux selon equals(), meme si
leurs elements sont identiques. Une comparaison element par element est alors
necessaire.

4.18 Conversions de references

Conversions elargissantes implicites :

  sous-classe -> superclasse
  classe      -> interface qu'elle implemente
  sous-interface -> superinterface
  tableau reference -> Object, Cloneable ou Serializable
  tableau S[] -> tableau T[] si S -> T est elargissante
  null -> tout type reference

Exemple :

  Chien chien = new Chien();
  Animal animal = chien;
  Object objet = animal;

Une conversion reductrice exige un cast :

  Animal animal = obtenirAnimal();
  Chien chien = (Chien)animal;

Le cast est autorise a la compilation lorsque les types peuvent plausiblement
etre lies. A l'execution :

  - null caste vers un type reference reste null;
  - une instance compatible est acceptee;
  - une instance incompatible leve ClassCastException.

instanceof permet le test prealable :

  if (animal instanceof Chien) {
      Chien chien = (Chien)animal;
      chien.aboyer();
  }

Il n'existe pas de conversion definie par l'utilisateur ni de surcharge des
operateurs de cast en Java/J++.

4.19 Conversions String

L'operateur + convertit un operande en String lorsqu'un autre operande est une
String. Les primitifs utilisent leur representation textuelle; une reference
non nulle utilise String.valueOf, qui appelle normalement toString(); null
devient le texte "null".

  String s1 = "valeur=" + 42;        // "valeur=42"
  String s2 = "actif=" + true;       // "actif=true"
  String s3 = "objet=" + (Object)null; // "objet=null"

L'association de gauche a droite est importante :

  "total=" + 1 + 2                // "total=12"
  "total=" + (1 + 2)              // "total=3"

Il n'existe aucune conversion generale implicite de String vers nombre. Il faut
employer les methodes des classes enveloppes, par exemple :

  int n = Integer.parseInt("42");
  double d = Double.valueOf("3.5").doubleValue();

Une chaine invalide leve NumberFormatException.

4.20 Passage des arguments et affectation

Java et J++ passent toujours les arguments par valeur.

Pour un primitif, la valeur numerique ou booleenne est copiee :

  void incrementer(int x) { x++; }   // ne modifie pas l'argument appelant

Pour un objet, la reference est copiee. La methode peut modifier l'objet vise,
mais remplacer sa copie locale de la reference ne remplace pas la variable de
l'appelant :

  void modifier(Point p) {
      p.x = 10;                       // visible par l'appelant
      p = new Point(0, 0);            // seulement la copie locale
  }

Les tableaux suivent la meme regle puisqu'ils sont des objets. Les conventions
J/Direct peuvent utiliser un tableau a un element pour simuler un parametre
natif de sortie ou passe par adresse; cela reste une convention de marshalling,
pas un passage par reference du langage J++.

4.21 Type compile, type reel et liaison dynamique

Toute expression a un type compile connu du compilateur. Une reference peut
designer a l'execution une instance d'un sous-type :

  Forme f = new Cercle();

Ici, Forme est le type compile de f et Cercle son type reel. Les membres
accessibles sont controles avec Forme. Une methode d'instance redefinie est
selectionnee dynamiquement d'apres Cercle. Les champs, methodes static et
methodes private ne suivent pas cette selection dynamique.

getClass() retourne l'objet java.lang.Class representant le type reel :

  Class typeReel = f.getClass();
  String nom = typeReel.getName();

Deux classes de meme nom chargees par deux chargeurs de classes differents
peuvent etre considerees comme deux types distincts par la JVM.

4.22 Compatibilite d'affectation et d'appel

Une expression est compatible avec une variable, un parametre ou un retour si
au moins une des operations suivantes suffit :

  - identite de type;
  - conversion primitive elargissante;
  - conversion de reference elargissante;
  - conversion constante int vers byte, short ou char dans les cas permis;
  - conversion de null vers un type reference.

Le compilateur n'insere ni cast reducteur, ni boxing, ni conversion String vers
nombre. Pour une surcharge, il cherche une methode applicable aux arguments et
choisit la plus specifique. Java 1.1/J++ ne dispose pas de parametres
optionnels, nommes ou varargs :

  void f(int x) { ... }
  void f(long x) { ... }

  f(1);                           // choisit f(int)
  f(1L);                          // choisit f(long)

Le type retour seul ne participe jamais au choix d'une surcharge.

4.23 Types et extensions Microsoft

Les extensions Microsoft reposent toujours sur les types precedents :

  - un delegate est un type reference genere par le compilateur;
  - un handle Win32 est souvent represente par int dans la MSJVM 32 bits;
  - un pointeur ou parametre de sortie peut etre represente par une classe,
    un tableau ou un type de com.ms.dll/com.ms.win32;
  - une valeur Automation utilise souvent com.ms.com.Variant;
  - une interface COM est exposee comme une interface reference Java;
  - les structures Win32 deviennent des classes de donnees Java.

Exemple conceptuel :

  Variant valeur = new Variant(42);
  int hwnd = 0;
  User32.MessageBox(hwnd, "Texte", "Titre", winm.MB_OK);

Ces correspondances ne sont pas des equivalences universelles. Leur taille,
leur sens entree/sortie, leur encodage et leur duree de vie sont fixes par les
declarations J/Direct et le runtime Microsoft. Il faut notamment eviter de
supposer qu'un int reste un type de pointeur correct hors de la MSJVM Win32.

4.24 Resume des erreurs liees aux types

  Erreur/exception                    Cause typique
  ----------------------------------  ------------------------------------------
  erreur de compilation               types incompatibles sans conversion
  ClassCastException                  cast de reference incompatible
  NullPointerException                dereferencement de null
  ArrayStoreException                 objet incompatible stocke dans un tableau
  ArrayIndexOutOfBoundsException      indice hors de 0..length-1
  NegativeArraySizeException          dimension negative a la creation
  ArithmeticException                 division ou reste entier par zero
  NumberFormatException               texte numerique invalide

Un debordement entier, une perte de precision flottante ou un cast numerique
reducteur ne leve normalement aucune exception. La verification de plage doit
etre programmee explicitement lorsque ces cas sont interdits par l'application.


================================================================================
5. VARIABLES, CHAMPS ET CONSTANTES
================================================================================

5.1 Categories de variables

Le langage distingue sept emplacements contenant une valeur :

  1. champs d'instance;
  2. champs de classe, declares static;
  3. variables locales;
  4. parametres de methodes et de constructeurs;
  5. parametres de gestionnaires catch;
  6. composantes de tableaux;
  7. variables cachees creees par le compilateur ou la machine virtuelle.

Les six premieres categories sont visibles dans le langage source. La derniere
comprend par exemple la reference vers l'instance englobante d'une classe
interne ou des temporaires de compilation; leurs noms et leur representation ne
font pas partie de la specification.

Une variable possede :

  - un type fixe a la compilation;
  - un nom, sauf pour certaines variables cachees;
  - une valeur courante;
  - une portee lexicale, qui indique ou son nom est utilisable;
  - une duree de vie, qui indique combien de temps son stockage existe;
  - eventuellement des modificateurs tels que final, static ou volatile.

Le contenu d'une variable primitive est une valeur primitive. Le contenu d'une
variable reference est null ou une reference; l'objet vise reste distinct de la
variable qui contient cette reference.

5.2 Syntaxe generale des declarations

Declaration simple :

  [modificateurs] Type nom [= initialiseur] ;

Declaration de plusieurs variables du meme type :

  [modificateurs] Type nom1 [= init1], nom2 [= init2], nom3 ;

Exemples :

  int compteur;
  int x = 1, y = 2;
  String message = "Bonjour";
  final int maximum = 100;

La virgule separe des declarateurs, pas des instructions. Chaque initialiseur
est evalue de gauche a droite :

  int a = 1, b = a + 1, c = b + 1;

Le type apparait une seule fois et s'applique a tous les declarateurs. J++ 6.0
n'a ni inference locale var, ni declaration multiple de types differents, ni
deconstruction.

La notation de tableau peut suivre le type ou chaque nom :

  int[] a, b;              // a et b sont int[]
  int c[], d;              // c est int[], d est int
  int[] e[];               // e est int[][]

La premiere forme est recommandee, car elle rend le type commun plus visible.

5.3 Champs d'instance

Syntaxe :

  [public | protected | private]
  [final] [transient] [volatile] Type nom [= initialiseur] ;

Un champ d'instance appartient a un objet. Chaque instance possede son propre
emplacement :

  class Compteur {
      private int valeur;

      public void incrementer() {
          valeur++;
      }
  }

  Compteur a = new Compteur();
  Compteur b = new Compteur();
  a.incrementer();         // ne modifie pas b.valeur

Dans une methode d'instance, un champ peut etre nomme directement ou avec this :

  valeur = 10;
  this.valeur = 10;

Depuis une autre classe, l'acces passe par une expression reference :

  objet.champ

L'expression est evaluee avant l'acces. Si elle vaut null, l'acces a un champ
d'instance leve NullPointerException.

Un champ est selectionne statiquement selon le type compile de la reference; il
n'est pas polymorphe comme une methode redefinie :

  class A { int valeur = 1; }
  class B extends A { int valeur = 2; }

  A objet = new B();
  objet.valeur             // 1, champ declare par A

Le champ de B masque celui de A mais ne le remplace pas. Dans B, super.valeur
accede au champ de A.

5.4 Champs de classe static

Syntaxe :

  [visibilite] static [final] [transient] [volatile]
  Type nom [= initialiseur] ;

Un champ static appartient a la classe. Il existe en un seul exemplaire par
classe chargee, et non un exemplaire par objet :

  class Compteur {
      private static int nombreInstances;

      Compteur() {
          nombreInstances++;
      }

      public static int getNombreInstances() {
          return nombreInstances;
      }
  }

Acces recommande :

  Compteur.getNombreInstances();
  Compteur.nombreInstances;       // seulement si le champ est accessible

Java 1.1 autorise egalement objet.champStatic, mais cette forme est trompeuse :
le champ est choisi selon le type compile de l'expression, pas selon l'objet.
Utiliser le nom de classe rend la liaison statique explicite.

Une methode static peut acceder directement aux champs static de sa classe.
Elle ne peut acceder a un champ d'instance sans posseder une reference vers un
objet, car elle n'a pas de this.

Chaque chargeur de classes peut charger sa propre copie d'une classe; chacune
possede alors ses propres champs static. "Une seule copie" signifie donc une
copie par classe telle que definie par la JVM, pas necessairement une copie pour
tout le processus.

5.5 Variables locales

Syntaxe dans un bloc :

  [final] Type nom [= initialiseur] ;

Exemple :

  void calculer() {
      int total = 0;
      final int facteur = 2;

      for (int i = 0; i < 10; i++) {
          int produit = i * facteur;
          total += produit;
      }
  }

Les modificateurs public, protected, private, static, transient et volatile ne
s'appliquent pas aux variables locales. final est permis.

Une variable locale est creee lorsque l'execution atteint son bloc ou son
declarateur et cesse d'etre accessible a la sortie de sa portee. Son stockage
peut etre place dans une pile, un registre ou optimise; cette representation
n'est pas garantie par le langage.

Contrairement aux champs, une locale n'a aucune valeur par defaut. Elle doit
etre definitivement affectee avant toute lecture. Le ramasse-miettes peut
recuperer un objet des qu'aucune reference encore utilisable ne le rend
atteignable, meme si le bloc source n'est pas encore termine.

5.6 Parametres

Syntaxe d'une methode ou d'un constructeur :

  TypeRetour methode([final] Type p1, [final] Type p2) { ... }
  NomClasse([final] Type p1, [final] Type p2) { ... }

Les parametres sont des variables locales initialisees avec les valeurs des
arguments lors de l'appel :

  int additionner(int gauche, int droite) {
      return gauche + droite;
  }

J++ passe toujours les arguments par valeur. Affecter un parametre primitif ne
modifie pas la variable de l'appelant. Pour un objet, la reference est copiee :
la methode peut modifier l'objet vise, mais remplacer le parametre ne remplace
pas la reference de l'appelant.

  void changer(Point p) {
      p.x = 10;                   // modifie l'objet de l'appelant
      p = new Point(0, 0);        // modifie seulement le parametre local
  }

Un parametre final ne peut pas recevoir une nouvelle valeur dans la methode,
mais l'objet qu'il designe peut rester mutable.

Java 1.1/J++ ne possede pas de parametres optionnels, nommes, par defaut ou
varargs. Une methode voulant accepter plusieurs combinaisons emploie des
surcharges ou un tableau explicite.

5.7 Parametre de catch

Syntaxe :

  catch ([final] TypeException nom) {
      instructions
  }

La machine virtuelle initialise ce parametre avec l'objet Throwable capture.
Sa portee est le bloc catch correspondant :

  try {
      lire();
  } catch (IOException erreur) {
      System.err.println(erreur.getMessage());
  }

Il ne peut etre lu hors de ce bloc. Son type determine les exceptions capturees
et doit etre Throwable ou une sous-classe. Un parametre final ne peut etre
remplace par une autre reference.

5.8 Composantes de tableaux

Chaque element d'un tableau se comporte comme une variable anonyme selectionnee
par un indice :

  int[] valeurs = new int[3];
  valeurs[0] = 10;
  valeurs[1] = valeurs[0] + 5;

Le type de valeurs[i] est le type element du tableau. L'expression tableau puis
l'indice sont evalues avant l'acces. Une reference null leve
NullPointerException; un indice hors de 0..length-1 leve
ArrayIndexOutOfBoundsException.

Les composantes recoivent automatiquement leur valeur par defaut a la creation
du tableau. Elles ne peuvent pas etre declarees individuellement final,
volatile ou transient. Une variable final contenant un tableau empeche de
remplacer la reference, mais pas de modifier ses elements :

  final int[] nombres = new int[3];
  nombres[0] = 42;              // valide
  nombres = new int[4];         // invalide

Pour un tableau covariant de references, une affectation incompatible leve
ArrayStoreException a l'execution.

5.9 Portee lexicale

La portee indique la region du source dans laquelle un nom simple designe une
variable.

  Type de variable       Debut de la portee            Fin
  ---------------------  -----------------------------  ------------------------
  champ                  corps de la classe            corps de la classe
  parametre de methode   declaration terminee          corps de la methode
  parametre constructeur declaration terminee          corps du constructeur
  parametre catch        apres sa declaration          fin du bloc catch
  locale de bloc         apres son declarateur         fin du bloc
  locale de for          apres son declarateur         instruction for incluse

Exemple :

  {
      int a = 1;
      {
          int b = a + 1;         // a et b visibles
      }
      // b n'est plus visible
  }

La variable declaree dans l'initialisation d'un for est visible dans la
condition, la mise a jour et le corps, mais pas apres la boucle :

  for (int i = 0; i < 10; i++) {
      System.out.println(i);
  }
  // i n'est pas visible ici

La portee n'est pas la duree de vie. Un objet reference par une locale peut
survivre au bloc s'il est stocke ailleurs; inversement, il peut devenir
inatteignable avant la fin lexicale de la portee.

5.10 Masquage et redeclaration des noms

Un parametre ou une variable locale peut masquer un champ :

  class Personne {
      private int age;

      Personne(int age) {
          this.age = age;
      }
  }

Dans le constructeur, age designe le parametre et this.age le champ.

Une classe derivee peut masquer un champ herite; Classe.nom ou super.nom permet
de preciser le champ souhaite selon le contexte. Un champ static peut aussi
etre masque par un champ static d'une sous-classe.

Le langage interdit de redeclarer une variable locale ou un parametre dans une
portee locale englobante qui se chevauche :

  void f(int x) {
      int x = 1;                 // invalide : parametre deja nomme x
      {
          int x = 2;             // invalide dans la portee de ce x
      }
  }

Deux blocs successifs sans chevauchement peuvent reutiliser un nom. Une locale
peut aussi masquer un champ, puisque les espaces de declaration concernes sont
differents.

5.11 Affectation certaine des variables locales

Le compilateur effectue une analyse de controle avant de permettre la lecture
d'une locale. Il faut que toutes les routes pouvant atteindre la lecture aient
affecte la variable :

  int resultat;
  if (condition)
      resultat = 1;
  else
      resultat = 2;
  System.out.println(resultat);  // valide

Exemple invalide :

  int resultat;
  if (condition)
      resultat = 1;
  System.out.println(resultat);  // condition peut etre false

Une boucle n'est generalement pas supposee s'executer au moins une fois, sauf
si le compilateur peut l'etablir avec les regles formelles. Une affectation
situee uniquement dans un while peut donc ne pas suffire apres la boucle.

L'operateur += lit l'ancienne valeur avant d'ecrire la nouvelle; la variable
doit deja etre initialisee :

  int n;
  n += 1;                        // invalide

Les champs ne sont pas soumis a cette lecture certaine ordinaire, car ils ont
une valeur par defaut. Les champs final suivent toutefois des regles
d'affectation certaine particulieres.

5.12 Modificateur final

final signifie qu'une variable ne peut etre affectee qu'une fois :

  final int maximum = 100;
  final String nom = "J++";

Une seconde affectation est une erreur de compilation :

  maximum = 200;                 // invalide

Pour une reference, final protege la variable et non l'objet :

  final Vector liste = new Vector();
  liste.addElement("A");         // valide : objet modifie
  liste = new Vector();          // invalide : reference remplacee

Une variable final peut etre initialisee dans son declarateur ou, dans certains
cas, plus tard lorsqu'elle est definitivement non affectee. On parle alors de
blank final.

  class Connexion {
      private final String serveur;

      Connexion(String serveur) {
          this.serveur = serveur;
      }
  }

Chaque constructeur doit affecter exactement une fois tout champ final
d'instance non initialise, directement ou par un autre constructeur. Un champ
static final vide doit etre affecte dans un initialisateur static.

  class Configuration {
      static final String RACINE;

      static {
          RACINE = chargerRacine();
      }
  }

final n'implique ni constante de compilation, ni immuabilite profonde, ni
synchronisation entre threads.

5.13 Constantes de compilation

Une variable constante au sens de la grammaire Java est une variable final de
type primitif ou String, initialisee par une expression constante :

  static final int TAILLE = 80;
  static final int DOUBLE = TAILLE * 2;
  static final String VERSION = "6." + "0";

Une expression constante n'utilise que des litteraux, certaines variables
constantes, des operateurs autorises et des conversions primitives ne demandant
aucun calcul a l'execution. Un appel de methode ou new exclut la constante :

  static final int A = 1 + 2;           // constante de compilation
  static final int B = Math.abs(-3);    // final, mais pas constante compilee
  static final Integer C = new Integer(3); // final, pas type primitif/String

Une constante de compilation peut etre utilisee dans case, dans certaines
initialisations reductrices et par le compilateur pour simplifier des
expressions.

Les clients peuvent incorporer directement la valeur d'un champ public static
final constant dans leur propre bytecode. Changer ensuite la constante sans
recompiler les clients peut leur laisser l'ancienne valeur. Pour une valeur
susceptible de changer entre versions, une methode d'acces peut etre preferable.

5.14 const n'est pas le mot-cle des constantes

const est reserve mais inutilise :

  const int MAX = 100;            // invalide en J++/Java

Il faut ecrire :

  final int max = 100;
  static final int MAX = 100;     // constante partagee par la classe

static indique l'appartenance a la classe; final interdit la reaffectation. La
combinaison static final est la forme habituelle d'une constante publique, mais
elle n'est une constante de compilation que si les conditions exposees dans la
section 5.13 sont satisfaites.

Les symboles de compilation conditionnelle de JVC ne sont pas des variables
const Java. Ils sont interpretes par les directives Microsoft avant ou pendant
la compilation; voir section 17.

5.15 Champs d'interface

Tout champ declare dans une interface est implicitement :

  public static final

Il doit posseder un initialiseur :

  interface Limites {
      int MINIMUM = 0;
      int MAXIMUM = 100;
  }

La declaration equivaut a :

  public static final int MINIMUM = 0;

Une classe qui implemente l'interface peut utiliser le nom si aucune ambiguite
n'existe, mais Limites.MINIMUM est plus clair. Un champ d'interface ne constitue
jamais un etat d'instance et ne peut etre modifie.

Si deux interfaces heritees fournissent le meme nom, un acces simple peut etre
ambigu; qualifier avec le nom de l'interface resout l'ambiguite.

5.16 Valeurs par defaut des champs

Avant tout initialiseur Java, la JVM attribue a chaque champ sa valeur par
defaut :

  boolean              false
  byte, short, int     0
  long                 0L
  char                 '\u0000'
  float                0.0F
  double               0.0D
  reference            null

Exemple :

  class Etat {
      int nombre;                // 0
      boolean actif;             // false
      String texte;              // null
  }

L'initialiseur explicite remplace ensuite cette valeur :

  int nombre = 10;
  String texte = "pret";

Il est legal mais rarement utile de recopier explicitement la valeur par
defaut. La valeur par defaut existe avant l'execution des constructeurs et peut
etre observee dans des situations d'initialisation mal concues; un constructeur
ne doit pas publier this avant d'avoir etabli ses invariants.

5.17 Ordre d'initialisation static

Une classe est initialisee lors de sa premiere utilisation active, par exemple
avant la creation de sa premiere instance, l'appel d'une methode static ou
l'acces a un champ static non constant. Avant cela, ses champs static ont deja
leur valeur par defaut.

Les initialisateurs de champs static et blocs static s'executent dans l'ordre
textuel :

  class Exemple {
      static int a = 1;

      static {
          a = a + 1;             // a vaut 2
          b = 5;                 // ecriture permise
      }

      static int b = 3;          // remplace ensuite 5 par 3
  }

La superclasse est initialisee avant la sous-classe. Une classe n'est
normalement initialisee qu'une fois par son chargeur de classes. Si
l'initialisation echoue par exception, l'utilisation ulterieure peut produire
ExceptionInInitializerError ou NoClassDefFoundError selon le moment.

Les references simples illegales vers un champ declare plus bas sont soumises
aux regles de reference avant declaration. Ecrire un champ peut etre permis la
ou le lire directement ne l'est pas. Il est preferable de declarer les champs
avant les blocs qui les utilisent.

5.18 Ordre d'initialisation d'une instance

Lors de new Classe(arguments), l'ordre conceptuel est :

  1. allocation de l'objet;
  2. mise a la valeur par defaut de tous ses champs;
  3. appel du constructeur de la superclasse;
  4. initialisateurs de champs d'instance, dans l'ordre textuel;
  5. blocs d'initialisation d'instance, dans l'ordre textuel;
  6. reste du corps du constructeur.

Exemple :

  class Exemple extends Base {
      int a = calculerA();

      {
          preparer();
      }

      Exemple() {
          terminer();
      }
  }

La premiere instruction d'un constructeur peut etre this(...) ou super(...).
Si elle est absente, super() est insere. Lorsqu'un constructeur appelle
this(...), l'initialisation d'instance est effectuee une seule fois par le
constructeur qui finit par appeler super(...), et non a chaque maillon.

Pendant l'execution du constructeur de la superclasse, les champs propres a la
sous-classe n'ont encore que leur valeur par defaut. Appeler une methode
redefinissable depuis un constructeur est donc dangereux : l'implementation de
la sous-classe peut observer un objet partiellement initialise.

5.19 Initialiseur de champ

Un initialiseur de champ est une expression :

  private int largeur = 80;
  private Vector elements = new Vector();
  private static String version = calculerVersion();

Un initialiseur d'instance peut utiliser this implicitement et appeler des
methodes d'instance, sous reserve des regles de reference avant declaration. Un
initialiseur static n'a pas de this et n'accede directement qu'au contexte
static.

L'expression doit etre compatible par affectation avec le type du champ. Elle
peut lever une exception non verifiee. Une exception verifiee dans un
initialiseur de champ est fortement contrainte : elle doit etre geree ou etre
compatible avec les constructeurs selon les regles du langage.

Creer un objet mutable dans un champ static le partage entre toutes les
instances. Creer l'objet dans un champ d'instance donne normalement une
instance distincte a chaque construction.

5.20 Blocs d'initialisation

Bloc d'instance :

  {
      instructions
  }

Bloc de classe :

  static {
      instructions
  }

Les blocs d'instance factorisent du code execute par tous les constructeurs.
Ils peuvent completer l'affectation de champs final d'instance sous les regles
d'affectation certaine. Ils ne peuvent contenir return.

Les blocs static initialisent les ressources partagees et les champs static
final vides. Ils ne peuvent utiliser this ou super comme reference d'instance.
Une exception verifiee ne peut pas simplement en sortir; elle doit etre
capturee. Une exception non verifiee rend l'initialisation de classe fautive.

Un bloc d'initialisation n'est pas une methode : il n'a ni nom, ni parametres,
ni type retour et ne peut etre appele explicitement.

5.21 transient

Syntaxe :

  transient Type champ;

transient ne s'applique qu'aux champs. Il indique au mecanisme standard de
serialisation que le champ ne fait pas partie de l'etat persistant ordinaire :

  class Session implements Serializable {
      String utilisateur;
      transient String motDePasse;
  }

Apres une deserialisation standard, motDePasse vaut null, sauf si la classe le
restaure par un mecanisme personnalise. transient ne masque pas le champ,
n'efface pas sa memoire et ne constitue pas une protection cryptographique.

Les champs static ne sont pas l'etat d'une instance serialisee; transient y est
donc sans utilite pratique meme si certaines grammaires anciennes acceptent la
combinaison.

5.22 volatile

Syntaxe :

  volatile Type champ;

volatile ne s'applique qu'aux champs. Il demande que les lectures et ecritures
passent par la semantique de memoire partagee de la JVM, au lieu de conserver
indefiniment une copie invisible aux autres threads :

  private volatile boolean arreter;

  public void demanderArret() {
      arreter = true;
  }

  public void executer() {
      while (!arreter) {
          travailler();
      }
  }

volatile ne rend pas atomique une suite lecture-calcul-ecriture :

  volatile int compteur;
  compteur++;                     // operation composee, pas atomique

Pour proteger un invariant ou une operation composee, utiliser synchronized.
Le modele memoire de Java 1.1/MSJVM est historique et moins precis que celui
formalise par Java 5; ne pas lui attribuer retroactivement toutes les garanties
modernes. Les anciennes JVM peuvent aussi traiter long/double volatile selon
leurs propres contraintes d'atomicite.

5.23 Visibilite des champs

  Modificateur  Acces principal
  ------------  --------------------------------------------------------------
  private       classe qui declare le champ
  aucun         classes du meme package
  protected     meme package et sous-classes selon les regles de receveur
  public        partout ou la classe englobante est accessible

La visibilite est controlee a la compilation. Elle ne change ni le type ni la
duree de vie du champ. Un champ public d'une classe non publique reste limite
par l'accessibilite de cette classe.

Les champs publics mutables exposent directement la representation interne :

  public int age;                 // possible, mais peu encapsule

La forme habituelle emploie un champ private et des methodes :

  private int age;
  public int getAge() { return age; }
  public void setAge(int valeur) { age = valeur; }

Comme explique a la section 15.3, l'IDE J++ peut presenter ces methodes comme
une propriete de composant; cela ne change pas les regles d'acces du langage.

5.24 Variables locales et classes internes

Java 1.1 autorise une classe locale ou anonyme a utiliser une variable locale
englobante seulement si cette variable est final :

  final String message = "Bonjour";

  Runnable action = new Runnable() {
      public void run() {
          System.out.println(message);
      }
  };

La variable capturee doit etre definitivement affectee avant la creation de
l'objet interne. Le compilateur peut copier sa valeur dans un champ cache de
la classe generee.

La notion moderne de variable "effectivement finale" n'existe pas encore :
omettre final rend la capture invalide en Java 1.1/J++ 6.0, meme si le source
ne reaffecte jamais la variable.

Les champs de l'instance englobante sont accessibles par la reference cachee et
n'ont pas besoin d'etre final. Une classe imbriquee static ne possede pas cette
reference d'instance englobante.

5.25 Duree de vie et ramasse-miettes

La duree de vie du stockage depend de la categorie :

  champ d'instance   aussi longtemps que l'objet est atteignable
  champ static       aussi longtemps que la classe reste chargee
  variable locale    pendant l'activation utile de son bloc/methode
  parametre          pendant l'appel
  element tableau    aussi longtemps que le tableau est atteignable

La fin de portee d'un nom ne detruit pas explicitement l'objet. Le
ramasse-miettes recupere les objets devenus inatteignables a un moment non
determine. Affecter null peut supprimer une reference, mais n'oblige pas une
collecte immediate :

  objet = null;
  System.gc();                    // simple demande, aucune garantie

Les ressources externes telles que fichiers, handles Win32 et objets COM ne
doivent pas dependre seulement du ramasse-miettes. Il faut les fermer ou les
liberer explicitement, notamment avec close() ou ComLib.release selon l'API.

5.26 Variables et interoperabilite Microsoft

Les paquets J/Direct et COM utilisent des variables Java pour representer des
valeurs natives :

  int hwnd;                       // handle Win32 dans la MSJVM 32 bits
  int[] resultat = new int[1];    // parametre de sortie simule
  Variant valeur;                 // valeur COM Automation
  IUnknown objetCom;              // interface COM

Un tableau a un element peut servir a simuler un pointeur de sortie, mais la
variable Java contient toujours une reference vers le tableau; le langage
n'acquiert pas un passage par reference.

Les declarations J/Direct determinent le marshalling, le sens des parametres et
la taille native. Un int convenable pour un handle dans Visual J++ 6.0/MSJVM
32 bits ne doit pas etre suppose portable vers un processus 64 bits.

Un objet COM stocke dans une variable reference peut demander une liberation
deterministe. Affecter null ne garantit pas que les references COM soient
relachees immediatement; voir com.ms.com.ComLib a la section 18.2.

5.27 Conventions de nommage

Les conventions n'ont aucun effet grammatical, mais ameliorent la lecture :

  compteur, nombreElements       variables et champs
  MAXIMUM, TAILLE_BLOC           constantes static final
  nomDeClasse.compteurGlobal     acces explicite a un champ static
  this.valeur                    distinction champ/parametre

Le prefixe hongrois lie au type, courant dans certains programmes Windows
anciens, est autorise mais non exige :

  int nCompteur;
  String strNom;

Comme J++ est sensible a la casse, MAXIMUM, Maximum et maximum sont trois noms
distincts. Eviter les noms commencant par $ destines aux outils et les noms qui
ressemblent aux mots contextuels Microsoft delegate et multicast.

5.28 Erreurs frequentes

  Erreur                               Explication
  -----------------------------------  -----------------------------------------
  lire une locale non initialisee      affectation certaine absente
  reaffecter une variable final        une seule affectation est permise
  modifier l'objet d'une final         permis : final protege la reference
  employer const                       mot reserve sans syntaxe executable
  acceder a l'instance depuis static   aucune reference this disponible
  croire un champ polymorphe           les champs sont resolus statiquement
  capturer une locale non final        interdit aux classes internes Java 1.1
  croire compteur++ atomique           volatile ne suffit pas
  attendre la liberation par x=null    collecte et finalisation non garanties
  confondre portee et duree de vie     notions distinctes

Exemple recapitulatif :

  public class Mesure {
      public static final int LIMITE = 100;
      private static int nombre;

      private final int identifiant;
      private volatile boolean active;
      private transient String cache;
      private int valeur;

      public Mesure(int valeur) {
          identifiant = ++nombre;
          this.valeur = valeur;
          active = true;
      }

      public int getValeur() {
          return valeur;
      }

      public void setValeur(int valeur) {
          if (valeur < 0 || valeur > LIMITE)
              throw new IllegalArgumentException("hors limites");
          this.valeur = valeur;
      }
  }

LIMITE est une constante de compilation partagee; nombre est un etat de
classe; identifiant est un blank final initialise par le constructeur; active
est un indicateur volatile; cache est exclu de la serialisation ordinaire; et
valeur est un champ d'instance encapsule par des methodes.


================================================================================
6. OPERATEURS ET EXPRESSIONS
================================================================================

  postfixes       expr++ expr-- () [] .
  unaires         ++expr --expr + - ~ ! (type)
  multiplicatifs  * / %
  additifs        + -                 (+ concatene aussi les String)
  decalages       << >> >>>
  relationnels    < <= > >= instanceof
  egalite         == !=
  bits/logique    & puis ^ puis |
  conditionnels   && puis ||
  ternaire        condition ? a : b
  affectations    = *= /= %= += -= <<= >>= >>>= &= ^= |=

Les operateurs && et || sont a evaluation conditionnelle. == compare les
valeurs primitives, mais l'identite des references; equals compare en general
le contenu selon le contrat de la classe. La division entiere par zero leve
ArithmeticException. Les calculs entiers debordent sans exception.

L'evaluation des operandes se fait de gauche a droite. Les operateurs de meme
niveau sont generalement associatifs a gauche; le ternaire et les affectations
sont associatifs a droite.

6.1 Precedence complete

Du plus prioritaire au moins prioritaire :

  Niveau  Operateurs/constructions                 Associativite
  ------  ---------------------------------------  -------------
   1      nom, litteral, this, super, new           -
          appel (), indice [], membre .
          expr++, expr--
   2      ++expr, --expr, +expr, -expr, ~, !        droite
          cast (Type)expr
   3      *, /, %                                   gauche
   4      +, -                                      gauche
   5      <<, >>, >>>                               gauche
   6      <, <=, >, >=, instanceof                  gauche
   7      ==, !=                                    gauche
   8      &                                         gauche
   9      ^                                         gauche
  10      |                                         gauche
  11      &&                                        gauche
  12      ||                                        gauche
  13      ?:                                        droite
  14      =, +=, -=, *=, /=, %=                     droite
          &=, ^=, |=, <<=, >>=, >>>=

  a + b * c                  equivaut a a + (b * c)
  (a + b) * c                parentheses prioritaires
  a - b - c                  equivaut a (a - b) - c
  a = b = 0                  equivaut a a = (b = 0)

La precedence fixe le groupement syntaxique. Elle ne change pas l'ordre
temporel : J++ evalue les operandes de gauche a droite.

6.2 Ordre d'evaluation et effets de bord

Dans f() + g(), f() se termine avant l'appel de g(). Dans :

  receveur.methode(arg1, arg2, arg3)

le receveur est evalue en premier, puis les arguments de gauche a droite. Dans
tableau[index], la reference tableau est evaluee avant l'indice.

Si un operande leve une exception, les operandes suivants ne sont pas evalues.
Les effets deja produits ne sont pas annules. &&, || et ?: sont particuliers :
ils choisissent conditionnellement les operandes a evaluer.

  int i = 1;
  int r = i++ + i++;          // r vaut 3, puis i vaut 3

Ce comportement est defini, mais multiplier les modifications dans une seule
expression nuit fortement a la lisibilite.

6.3 Operateurs unaires +, -, ~ et !

  +expressionNumerique
  -expressionNumerique
  ~expressionEntiere
  !expressionBooleenne

+ et - appliquent la promotion unaire : byte, short et char deviennent int.

  byte b = 10;
  int n = -b;                 // -10, de type int

La negation de MIN_VALUE int/long deborde et conserve cette valeur minimale.
Sur float/double, - inverse aussi le signe de zero et de l'infini; NaN reste
NaN.

~ inverse tous les bits d'un int ou long promu :

  ~0                          // -1
  ~0x0F                       // 0xFFFFFFF0

! inverse true et false. Il n'accepte ni entier ni reference; !0 et !null sont
invalides.

6.4 Increment et decrement

Formes prefixees :

  ++variable
  --variable

La variable est modifiee puis la nouvelle valeur est produite :

  int i = 3;
  int j = ++i;                // i = 4, j = 4

Formes postfixees :

  variable++
  variable--

L'ancienne valeur est produite, puis la variable est modifiee :

  int i = 3;
  int j = i++;                // i = 4, j = 3

La variable doit etre numerique, modifiable et non final. Elle n'est evaluee
qu'une fois. La conversion vers son propre type est implicite :

  byte b = 127;
  b++;                        // devient -128

b = b + 1 serait invalide sans cast, puisque b + 1 est int.

6.5 Promotions binaires

Pour les calculs numeriques :

  si un operande est double   les deux deviennent double
  sinon si l'un est float     les deux deviennent float
  sinon si l'un est long      les deux deviennent long
  sinon                       les deux deviennent int

  byte a = 1, b = 2;
  int somme = a + b;
  double total = 1 + 2.5;

boolean ne participe a aucune promotion numerique.

6.6 Multiplication, division et reste

  gauche * droite
  dividende / diviseur
  dividende % diviseur

* multiplie les valeurs promues. Un resultat entier peut deborder sans
exception; un flottant peut devenir infini ou NaN.

La division entiere tronque vers zero :

  7 / 3                      // 2
 -7 / 3                      // -2

Un diviseur entier nul leve ArithmeticException. MIN_VALUE / -1 deborde et
conserve MIN_VALUE.

La division flottante suit IEEE 754 :

  1.0 / 0.0                  // Infinity
 -1.0 / 0.0                  // -Infinity
  0.0 / 0.0                  // NaN

Le reste entier porte le signe du dividende :

  7 % 3                      // 1
 -7 % 3                      // -1

% accepte aussi float/double. Un diviseur flottant nul produit NaN.

6.7 Addition, soustraction et String

Avec des nombres, + et - appliquent les promotions :

  int somme = 2 + 3;
  double ecart = 10 - 0.5;

Si au moins un operande de + est String, + concatene :

  "age=" + 42                // "age=42"
  "actif=" + true            // "actif=true"
  "objet=" + (Object)null    // "objet=null"

Une primitive est convertie en texte. Pour une reference non nulle,
String.valueOf appelle normalement toString(); null devient "null".

+ est associatif a gauche :

  "total=" + 1 + 2           // "total=12"
  "total=" + (1 + 2)         // "total=3"

String est immuable. Une concatenation produit une nouvelle valeur et peut etre
traduite en operations StringBuffer.

6.8 Decalages

  valeur << distance          gauche, zeros a droite
  valeur >> distance          droite, prolongement du signe
  valeur >>> distance         droite, zeros a gauche

Les operandes sont promus. Le resultat prend le type int ou long de l'operande
gauche. Pour int, seuls les 5 bits faibles de la distance sont retenus; pour
long, les 6 bits faibles :

  1 << 3                      // 8
  1 << 32                     // 1, car 32 & 31 == 0
  1L << 32                    // 4294967296L
 -8 >> 1                      // -4
 -1 >>> 1                     // 2147483647

Les bits sortants sont perdus. Une distance negative est elle aussi masquee.
>>> ne cree pas un type unsigned; il effectue seulement un decalage logique.

6.9 Comparaisons ordonnees

  gauche < droite
  gauche <= droite
  gauche > droite
  gauche >= droite

Les operandes doivent etre numeriques et sont promus. Le resultat est boolean.
Si un operande flottant est NaN, toutes ces comparaisons retournent false.

  age >= 18
  0 < x && x < 10            // valide
  0 < x < 10                 // invalide

La derniere forme est invalide car 0 < x produit boolean.

6.10 instanceof

  expressionReference instanceof TypeReference

Retourne true si la valeur non nulle est compatible avec le type :

  objet instanceof String
  animal instanceof Chien
  tableau instanceof Object

null instanceof Type vaut false. Les types doivent etre plausiblement
compatibles. Le test ne change pas le type de la variable :

  if (animal instanceof Chien) {
      Chien chien = (Chien)animal;
  }

Le pattern matching moderne n'existe pas en J++.

6.11 Egalite et identite

  gauche == droite
  gauche != droite

Pour les nombres, les operandes sont promus :

  1 == 1L                    // true
  1 == 1.0                   // true
  0.0 == -0.0                // true
  Double.NaN == Double.NaN   // false

Deux boolean peuvent etre compares, sans conversion vers un entier.

Pour les references, == compare l'identite :

  Point p = new Point(1, 2);
  Point q = p;
  p == q                     // true

  new String("a") == new String("a")       // false en general
  new String("a").equals(new String("a"))  // true

equals() peut comparer le contenu selon le contrat de la classe. Deux
references sont identiques si elles visent le meme objet ou valent toutes deux
null. Les tableaux ne comparent pas automatiquement leurs elements.

6.12 Operateurs &, ^ et |

Sur les entiers, ils travaillent bit par bit :

  0x0F & 0x33                // 0x03
  0x0F ^ 0x33                // 0x3C
  0x0F | 0x30                // 0x3F

| combine souvent des indicateurs Win32 :

  winm.MB_OK | winm.MB_ICONINFORMATION

Sur boolean :

  a & b       true si les deux sont true
  a ^ b       true si exactement un est true
  a | b       true si au moins un est true

Les deux operandes sont toujours evalues. Les operandes doivent etre tous deux
entiers ou tous deux boolean.

6.13 Court-circuit && et ||

  conditionGauche && conditionDroite
  conditionGauche || conditionDroite

Les operandes doivent etre boolean. Pour &&, la droite n'est evaluee que si la
gauche vaut true :

  objet != null && objet.estValide()

Pour ||, la droite n'est evaluee que si la gauche vaut false :

  objet == null || objet.estVide()

Remplacer &&/|| par &/| force l'evaluation des deux cotes et peut modifier les
effets ou exceptions.

6.14 Operateur conditionnel ?:

  condition ? expressionSiVrai : expressionSiFaux

La condition doit etre boolean. Une seule branche est evaluee :

  int absolu = x >= 0 ? x : -x;
  String nom = objet == null ? "aucun" : objet.toString();

Le type resultat vient des deux branches :

  - types identiques : ce type;
  - nombres : promotions et regles de constantes representables;
  - null et reference : le type reference;
  - references : un type commun permis par les conversions;
  - void n'est pas admis comme branche valeur.

?: est associatif a droite.

6.15 Affectation simple

  variable = expression

Le cote gauche doit designer une locale, un parametre, un champ ou un element
de tableau modifiable. Une final deja affectee est interdite. La droite doit
etre compatible par affectation.

Ordre :

  1. evaluation du receveur ou de l'indice gauche;
  2. evaluation de la droite;
  3. conversion autorisee;
  4. stockage;
  5. production de la valeur stockee.

L'affectation est une expression et peut etre chainee :

  a = b = c = 0;

Le cote gauche peut lever NullPointerException,
ArrayIndexOutOfBoundsException ou ArrayStoreException.

6.16 Affectations composees

  +=  -=  *=  /=  %=  &=  ^=  |=  <<=  >>=  >>>=

  variable op= expression

Cette forme equivaut approximativement a :

  variable = (TypeVariable)(variable op expression)

mais variable n'est evaluee qu'une fois. La conversion reductrice est
implicite :

  byte b = 100;
  b += 30;                    // valide, devient -126
  b = b + 30;                // invalide sans cast

Une locale doit deja etre initialisee. += concatene aussi String :

  String s = "a";
  s += 1;                     // "a1"

6.17 Acces aux champs

  expressionReference.champ
  NomClasse.champStatic
  this.champ
  super.champ

Un champ est choisi selon le type compile et n'est pas virtuel. Une reference
null leve NullPointerException pour un champ d'instance.

Un champ static doit etre qualifie par le nom de classe. La forme
objet.champStatic est acceptee historiquement mais trompeuse : le type compile,
non l'objet reel, choisit le champ.

Les controles public, protected, private et package sont effectues a la
compilation.

6.18 Tableaux et length

  expressionTableau[expressionIndice]

La reference est evaluee avant l'indice. L'indice doit etre convertible en int;
long exige un cast. L'acces produit une variable et peut etre a gauche de =,
+= ou ++.

  tableau.length

length est un champ final int, non une methode :

  tableau.length              // valide
  tableau.length()            // invalide

String utilise length(). Une reference null leve NullPointerException; un
indice invalide leve ArrayIndexOutOfBoundsException.

6.19 Invocation et surcharge

  methode(arguments)
  objet.methode(arguments)
  NomClasse.methodeStatic(arguments)
  super.methode(arguments)

Le receveur puis les arguments sont evalues de gauche a droite. Il n'existe ni
argument nomme, ni optionnel, ni varargs.

Le compilateur garde les surcharges applicables et choisit la plus specifique :

  void f(int x)  { ... }
  void f(long x) { ... }

  f(1);                       // f(int)
  f(1L);                      // f(long)

Le type retour ne participe pas au choix. L'implementation d'une methode
d'instance redefinie est ensuite choisie dynamiquement.

6.20 Creation d'objet

  new TypeClasse(arguments)

Les arguments sont evalues de gauche a droite. L'objet est alloue avec ses
champs aux valeurs par defaut, puis les constructeurs et initialisateurs sont
executes :

  Point p = new Point(10, 20);

Une classe abstract ou une interface ne peut etre instanciee directement. Une
classe anonyme combine implementation et creation :

  Runnable r = new Runnable() {
      public void run() {
          travailler();
      }
  };

OutOfMemoryError peut signaler un echec d'allocation. Une exception du
constructeur empeche le retour normal de la reference.

6.21 Creation de tableau

  new Type[dimension] { [dimension] }
  new Type[] { initialisateurs }

  new int[10]
  new String[2][3]
  new int[2][]
  new int[] { 2, 3, 5, 7 }

Les dimensions sont evaluees de gauche a droite et converties en int. Une
dimension negative leve NegativeArraySizeException. Les dimensions vides
suivent les dimensions renseignees :

  new int[2][]                // valide
  new int[][2]                // invalide

Une taille et un initialiseur ne se combinent pas :

  new int[] { 1, 2 }          // valide
  new int[2] { 1, 2 }         // invalide

6.22 Casts

Cast primitif :

  (TypePrimitif)expression

  int n = (int)3.9;           // 3
  byte b = (byte)300;         // 44
  char c = (char)-1;          // '\uffff'

Il peut perdre plage ou precision sans exception. boolean ne se caste pas vers
ou depuis un nombre.

Cast de reference :

  (TypeReference)expressionReference

  Animal animal = obtenirAnimal();
  Chien chien = (Chien)animal;

null reste null; un objet compatible conserve sa reference; un objet
incompatible leve ClassCastException. Le cast ne transforme pas l'objet.

6.23 this, super et Class

this designe l'objet courant :

  this.champ
  this.methode()

Il est interdit en contexte static. this(arguments); appelle un autre
constructeur et doit etre la premiere instruction.

super accede a la superclasse immediate :

  super.champ
  super.methode(arguments)

super.methode force l'implementation heritee. super(arguments); appelle un
constructeur parent et doit etre la premiere instruction si this(...) ne l'est
pas. super seul n'est pas une valeur stockable.

Obtention d'un objet java.lang.Class :

  String.class
  int.class
  int[].class
  objet.getClass()

Type.class ne cree aucune instance. getClass() sur null leve
NullPointerException.

6.24 Expressions constantes

Une expression constante peut employer :

  - litteraux primitifs et String;
  - variables constantes;
  - parentheses et casts primitifs;
  - +, -, ~ et ! unaires;
  - calculs, decalages, comparaisons et egalite;
  - &, ^, |, &&, || et ?:;
  - concatenation de String constantes.

  static final int A = 1 + 2 * 3;
  static final String NOM = "J" + "++";
  case 1 << 3:

new, les appels et effets de bord ne sont pas constants. Une division entiere
constante par zero est une erreur de compilation.

6.25 Exceptions des expressions

  Exception/erreur                    Cause
  ----------------------------------  ------------------------------------------
  ArithmeticException                 / ou % entier avec diviseur zero
  NullPointerException                membre, appel ou tableau via null
  ArrayIndexOutOfBoundsException      indice invalide
  ArrayStoreException                 stockage incompatible dans tableau
  NegativeArraySizeException          dimension negative
  ClassCastException                  cast reference incompatible
  OutOfMemoryError                    echec de new
  ExceptionInInitializerError         initialisation de classe fautive

Un debordement entier, un arrondi flottant ou un cast numerique reducteur ne
leve normalement aucune exception.

6.26 Absence de surcharge des operateurs

Une classe utilisateur ne peut pas redefinir +, -, *, == ou un autre operateur.
String beneficie d'un traitement integre pour + et +=, non reproductible.

  montant.ajouter(autre)      // methode valide
  montant + autre             // invalide pour une classe utilisateur

Pour une egalite logique, une classe redefinit equals() et hashCode().

6.27 Expressions propres aux delegates J++

Les extensions Microsoft ajoutent une reference liee a une methode :

  Notification n = new Notification(objet.traiter);
  n.invoke(source, code);

La signature de traiter doit correspondre au delegate. objet.traiter n'est pas
une expression Java 1.1 standard et demande les extensions J++.

Un multicast delegate peut contenir plusieurs cibles selon le runtime. Ne pas
supposer que les operateurs + et - de C# sur les delegates ont la meme
semantique en J++.

6.28 Exemple recapitulatif

  public int calculer(Object objet, int[] valeurs, boolean absolu) {
      if (objet == null || !(objet instanceof Mesure))
          throw new IllegalArgumentException("objet invalide");

      Mesure mesure = (Mesure)objet;
      int total = mesure.getValeur();

      for (int i = 0; i < valeurs.length; i++)
          total += valeurs[i] * (i + 1);

      int resultat = absolu && total < 0 ? -total : total;
      return resultat > Mesure.LIMITE ? Mesure.LIMITE : resultat;
  }

L'exemple combine identite, court-circuit, instanceof, cast, appel, tableau,
increment, calculs, affectation composee, ?: et champ static.


================================================================================
7. INSTRUCTIONS
================================================================================

7.1 Classification et grammaire generale

Une instruction commande une action ou un transfert de controle. Elle ne
produit pas une valeur directement utilisable par une expression englobante.

Forme simplifiee :

  instruction ::=
      bloc
    | ;
    | expressionInstruction ;
    | declarationLocale ;
    | if (expression) instruction [else instruction]
    | switch (expression) blocSwitch
    | while (expression) instruction
    | do instruction while (expression) ;
    | for (initialisation ; condition ; miseAJour) instruction
    | etiquette : instruction
    | break [etiquette] ;
    | continue [etiquette] ;
    | return [expression] ;
    | throw expression ;
    | synchronized (expression) bloc
    | try bloc catchEtOuFinally

Une instruction peut se terminer normalement, continuer vers une autre
instruction, ou se terminer brutalement par break, continue, return, throw ou
une exception. Le mot goto est reserve mais ne possede aucune instruction.

J++ historique ne comprend ni assert, ni foreach, ni try-with-resources, ni
switch sur String, ni expressions switch.

7.2 Bloc

Syntaxe :

  {
      { declarationLocale | instruction }
  }

Un bloc regroupe zero ou plusieurs declarations et instructions :

  {
      int total = 0;
      total += calculer();
      afficher(total);
  }

Il peut remplacer une instruction simple apres if, else, while, do, for,
synchronized ou une etiquette. Il cree une portee locale. Une variable
declaree dans le bloc n'est plus visible apres sa fermeture.

Les instructions sont executees dans l'ordre textuel, sauf transfert de
controle. Un bloc vide est valide :

  { }

Les accolades sont recommandees meme pour une seule instruction, afin d'eviter
les erreurs lors d'une modification ulterieure.

7.3 Instruction vide

Syntaxe :

  ;

Elle ne fait rien et se termine normalement. Elle peut servir de corps a une
boucle dont tout le travail est effectue dans la condition ou la mise a jour :

  while (lireEncore())
      ;

Un point-virgule accidentel peut toutefois detacher un bloc :

  if (condition);             // instruction if deja terminee
  {
      executer();             // bloc toujours execute
  }

Cette construction est valide mais souvent erronee.

7.4 Instructions-expression

Seules certaines expressions peuvent etre employees comme instructions :

  - affectation;
  - pre-increment et pre-decrement;
  - post-increment et post-decrement;
  - invocation de methode;
  - creation d'une instance de classe.

Exemples :

  x = 10;
  compteur++;
  --reste;
  objet.executer();
  new Travail();

Les expressions purement calculatoires ne sont pas des instructions :

  x + y;                      // invalide
  objet.champ;                // invalide
  true;                       // invalide

La valeur produite par une expression-instruction est ignoree. Les effets et
exceptions restent applicables.

7.5 Declaration locale

Syntaxe :

  [final] Type nom [= initialiseur]
      {, nom [= initialiseur]} ;

Exemples :

  int total;
  int x = 1, y = 2;
  final String titre = "J++";
  int[] valeurs = new int[10];

Une declaration locale est permise dans un bloc et dans l'initialisation d'un
for. La variable doit etre definitivement affectee avant lecture. Sa portee
commence apres son declarateur et se termine avec le bloc applicable.

Les modificateurs d'acces, static, transient et volatile ne sont pas permis sur
une locale. J++ 6.0 n'accepte pas var.

Une declaration n'est pas une instruction partout ou une instruction simple
est attendue. Pour declarer apres if sans bloc, ajouter des accolades :

  if (condition) {
      int valeur = calculer();
      utiliser(valeur);
  }

7.6 Declaration de classe locale

Une classe peut etre declaree dans un bloc :

  void construire() {
      class Tache implements Runnable {
          public void run() {
              travailler();
          }
      }

      new Thread(new Tache()).start();
  }

Son nom est visible de sa declaration a la fin du bloc. Elle n'est pas un
membre de la classe englobante et ne peut etre public, protected, private ou
static.

Une classe locale Java 1.1 peut utiliser les variables locales englobantes
seulement si elles sont final. Elle peut aussi acceder aux champs de l'instance
englobante.

7.7 if

Syntaxe :

  if (conditionBooleenne)
      instruction

  if (conditionBooleenne)
      instructionSiVrai
  else
      instructionSiFaux

La condition est evaluee une fois et doit etre boolean :

  if (age >= 18) {
      categorie = "adulte";
  } else {
      categorie = "mineur";
  }

Il n'existe aucune conversion implicite int vers boolean :

  if (1) { ... }              // invalide

Une seule branche est executee. Si la condition leve une exception, aucune
branche ne commence.

7.8 Association du else

Sans accolades, else est associe au if sans else le plus proche :

  if (a)
      if (b)
          f();
      else
          g();

equivaut a :

  if (a) {
      if (b)
          f();
      else
          g();
  }

Pour rattacher else au premier if, utiliser un bloc explicite. Cette regle est
souvent appelee le probleme du dangling else.

Une chaine else if n'est pas une instruction speciale : le else contient
simplement un autre if :

  if (x < 0) {
      signe = -1;
  } else if (x > 0) {
      signe = 1;
  } else {
      signe = 0;
  }

7.9 switch

Syntaxe :

  switch (selecteur) {
      case constante1:
          instructions
      case constante2:
          instructions
      default:
          instructions
  }

Le selecteur est evalue une seule fois. Son type doit etre char, byte, short ou
int, apres les conversions autorisees. long, boolean, float, double, String,
enum et Object ne sont pas acceptes en J++ 6.0.

Exemple :

  switch (code) {
      case 1:
          nom = "un";
          break;
      case 2:
          nom = "deux";
          break;
      default:
          nom = "autre";
  }

Si un case correspond, l'execution commence apres son etiquette. Sinon elle
commence au default eventuel. En l'absence de correspondance et de default, le
switch ne fait rien.

7.10 Etiquettes case et default

Syntaxes :

  case expressionConstante :
  default :

Chaque case doit etre une expression constante representable dans le type du
selecteur :

  final int OUVRIR = 1;

  switch (commande) {
      case OUVRIR:
          ouvrir();
          break;
      case 2 + 3:
          fermer();
          break;
  }

Deux case ayant la meme valeur apres conversion sont interdits. Un seul default
est permis. default peut apparaitre a n'importe quelle position dans le bloc;
sa position n'affecte pas la recherche initiale mais affecte la poursuite
sequentielle.

case n'est pas une etiquette generale et ne peut apparaitre hors d'un switch.

7.11 Poursuite entre case

L'execution ne s'arrete pas automatiquement a la prochaine etiquette :

  switch (jour) {
      case 6:
      case 7:
          type = "week-end";
          break;
      default:
          type = "semaine";
  }

Les case 6 et 7 partagent ici le meme traitement. Cette poursuite, appelee
fall-through, peut aussi etre volontaire :

  switch (niveau) {
      case 3:
          droits += ECRITURE;
      case 2:
          droits += LECTURE;
      case 1:
          droits += CONNEXION;
  }

Sans commentaire clair, un break oublie est une source frequente d'erreurs.
return ou throw terminent egalement la branche.

7.12 while

Syntaxe :

  while (conditionBooleenne)
      instruction

La condition est testee avant chaque iteration :

  while (reste > 0) {
      traiter();
      reste--;
  }

Si elle vaut false au premier test, le corps ne s'execute jamais. Si elle vaut
true, le corps s'execute puis la condition est reevaluee.

continue passe directement au prochain test. break termine la boucle.

  while (true) {
      if (termine())
          break;
      travailler();
  }

Une boucle while(true) sans break atteignable ne peut pas se terminer
normalement selon l'analyse de controle.

7.13 do-while

Syntaxe :

  do
      instruction
  while (conditionBooleenne) ;

Le corps est execute avant le premier test :

  do {
      ligne = lireLigne();
      if (ligne != null)
          traiter(ligne);
  } while (ligne != null);

Il s'execute donc au moins une fois. Le point-virgule apres la parenthese est
obligatoire.

continue transfere au test de la condition. Si elle vaut true, une nouvelle
iteration commence; sinon la boucle se termine.

7.14 for

Syntaxe :

  for ([initialisation] ; [condition] ; [miseAJour])
      instruction

Exemple :

  for (int i = 0; i < 10; i++) {
      System.out.println(i);
  }

Ordre d'execution :

  1. initialisation, une seule fois;
  2. test de condition, si elle existe;
  3. corps si la condition vaut true;
  4. mise a jour;
  5. retour au test.

Une condition omise equivaut a true :

  for (;;) {
      travailler();
  }

Les trois zones peuvent etre vides, mais les deux points-virgules sont
obligatoires.

7.15 Initialisation et mise a jour du for

L'initialisation est soit une declaration locale, soit une liste
d'expressions-instructions :

  for (int i = 0; i < n; i++) { ... }
  for (i = 0, j = n - 1; i < j; i++, j--) { ... }

Une declaration ne peut melanger plusieurs types :

  for (int i = 0, j = 0; ... )     // valide
  for (int i = 0, long j = 0; ... ) // invalide

La mise a jour accepte une liste d'expressions-instructions separees par des
virgules. L'operateur virgule general n'existe pas :

  i++, j--                    // valide dans la zone de mise a jour
  x = (i++, j--);             // invalide comme expression Java

La portee d'une variable declaree par for couvre la condition, la mise a jour
et le corps, mais pas l'instruction suivant la boucle.

continue execute la mise a jour avant de retester la condition.

7.16 Absence de foreach

La boucle amelioree introduite par Java 5 n'existe pas :

  for (String s : tableau)    // invalide en J++ 6.0

Pour un tableau, utiliser les indices :

  for (int i = 0; i < tableau.length; i++) {
      String s = tableau[i];
      traiter(s);
  }

Pour Vector et les collections historiques, utiliser Enumeration :

  for (Enumeration e = vecteur.elements(); e.hasMoreElements();) {
      Object element = e.nextElement();
      traiter(element);
  }

Iterator et le Collections Framework sont posterieurs a Java 1.1.

7.17 Instruction etiquetee

Syntaxe :

  etiquette : instruction

Exemple :

  externe:
  for (int i = 0; i < n; i++) {
      for (int j = 0; j < m; j++) {
          if (fini(i, j))
              break externe;
      }
  }

Une etiquette est un identificateur et possede son propre espace de noms. Elle
peut donc porter le meme texte qu'une variable sans la designer.

Sa portee est l'instruction qui la suit. Une etiquette peut viser un bloc, un
if, un switch ou une boucle. Elle n'est pas une destination goto : on ne peut
que sortir avec break, ou continuer si elle designe une boucle.

7.18 break

Syntaxes :

  break ;
  break etiquette ;

Sans etiquette, break termine la boucle while, do, for ou le switch englobant
le plus proche :

  while (true) {
      if (termine())
          break;
  }

Avec etiquette, il termine l'instruction etiquetee correspondante :

  recherche: {
      if (erreur)
          break recherche;
      executer();
  }

L'etiquette doit etre dans une portee englobante. break ne peut entrer dans un
bloc ni viser une methode. Les blocs finally rencontres sont executes avant le
transfert effectif.

7.19 continue

Syntaxes :

  continue ;
  continue etiquette ;

Sans etiquette, continue passe a l'iteration suivante de la boucle la plus
proche. Avec etiquette, la cible doit etre une boucle englobante :

  externe:
  for (int i = 0; i < n; i++) {
      for (int j = 0; j < m; j++) {
          if (ignorer(i, j))
              continue externe;
          traiter(i, j);
      }
  }

Dans while, le transfert va au test. Dans do, il va au test final. Dans for, il
va d'abord a la mise a jour puis au test.

continue ne peut cibler un simple bloc ou un switch. Les finally traverses sont
executes.

7.20 return

Syntaxes :

  return ;
  return expression ;

return; termine une methode void ou un constructeur. return expression;
termine une methode non void et fournit une valeur compatible avec son type :

  int carre(int x) {
      return x * x;
  }

Une methode non void doit pouvoir retourner une valeur sur toute route qui se
termine normalement. Le type retour seul n'autorise aucune conversion
reductrice implicite non permise.

Dans un constructeur, return; est possible mais aucune valeur ne peut etre
retournee. La reference creee est produite par l'expression new, pas par une
instruction return du constructeur.

Avant le retour effectif, les blocs finally traverses sont executes. Si un
finally effectue lui-meme return ou throw, il remplace le retour initial.

7.21 throw

Syntaxe :

  throw expressionThrowable ;

L'expression doit etre assignable a Throwable :

  throw new IllegalArgumentException("age negatif");
  throw erreur;

L'objet est evalue puis l'execution cherche un gestionnaire catch compatible en
remontant les appels. Les finally traverses s'executent. Si aucun gestionnaire
n'existe, le thread se termine selon le mecanisme d'exception de la JVM.

throw null; est compilable comme reference Throwable mais provoque
NullPointerException a l'execution.

Une exception verifiee doit etre capturee ou declaree dans throws. Error et
RuntimeException, ainsi que leurs sous-classes, sont non verifiees.

7.22 try et catch

Syntaxe :

  try {
      instructionsProtegees
  } catch (Type1 e) {
      traitement1
  } catch (Type2 e) {
      traitement2
  }

Le bloc try s'execute d'abord. Si une exception en sort, les catch sont testes
dans l'ordre textuel. Le premier dont le type accepte l'objet est execute.

  try {
      lire();
  } catch (FileNotFoundException e) {
      fichierAbsent(e);
  } catch (IOException e) {
      erreurLecture(e);
  }

Les types les plus precis doivent preceder les types generaux. Placer
Exception avant IOException rendrait le second catch inaccessible.

Le parametre catch est une variable locale dont la portee est son bloc. Il peut
etre final. J++ ne permet ni multi-catch avec |, ni capture sans parametre.

7.23 finally

Syntaxe :

  try {
      instructions
  } finally {
      nettoyage
  }

ou :

  try {
      instructions
  } catch (Exception e) {
      traitement
  } finally {
      nettoyage
  }

finally s'execute lors d'une sortie normale ou brutale du try/catch :

  - fin normale;
  - exception;
  - break;
  - continue;
  - return.

Il sert au nettoyage :

  FileInputStream f = new FileInputStream(nom);
  try {
      lire(f);
  } finally {
      f.close();
  }

Une fin brutale de la JVM peut empecher son execution. Un return, break,
continue ou throw dans finally remplace le transfert deja en cours et peut
masquer une exception; cette pratique est fortement deconseillee.

J++ historique ne possede pas try-with-resources.

7.24 Combinaisons valides de try

Un try doit etre suivi :

  - d'au moins un catch;
  - d'un finally;
  - ou de catch puis finally.

Formes valides :

  try { ... } catch (Exception e) { ... }
  try { ... } finally { ... }
  try { ... } catch (Exception e) { ... } finally { ... }

Forme invalide :

  try { ... }                 // ni catch ni finally

Un catch appartient uniquement au try qui le precede. Pour proteger aussi le
code d'un catch par un autre gestionnaire, imbriquer un nouveau try.

Une exception levee dans finally ne peut pas etre capturee par les catch du
meme try, car ceux-ci protegent seulement le bloc try initial.

7.25 synchronized

Bloc synchronise :

  synchronized (expressionReference) {
      instructions
  }

L'expression est evaluee une fois. Elle doit produire une reference non nulle.
Le thread acquiert le moniteur de l'objet avant le bloc et le libere lors de
toute sortie, normale ou brutale :

  synchronized (verrou) {
      compteur++;
  }

Une reference null leve NullPointerException avant l'entree. L'acquisition peut
attendre qu'un autre thread libere le moniteur.

Une methode synchronized est une declaration de methode, non une instruction :

  public synchronized void incrementer() {
      compteur++;
  }

Elle verrouille this; une methode static synchronized verrouille l'objet Class.
synchronized est reentrant : un thread possedant deja le moniteur peut
l'acquerir a nouveau.

7.26 Attente et notification

Object fournit wait(), notify() et notifyAll(). Le thread doit posseder le
moniteur :

  synchronized (file) {
      while (file.estVide())
          file.wait();
      element = file.retirer();
  }

wait libere temporairement le moniteur et suspend le thread. Avant de revenir,
le thread doit reacquerir ce moniteur. La condition est retestee dans une boucle
afin de tenir compte des autres threads et des reveils.

notify reveille un thread en attente choisi par la JVM; notifyAll les rend tous
eligibles. Ces appels ne liberent pas immediatement le verrou : il l'est a la
sortie du bloc synchronized.

wait peut lever InterruptedException, exception verifiee a capturer ou
declarer.

7.27 Appels explicites de constructeurs

Dans un constructeur seulement :

  this(arguments);
  super(arguments);

L'une de ces instructions peut etre la premiere instruction du corps :

  Point(int x, int y) {
      this(x, y, 0);
  }

  Point3D(int x, int y, int z) {
      super(x, y);
      this.z = z;
  }

Si aucune n'est ecrite, super(); est insere implicitement. this(...) appelle un
autre constructeur de la meme classe; super(...) un constructeur de la
superclasse directe.

Elles ne peuvent apparaitre ailleurs, ni etre precedees par une instruction.
Une chaine this(...) doit finir par un super(...) explicite ou implicite et ne
peut etre cyclique.

7.28 Accessibilite des instructions

Le compilateur rejette en general une instruction qu'aucun chemin d'execution
ne peut atteindre :

  return;
  executer();                 // inaccessible

De meme :

  while (true) {
  }
  executer();                 // inaccessible sans sortie possible

Les regles formelles ne constituent pas une preuve generale de toutes les
conditions. Par exemple, if (false) peut etre traite selon les regles speciales
permettant la compilation conditionnelle de code, alors qu'une instruction
apres return reste interdite.

L'accessibilite sert aussi a verifier qu'une methode non void ne peut atteindre
la fin sans return et qu'un break est susceptible de permettre a une boucle de
se terminer normalement.

7.29 Affectation certaine et controle

Une variable locale doit etre affectee sur tous les chemins avant sa lecture :

  int valeur;
  if (condition)
      valeur = 1;
  else
      valeur = 2;
  afficher(valeur);           // valide

Sans else, la lecture pourrait etre invalide. Le compilateur combine les routes
de if, switch, boucles, break, continue, return, throw et try/finally.

Une variable final doit inversement etre definitivement non affectee avant son
unique affectation. Les chemins de constructeurs sont analyses pour garantir
l'initialisation des blank final.

7.30 Imbrication et transfert

Les instructions peuvent etre imbriquees librement :

  for (int i = 0; i < lignes; i++) {
      try {
          if (ignorer(i))
              continue;
          traiter(i);
      } catch (Exception e) {
          journaliser(e);
          break;
      } finally {
          nettoyer(i);
      }
  }

Dans cet exemple :

  - continue execute enfin nettoyer(i), puis passe a la mise a jour du for;
  - une exception capturee execute le catch;
  - break execute nettoyer(i), puis termine la boucle;
  - une exception non capturee execute nettoyer(i), puis continue sa remontee.

Un transfert ne peut pas entrer dans un bloc, un try ou une boucle depuis
l'exterieur. Les etiquettes permettent seulement de sortir ou de continuer une
boucle englobante.

7.31 Instructions absentes de J++ historique

Les constructions suivantes n'existent pas dans Visual J++ 6.0 :

  assert expression;                  ajoute a Java 1.4
  for (Type x : collection)           ajoute a Java 5
  try (Ressource r = ...) { ... }     ajoute a Java 7
  catch (E1 | E2 e) { ... }           ajoute a Java 7
  switch (chaine)                     ajoute a Java 7
  case valeur -> expression           switch moderne
  yield expression                    switch expression moderne

goto et const sont reserves mais sans instruction. Il n'existe pas non plus de
defer, using, with, repeat-until ou instruction de propriete C#.

Les memes resultats doivent etre obtenus avec les constructions Java 1.1 :
try/finally, Enumeration, if/else, boucles classiques et methodes.

7.32 Exemple recapitulatif

  public int chercher(int[][] matrice, int cible)
      throws InterruptedException {

      int resultat = -1;

      recherche:
      for (int ligne = 0; ligne < matrice.length; ligne++) {
          if (matrice[ligne] == null)
              continue;

          synchronized (matrice[ligne]) {
              for (int colonne = 0;
                   colonne < matrice[ligne].length;
                   colonne++) {
                  try {
                      if (matrice[ligne][colonne] != cible)
                          continue;

                      resultat = ligne;
                      break recherche;
                  } finally {
                      journaliser(ligne, colonne);
                  }
              }
          }
      }

      return resultat;
  }

L'exemple montre une declaration locale, deux for, if, continue, une etiquette,
break etiquete, synchronized, try/finally et return. Le finally est execute
avant le break recherche.


================================================================================
8. CLASSES ET OBJETS
================================================================================

8.1 Declaration generale

Syntaxe :

  [modificateurs] class Nom
      [extends SuperClasse]
      [implements Interface1 {, InterfaceN}] {
      membres
  }

Exemple :

  public class Point extends Object implements Dessinable {
      private int x;
      private int y;

      public Point(int x, int y) {
          this.x = x;
          this.y = y;
      }

      public int getX() {
          return x;
      }

      public void dessiner() {
          System.out.println("(" + x + "," + y + ")");
      }
  }

Nom declare un nouveau type reference. Une classe etend au plus une classe et
peut implementer plusieurs interfaces. Sans extends explicite, elle derive de
java.lang.Object, sauf Object lui-meme.

Une declaration peut etre de premier niveau, membre d'une autre classe, locale
a un bloc ou anonyme dans une expression new. Les regles propres aux classes
internes sont detaillees a la section 9.

8.2 Modificateurs d'une classe de premier niveau

Une classe de premier niveau Java 1.1 accepte principalement :

  public       accessible depuis tout package autorise
  abstract     classe incomplete, non instanciable directement
  final        classe qui ne peut avoir de sous-classe

Sans public, la classe est accessible uniquement dans son package. private,
protected et static sont interdits sur une classe de premier niveau, mais
peuvent s'appliquer a certains types membres.

abstract et final sont incompatibles : une classe abstract demande une
sous-classe pour completer son implementation, alors qu'une classe final
interdit toute sous-classe.

J++ est anterieur a strictfp, aux annotations, aux classes sealed, aux records
et aux enums.

8.3 Corps et membres

Le corps d'une classe peut contenir :

  - champs d'instance;
  - champs static;
  - methodes d'instance;
  - methodes static;
  - constructeurs;
  - blocs d'initialisation d'instance;
  - blocs static;
  - classes et interfaces membres;
  - declarations vides ';'.

Exemple :

  class Exemple {
      static int nombre;
      int valeur;

      static {
          nombre = 0;
      }

      {
          valeur = 1;
      }

      Exemple() {
          nombre++;
      }

      void executer() {
          valeur++;
      }

      class Interne { }
  }

L'ordre textuel influence l'initialisation des champs et des blocs, mais pas la
possibilite generale d'appeler une methode declaree plus bas.

8.4 Classes et objets

Une classe est une description de structure et de comportement. Un objet est
une instance creee a l'execution :

  Point p;                    // variable reference, aucun objet
  p = new Point(10, 20);      // creation et affectation

La variable p contient une reference, non l'objet lui-meme. Plusieurs variables
peuvent designer le meme objet :

  Point q = p;
  q.deplacer(1, 0);           // p observe aussi le Point modifie

Affecter null ou une autre reference a q ne detruit pas l'objet tant que p ou
une autre reference le rend atteignable.

Chaque objet possede ses champs d'instance et un moniteur utilise par
synchronized, wait, notify et notifyAll. Les champs static appartiennent a la
classe chargee et sont partages.

8.5 Creation avec new

Syntaxe :

  new NomClasse([arguments])

Exemple :

  Point p = new Point(10, 20);

Le compilateur choisit un constructeur accessible dont les parametres sont
compatibles avec les arguments. Ceux-ci sont evalues de gauche a droite.

Etapes conceptuelles :

  1. chargement, liaison et initialisation de la classe si necessaire;
  2. evaluation des arguments;
  3. allocation de l'objet;
  4. initialisation de tous les champs a leur valeur par defaut;
  5. execution du constructeur de superclasse;
  6. initialisateurs de champs et blocs d'instance;
  7. corps du constructeur;
  8. production de la reference.

Une classe abstract ne peut etre creee directement. Une allocation peut lever
OutOfMemoryError. Une exception du constructeur empeche new de retourner
normalement la reference.

8.6 Constructeurs

Syntaxe :

  [public | protected | private] NomClasse([parametres])
      [throws types] {
      instructions
  }

Un constructeur porte exactement le nom simple de la classe et n'a aucun type
retour, pas meme void :

  class Point {
      Point(int x, int y) {
          ...
      }
  }

Ecrire void Point(...) declare une methode ordinaire nommee Point, pas un
constructeur.

Les constructeurs ne sont ni herites, ni redefinis. Ils peuvent etre
surcharges. Leur visibilite controle qui peut creer l'objet ou appeler le
constructeur depuis une sous-classe.

Un constructeur peut declarer throws. Il ne peut etre abstract, static, final,
synchronized, native ou transient.

8.7 Constructeur par defaut

Si une classe ne declare aucun constructeur, le compilateur fournit un
constructeur sans argument :

  class Vide {
  }

equivaut approximativement a :

  class Vide {
      Vide() {
          super();
      }
  }

Sa visibilite est celle de la classe. Il appelle le constructeur sans argument
de la superclasse. Si celui-ci n'existe pas ou n'est pas accessible, la classe
ne compile pas.

Des qu'un constructeur quelconque est declare, aucun constructeur par defaut
n'est ajoute :

  class A {
      A(int valeur) { }
  }

  new A();                     // invalide

Il faut declarer explicitement A() si cette creation est souhaitee.

8.8 Appels this(...) et super(...)

La premiere instruction d'un constructeur peut etre :

  this(arguments);            // autre constructeur de la meme classe
  super(arguments);           // constructeur de la superclasse directe

Si aucune forme n'est presente, super(); est insere.

  Point(int valeur) {
      this(valeur, valeur);
  }

  Point(int x, int y) {
      super();
      this.x = x;
      this.y = y;
  }

Une seule de ces formes peut apparaitre et elle doit etre la premiere
instruction. Une chaine this(...) doit finir par un constructeur appelant
super(...) explicitement ou implicitement. Un cycle entre constructeurs est
interdit.

Les arguments de super(...) ne peuvent employer l'objet avant l'initialisation
de la superclasse selon les restrictions du langage.

8.9 Initialisation d'une instance

Avant le constructeur, les champs valent false, zero ou null. Apres le retour
du constructeur de la superclasse, les initialisateurs de champs et blocs
d'instance de la classe courante s'executent dans l'ordre textuel :

  class Exemple extends Base {
      int a = calculerA();

      {
          preparer();
      }

      int b = calculerB();

      Exemple() {
          terminer();
      }
  }

Ordre dans Exemple :

  1. constructeur Base;
  2. calculerA();
  3. preparer();
  4. calculerB();
  5. terminer().

Lorsqu'un constructeur appelle this(...), ces initialisateurs ne sont executes
qu'une fois par la chaine de construction.

Appeler une methode redefinissable depuis un constructeur est dangereux :
l'implementation d'une sous-classe peut s'executer avant l'initialisation de
ses propres champs.

8.10 Initialisation d'une classe

Les champs static prennent d'abord leur valeur par defaut. La superclasse est
initialisee, puis les initialisateurs static et blocs static de la classe
s'executent dans l'ordre textuel :

  class Configuration {
      static String nom = chargerNom();

      static {
          verifier(nom);
      }
  }

L'initialisation est normalement declenchee par une premiere utilisation
active : new, appel static ou acces a un champ static non constant. La simple
mention d'un type ou l'utilisation d'une constante incorporee peut ne pas la
declencher.

Une classe est initialisee au plus une fois pour un chargeur de classes. La JVM
synchronise cette operation. Une panne peut produire
ExceptionInInitializerError, puis rendre les utilisations ulterieures fautives.

8.11 Champs

Champ d'instance :

  [visibilite] [final] [transient] [volatile]
  Type nom [= initialiseur] ;

Champ de classe :

  [visibilite] static [final] [transient] [volatile]
  Type nom [= initialiseur] ;

Chaque objet possede ses champs d'instance. Un champ static existe une fois par
classe chargee.

  class Compteur {
      private int valeur;
      private static int nombreObjets;
  }

Les champs sont resolus statiquement et ne sont pas polymorphes :

  class A { int n = 1; }
  class B extends A { int n = 2; }

  A a = new B();
  a.n                          // 1

Le champ B.n masque A.n mais ne le remplace pas. super.n permet l'acces au
champ de la superclasse depuis B.

Les declarations et modificateurs de champs sont detailles a la section 5.

8.12 Methodes

Syntaxe :

  [modificateurs] TypeRetour nom([parametres])
      [throws types] {
      instructions
  }

ou, sans corps :

  [modificateurs] abstract TypeRetour nom(parametres)
      [throws types] ;

  [modificateurs] native TypeRetour nom(parametres)
      [throws types] ;

Exemple :

  public int distanceX(Point autre) {
      return x - autre.x;
  }

Une methode void ne produit aucune valeur. Une methode non void doit retourner
une valeur compatible sur toute route qui peut atteindre sa fin.

Les parametres sont passes par valeur. La signature pertinente pour la
surcharge comprend le nom et les types des parametres; le type retour n'est pas
suffisant pour distinguer deux declarations.

8.13 Modificateurs de methodes

  public/protected/private  controle d'acces
  aucun                     acces de package
  static                    methode de classe
  abstract                  aucune implementation dans cette classe
  final                     interdiction de redefinition
  synchronized              acquisition automatique d'un moniteur
  native                    implementation externe

Combinaisons incompatibles courantes :

  abstract avec private     aucune sous-classe ne pourrait implementer
  abstract avec static      pas de liaison dynamique d'instance
  abstract avec final       redefinition interdite
  abstract avec native      deux formes differentes d'absence de corps
  abstract avec synchronized aucun corps a proteger

Une methode static n'a ni this ni acces direct aux champs d'instance. Une
methode synchronized d'instance verrouille this; si elle est static, elle
verrouille l'objet Class.

8.14 Surcharge

Plusieurs methodes ou constructeurs peuvent porter le meme nom si leurs listes
de parametres different :

  void afficher(int valeur) { ... }
  void afficher(long valeur) { ... }
  void afficher(String valeur) { ... }

  Point() { ... }
  Point(int x, int y) { ... }

Ne suffit pas :

  int calculer() { ... }
  long calculer() { ... }       // invalide : seul le retour differe

Lors d'un appel, le compilateur :

  1. rassemble les declarations accessibles;
  2. conserve celles applicables aux arguments;
  3. choisit la plus specifique;
  4. signale une ambiguite si aucun meilleur choix unique n'existe.

J++ historique n'utilise ni autoboxing ni varargs dans cette resolution.

8.15 Heritage

Syntaxe :

  class Fille extends Mere {
      ...
  }

Une classe n'a qu'une superclasse directe. L'heritage multiple de classes est
interdit. Plusieurs interfaces permettent de combiner des contrats :

  class Fenetre extends Composant
      implements Dessinable, Serializable {
      ...
  }

Une sous-classe herite des membres accessibles de sa superclasse, mais pas de
ses constructeurs. Les membres private existent dans la partie superclasse de
l'objet, sans etre directement accessibles par leur nom dans la sous-classe.

Un cycle est interdit :

  class A extends B { }
  class B extends A { }          // invalide

Object est la racine commune et n'a pas de superclasse.

8.16 Redefinition de methodes

Une methode d'instance d'une sous-classe redefinit une methode heritee de meme
signature :

  class Animal {
      public void parler() {
          System.out.println("...");
      }
  }

  class Chien extends Animal {
      public void parler() {
          System.out.println("Ouaf");
      }
  }

En Java 1.1, le type retour doit correspondre exactement; les retours covariants
sont posterieurs. La redefinition ne peut pas reduire la visibilite :

  public -> public
  protected -> protected ou public
  package -> package, protected ou public selon l'accessibilite

Une methode redefinie ne peut declarer de nouvelles exceptions verifiees plus
larges que celles de la methode parente. Elle peut en declarer moins ou des
sous-types compatibles.

Une methode final ou private n'est pas redefinissable. Une methode private de
meme signature dans la sous-classe est une nouvelle methode sans relation de
redefinition.

8.17 Liaison dynamique

Le type compile controle les membres visibles; le type reel choisit
l'implementation d'une methode d'instance redefinie :

  Animal animal = new Chien();
  animal.parler();             // Chien.parler()

Ce mecanisme est le polymorphisme dynamique. Il s'applique meme si la reference
est de type interface.

Il ne s'applique pas :

  - aux champs;
  - aux methodes static;
  - aux methodes private;
  - aux constructeurs;
  - aux initialisateurs.

La signature de la methode est d'abord resolue a la compilation. La JVM choisit
ensuite l'implementation redefinie de cette signature; elle ne recommence pas
la resolution des surcharges avec les types reels des arguments.

8.18 Masquage des methodes static

Une methode static de sous-classe masque une methode static de meme signature :

  class A {
      static void nom() { System.out.println("A"); }
  }

  class B extends A {
      static void nom() { System.out.println("B"); }
  }

  A reference = new B();
  reference.nom();             // A.nom(), selon le type compile
  B.nom();                     // B.nom()

Une methode static ne peut masquer une methode d'instance de meme signature, ni
l'inverse. Il est recommande de qualifier toute methode static avec le nom de
classe.

8.19 super et acces au comportement herite

Dans une methode d'instance :

  super.champ
  super.methode(arguments)

super.methode appelle l'implementation selectionnee dans la superclasse
immediate :

  class Chien extends Animal {
      public void parler() {
          super.parler();
          System.out.println("Ouaf");
      }
  }

super n'est pas une reference ordinaire que l'on peut affecter ou transmettre.
L'appel peut continuer plus haut dans la hierarchie si la superclasse emploie
elle-meme super.

Dans un constructeur, super(...) a le role distinct d'appel explicite au
constructeur parent.

8.20 this

this est la reference vers l'instance courante :

  this.x = x;
  this.deplacer(1, 2);

Il permet notamment de distinguer un champ d'un parametre de meme nom. Il peut
etre retourne ou transmis :

  Point identite() {
      return this;
  }

this est interdit dans une methode ou un bloc static. Publier this depuis un
constructeur peut exposer un objet partiellement initialise.

Dans une classe interne, NomExterne.this peut designer l'instance englobante
appropriee.

8.21 Classes abstract

Une classe abstract ne peut etre instanciee directement :

  abstract class Forme {
      public abstract double aire();

      public void afficher() {
          System.out.println(aire());
      }
  }

Elle peut contenir champs, constructeurs, methodes concretes et methodes
abstraites. Ses constructeurs sont appeles par les sous-classes.

Une classe contenant une methode abstract doit etre abstract. Une sous-classe
qui ne fournit pas toutes les implementations requises doit rester abstract.

Une classe peut etre declaree abstract sans methode abstraite, par exemple pour
interdire son instanciation et reserver son role a l'heritage.

8.22 Classes et methodes final

Classe final :

  final class Utilitaire {
      ...
  }

Aucune classe ne peut l'etendre. String est un exemple historique de classe
final.

Methode final :

  public final void verifier() {
      ...
  }

Une sous-classe ne peut la redefinir. final peut stabiliser un invariant ou
permettre certaines optimisations, mais ne rend pas automatiquement les objets
immuables.

Un champ final est une variable affectable une fois; cette notion est distincte
d'une classe ou methode final.

8.23 Classes imbriquees et internes

Une classe membre peut etre static :

  class Externe {
      static class Imbriquee {
      }
  }

Elle n'a pas de reference implicite vers une instance Externe.

Une classe membre non static est une classe interne :

  class Externe {
      class Interne {
          void f() {
              Externe.this.executer();
          }
      }
  }

Elle conserve une reference vers l'instance englobante. Java 1.1 permet aussi
les classes locales et anonymes. Une variable locale capturee doit etre final;
la notion moderne "effectivement finale" n'existe pas.

La section 9 detaille les interfaces et toutes les formes de classes internes.

8.24 Classes anonymes

Syntaxe :

  new SuperClasse(arguments) {
      membres
  }

ou :

  new Interface() {
      implementations
  }

Exemple :

  Runnable action = new Runnable() {
      public void run() {
          travailler();
      }
  };

Une classe anonyme n'a pas de nom source et ne peut donc declarer un
constructeur. Elle peut utiliser un initialisateur d'instance. Pour une
interface, le constructeur Object sans argument est utilise implicitement.

Elle peut capturer seulement des locales final et accede aux membres de son
instance englobante selon les regles d'acces.

8.25 Visibilite et encapsulation

  Modificateur  Acces
  ------------  --------------------------------------------------------------
  private       classe declarante
  aucun         meme package
  protected     meme package et sous-classes, avec regles de receveur
  public        partout si les types englobants sont accessibles

L'encapsulation consiste a cacher l'etat et a exposer des operations :

  class Personne {
      private int age;

      public int getAge() {
          return age;
      }

      public void setAge(int valeur) {
          if (valeur < 0)
              throw new IllegalArgumentException();
          age = valeur;
      }
  }

Un champ public permet toute modification sans validation. Les conventions
get/set peuvent etre presentees comme proprietes JavaBeans ou WFC par l'IDE,
mais elles restent des methodes en Java/J++.

8.26 Membres protected

protected donne deux formes d'acces :

  - tout code du meme package peut acceder au membre;
  - une sous-classe d'un autre package peut y acceder dans le contexte de
    l'objet en cours de derivation.

Dans un autre package, une sous-classe ne peut pas utiliser librement une
reference quelconque de la superclasse pour lire le membre protected. Le
receveur doit respecter les regles liees au type de la sous-classe.

Cette subtilite distingue protected de "public pour les sous-classes". Pour une
API destinee a l'heritage, une methode protected est souvent preferable a un
champ protected afin de conserver le controle de l'invariant.

8.27 java.lang.Object

Object est la racine des classes et fournit notamment :

  public final Class getClass()
  public boolean equals(Object obj)
  public int hashCode()
  public String toString()
  protected Object clone() throws CloneNotSupportedException
  protected void finalize() throws Throwable
  public final void wait(...)
  public final void notify()
  public final void notifyAll()

Les signatures exactes comportent plusieurs surcharges de wait. Les methodes
de moniteur exigent que le thread possede le verrou.

Une classe herite de ces methodes, sauf constructeurs. Elle peut redefinir
equals, hashCode, toString et finalize selon leurs contrats; getClass, wait,
notify et notifyAll sont final.

8.28 Identite, equals et hashCode

== compare l'identite des references. equals compare par defaut la meme
identite dans Object, mais une classe peut definir une equivalence logique :

  public boolean equals(Object objet) {
      if (objet == this)
          return true;
      if (!(objet instanceof Point))
          return false;
      Point autre = (Point)objet;
      return x == autre.x && y == autre.y;
  }

Le contrat usuel de equals est :

  - reflexif;
  - symetrique;
  - transitif;
  - coherent entre appels sans modification;
  - toujours false face a null.

Si equals est redefinie, hashCode doit l'etre : deux objets equals doivent
produire le meme hashCode. La reciproque n'est pas exigee.

Les champs utilises ne doivent pas rendre l'equivalence instable lorsque
l'objet sert de cle dans Hashtable.

8.29 toString

Object.toString retourne une representation technique. Une classe peut la
redefinir :

  public String toString() {
      return "Point(" + x + "," + y + ")";
  }

Elle est utilisee implicitement par la concatenation String et par
PrintStream.println(Object) :

  System.out.println(point);
  String texte = "position=" + point;

toString devrait retourner une information utile sans modifier l'objet ni
lever d'exception en fonctionnement normal. Son format n'est pas
necessairement un protocole stable sauf documentation explicite.

8.30 Clonage

Object.clone effectue une copie superficielle champ par champ. La classe doit
normalement implementer java.lang.Cloneable :

  class Point implements Cloneable {
      public Object clone() {
          try {
              return super.clone();
          } catch (CloneNotSupportedException e) {
              throw new InternalError();
          }
      }
  }

Cloneable est une interface marqueur sans methode. Sans elle, Object.clone
leve CloneNotSupportedException.

Une copie superficielle duplique les champs primitifs et les references, mais
pas les objets references. Les tableaux sont clonables et leur clone a la meme
longueur; les elements reference restent partages.

Pour une copie profonde, la classe doit copier explicitement ses objets
mutables. Les constructeurs ne sont pas appeles par Object.clone.

8.31 Finalisation

Une classe peut redefinir :

  protected void finalize() throws Throwable {
      try {
          libererRessource();
      } finally {
          super.finalize();
      }
  }

La JVM peut appeler finalize avant de recuperer un objet devenu inatteignable,
mais ni le moment ni l'appel lui-meme ne sont garantis. La finalisation peut
retarder la collecte et meme ressusciter temporairement l'objet.

Elle ne doit jamais etre le seul moyen de liberer un fichier, un handle Win32
ou un objet COM. Utiliser close(), un bloc finally ou ComLib.release.

System.gc() et System.runFinalization() ne sont que des demandes.

8.32 Ramasse-miettes et atteignabilite

Un objet devient candidat a la collecte lorsqu'il n'est plus atteignable depuis
les racines de la JVM : piles actives, champs static, threads, references
natives et autres racines du runtime.

  objet = null;               // retire une reference, ne force pas la collecte

Les cycles ne posent pas de probleme au ramasse-miettes par atteignabilite :
deux objets se referencant mutuellement peuvent etre collectes si rien
d'atteignable ne les vise.

La portee lexicale n'impose pas exactement le moment de collecte. Une JVM peut
constater qu'une reference ne sera plus utilisee avant la fin du bloc.

Les ressources natives possedent un cycle de vie distinct de la memoire Java
et demandent une gestion deterministe.

8.33 Reflection et objet Class

Chaque type charge possede un objet java.lang.Class :

  Class c1 = Point.class;
  Class c2 = point.getClass();
  Class c3 = Class.forName("exemple.Point");

getClass retourne le type reel. Class.forName peut charger et initialiser une
classe par son nom et lever ClassNotFoundException.

La reflection Java 1.1 permet d'examiner constructeurs, champs et methodes et
d'effectuer certains appels dynamiques par java.lang.reflect. Les controles
d'acces et exceptions de reflection restent applicables.

Deux classes de meme nom chargees par des chargeurs differents peuvent etre des
types distincts. Le nom seul ne definit donc pas toujours l'identite d'un type.

8.34 Serialisation

Une classe implemente java.io.Serializable pour participer au mecanisme
standard de serialisation :

  class Personne implements Serializable {
      String nom;
      transient String motDePasse;
  }

Serializable est une interface marqueur. Les champs d'instance non static et
non transient sont normalement enregistres. Les objets references doivent eux
aussi etre serialisables, sinon NotSerializableException peut survenir.

Les constructeurs des classes serialisables ne sont pas tous executes lors de
la restauration; le premier constructeur de superclasse non serialisable joue
un role particulier.

serialVersionUID, lorsqu'il est utilise par la version du runtime, stabilise
l'identite de version. Le mecanisme n'est pas un format d'echange securise :
des donnees provenant d'une source non fiable ne doivent pas etre acceptees
sans controle supplementaire.

8.35 Immutabilite

Une classe immuable ne change pas d'etat observable apres sa construction.
Technique courante :

  - classe final ou controle strict des sous-classes;
  - champs private final;
  - aucune methode de modification;
  - validation complete dans le constructeur;
  - copies defensives des objets mutables entrants et sortants;
  - ne pas publier this pendant la construction.

Exemple :

  final class Coordonnees {
      private final int x;
      private final int y;

      Coordonnees(int x, int y) {
          this.x = x;
          this.y = y;
      }

      public int getX() { return x; }
      public int getY() { return y; }
  }

final sur une reference seule ne rend pas l'objet vise immuable. String est
immuable; StringBuffer ne l'est pas.

8.36 Composition et heritage

L'heritage exprime une relation "est un" :

  class Chien extends Animal

La composition exprime une relation "possede un" :

  class Voiture {
      private Moteur moteur;
  }

La composition permet souvent de changer l'implementation sans exposer une
hierarchie et evite le couplage aux membres protected. L'heritage convient
lorsque la sous-classe respecte le contrat substituable de la superclasse.

Une classe ne peut etendre plusieurs implementations; elle peut composer
plusieurs objets et implementer plusieurs interfaces.

8.37 Objets, threads et moniteurs

Tout objet peut servir de moniteur :

  synchronized (verrou) {
      etat++;
  }

Une methode synchronized d'instance verrouille this. Une methode static
synchronized verrouille l'objet Class, pas chaque instance.

Le verrou est reentrant. Il est libere lors de toute sortie du bloc ou de la
methode, y compris par exception.

wait, notify et notifyAll doivent etre appeles pendant que le thread possede le
moniteur. equals et hashCode n'influencent pas l'identite du moniteur : deux
objets egaux mais distincts ont deux verrous distincts.

8.38 Classes WFC, COM et Win32

Les classes Microsoft restent des classes ou interfaces reference selon le
modele precedent :

  - une fenetre WFC derive d'une classe com.ms.wfc.ui;
  - un proxy COM implemente des interfaces generees par JActiveX;
  - une structure Win32 est souvent representee par une classe com.ms.win32;
  - un delegate est compile en type reference propre au runtime Microsoft.

Exemple schematique WFC :

  public class Fenetre extends Form {
      public Fenetre() {
          setText("Exemple");
      }
  }

Exemple COM :

  _Application application = new Application();

La creation peut allouer des ressources natives en plus de l'objet Java. Leur
liberation peut exiger dispose(), close() ou ComLib.release selon l'API.

Les classes proxy generees ne doivent pas etre modifiees sans connaitre leur
regeneration. Leur heritage apparent peut representer des interfaces COM et non
une implementation Java ordinaire.

8.39 Erreurs frequentes

  Erreur                              Explication
  ----------------------------------  ------------------------------------------
  void NomClasse()                    methode, pas constructeur
  new ClasseAbstracte()               classe non instanciable
  oublier le constructeur sans arg.   aucun defaut si un autre est declare
  croire les champs polymorphes       ils sont resolus statiquement
  reduire l'acces d'une redefinition  interdit
  elargir throws verifie              contrat parent viole
  redefinir equals sans hashCode      collections incoherentes
  croire final profond                objet reference encore mutable
  appeler methode virtuelle du ctor   objet partiellement initialise
  compter sur finalize                execution non garantie
  croire x = null destructeur         collecte non immediate
  confondre static et instance        absence de this en contexte static

8.40 Exemple recapitulatif

Les deux classes publiques suivantes resident dans des fichiers distincts.

  public abstract class Forme implements Cloneable, Serializable {
      private static int nombre;
      private final int identifiant;
      protected int x;
      protected int y;

      protected Forme(int x, int y) {
          identifiant = ++nombre;
          this.x = x;
          this.y = y;
      }

      public final int getIdentifiant() {
          return identifiant;
      }

      public abstract double aire();

      public void deplacer(int dx, int dy) {
          x += dx;
          y += dy;
      }

      public Object clone() {
          try {
              return super.clone();
          } catch (CloneNotSupportedException e) {
              throw new InternalError();
          }
      }

      public String toString() {
          return "Forme#" + identifiant + "(" + x + "," + y + ")";
      }
  }

  public final class Rectangle extends Forme {
      private int largeur;
      private int hauteur;

      public Rectangle(int x, int y, int largeur, int hauteur) {
          super(x, y);
          this.largeur = largeur;
          this.hauteur = hauteur;
      }

      public double aire() {
          return largeur * hauteur;
      }
  }

Forme est abstract, possede un etat static et un blank final, definit une
methode concrete et une methode abstraite, expose le clonage et toString.
Rectangle est final, appelle super(...) et fournit l'implementation dynamique
de aire().

9. INTERFACES ET CLASSES INTERNES
================================================================================

9.1 Role d'une interface

Une interface declare un contrat de types sans fournir d'etat d'instance ni
d'implementation de methode dans Java 1.1 :

  public interface Dessinable {
      void dessiner();
  }

Une classe promet ce contrat avec implements :

  public class Cercle implements Dessinable {
      public void dessiner() {
          System.out.println("cercle");
      }
  }

Une variable de type interface peut contenir toute instance d'une classe qui
implemente cette interface :

  Dessinable objet = new Cercle();
  objet.dessiner();

Le type compile Dessinable limite les membres directement visibles. La classe
reelle Cercle choisit dynamiquement l'implementation de dessiner().

Une interface n'est ni une classe abstraite speciale, ni un objet instanciable.
Elle peut participer a plusieurs hierarchies de contrats, contrairement a une
classe qui n'a qu'une superclasse.

9.2 Syntaxe d'une interface

Interface de premier niveau :

  [public] [abstract] interface Nom
      [extends I1 {, IN}] {
      membres
  }

Exemple :

  public interface Transformable extends Dessinable, Cloneable {
      int VERSION = 1;
      void transformer(int x, int y);
  }

abstract est implicite et rarement ecrit. Sans public, une interface de premier
niveau a l'acces de package. private, protected et static ne sont pas permis
sur une interface de premier niveau.

Le corps accepte des constantes, des declarations de methodes abstraites et des
types membres. Il ne contient ni constructeur, ni initialisateur d'instance,
ni bloc static ordinaire.

9.3 Methodes d'interface

Syntaxe source :

  [public] [abstract] TypeRetour nom([parametres])
      [throws types] ;

public et abstract sont implicites :

  void dessiner();

equivaut a :

  public abstract void dessiner();

La declaration se termine par ';' et n'a aucun corps. Une methode ne peut etre
private, protected, static, final, synchronized ou native dans une interface
Java 1.1.

Le type retour et les types de parametres doivent etre accessibles au moins
autant que l'interface si celle-ci constitue une API publique, sinon les
clients ne pourront pas employer utilement la signature.

Les methodes default, static et private d'interface sont des ajouts Java
ulterieurs et ne sont pas reconnues par J++ 6.0.

9.4 Champs d'interface

Tout champ d'interface est implicitement :

  public static final

Il doit etre initialise :

  interface Limites {
      int MINIMUM = 0;
      int MAXIMUM = 100;
      String NOM = "J++";
  }

Ces declarations equivalent a :

  public static final int MINIMUM = 0;

Il n'existe aucun champ d'instance d'interface. Les modificateurs private,
protected, transient et volatile sont interdits. Le champ est accessible par :

  Limites.MAXIMUM

Une classe qui implemente l'interface peut parfois utiliser MAXIMUM sans
qualification, mais qualifier evite les ambiguites.

Si l'initialiseur est une expression constante primitive ou String, la valeur
peut etre incorporee dans le bytecode des clients. Une recompilation des
clients peut etre necessaire apres modification.

9.5 Heritage multiple d'interfaces

Une interface peut etendre plusieurs interfaces :

  interface Imprimable {
      void imprimer();
  }

  interface Archivable {
      void archiver();
  }

  interface Document extends Imprimable, Archivable {
      void fermer();
  }

Document herite des trois obligations. Une classe concrete qui l'implemente
doit fournir imprimer(), archiver() et fermer(), directement ou par heritage.

Un cycle est interdit :

  interface A extends B { }
  interface B extends A { }       // invalide

Le graphe peut former un diamant sans dupliquer l'etat, car les interfaces ne
possedent pas de champs d'instance.

9.6 Conflits de methodes heritees

Deux superinterfaces peuvent declarer la meme signature. Si le type retour et
les clauses compatibles permettent une seule obligation, celle-ci est heritee
une fois :

  interface A {
      void executer();
  }

  interface B {
      void executer();
  }

  interface C extends A, B {
  }

Une classe CConcrete n'implemente executer() qu'une fois.

En Java 1.1, des declarations de meme signature avec types retour differents
sont incompatibles, meme si les types seraient covariants dans un Java plus
recent :

  interface A { Object obtenir(); }
  interface B { String obtenir(); }
  interface C extends A, B { }    // invalide en Java 1.1

Les clauses throws doivent permettre a l'implementation de respecter tous les
contrats herites.

9.7 Conflits de constantes

Deux interfaces peuvent fournir le meme nom de champ :

  interface A { int VALEUR = 1; }
  interface B { int VALEUR = 2; }

  class C implements A, B {
      int f() {
          return VALEUR;           // ambigu
      }
  }

Qualifier resout le conflit :

  return A.VALEUR + B.VALEUR;

Les champs ne sont pas fusionnes et ne sont pas polymorphes. Meme si les deux
valeurs sont identiques, un nom simple herite de sources distinctes peut rester
ambigu.

9.8 implements

Syntaxe :

  class Nom [extends SuperClasse]
      implements I1 {, IN} {
      membres
  }

Exemple :

  class Tache extends Object implements Runnable, Serializable {
      public void run() {
          travailler();
      }
  }

La liste accepte seulement des interfaces accessibles. Une interface heritee
par une autre n'a pas besoin d'etre repetee.

Une classe concrete doit implementer toutes les methodes abstraites accessibles
de ses interfaces et superclasses. Une classe abstract peut laisser certaines
obligations a ses sous-classes.

implements ne copie aucun code. Il etablit une relation de sous-typage verifiee
par le compilateur et la JVM.

9.9 Implementation d'une methode d'interface

La methode d'implementation doit etre public, car la methode d'interface l'est
implicitement :

  interface Action {
      void executer();
  }

  class ActionConcrete implements Action {
      public void executer() {
          ...
      }
  }

Omettre public donne un acces de package et ne satisfait pas le contrat.

En Java 1.1 :

  - le nom et les parametres doivent correspondre;
  - le type retour doit etre identique;
  - aucune exception verifiee nouvelle ou plus large ne peut etre ajoutee;
  - des exceptions verifiees plus precises peuvent etre declarees;
  - les exceptions non verifiees ne sont pas limitees par throws.

Une methode public heritee d'une superclasse peut satisfaire l'interface sans
nouvelle declaration.

9.10 Classe abstraite et implementation partielle

Une classe abstract peut implementer une interface sans fournir toutes ses
methodes :

  abstract class Forme implements Dessinable {
      protected int x;
      protected int y;

      public void deplacer(int dx, int dy) {
          x += dx;
          y += dy;
      }
  }

Une sous-classe concrete doit finalement fournir dessiner().

Une classe abstraite peut aussi declarer explicitement la methode :

  public abstract void dessiner();

Cette redeclaration peut documenter le contrat ou adapter la clause throws,
mais reste sans corps.

9.11 References d'interface

Conversion elargissante implicite :

  Cercle cercle = new Cercle();
  Dessinable d = cercle;

Conversion reductrice :

  Cercle c = (Cercle)d;

Le cast reussit si l'objet reel est compatible, sinon il leve
ClassCastException. null reste null.

Test :

  if (d instanceof Cercle) {
      Cercle c = (Cercle)d;
  }

Une reference interface donne acces aux methodes du contrat et de ses
superinterfaces. Elle peut etre convertie vers Object, puisque toute instance
qui l'implemente est un objet.

9.12 Liaison dynamique par interface

L'appel :

  Dessinable d = obtenirDessinable();
  d.dessiner();

est resolu en deux etapes :

  1. le compilateur verifie la signature dans Dessinable;
  2. la JVM choisit l'implementation de la classe reelle.

Deux objets de classes sans relation d'heritage peuvent donc etre utilises de
la meme facon :

  Dessinable a = new Cercle();
  Dessinable b = new Rectangle();

Cette separation constitue le principal moyen de polymorphisme multiple en
Java/J++.

Les champs d'interface sont static final et ne suivent aucune liaison
dynamique.

9.13 Interfaces marqueurs

Une interface peut ne declarer aucune methode :

  public interface Cloneable {
  }

  public interface Serializable {
  }

Elle marque une classe afin qu'une API ou la JVM lui applique un traitement.
Par exemple, Object.clone verifie Cloneable et la serialisation verifie
Serializable.

Une interface marqueur reste un type : instanceof, casts, parametres et
collections peuvent l'utiliser.

Les annotations, parfois employees pour ce role dans Java moderne, n'existent
pas en J++ 6.0.

9.14 Interface membre

Une interface peut etre membre d'une classe ou d'une autre interface :

  class Protocole {
      public interface Ecouteur {
          void evenement();
      }
  }

Utilisation :

  class MonEcouteur implements Protocole.Ecouteur {
      public void evenement() {
          ...
      }
  }

Une interface membre est implicitement static : elle ne conserve aucune
reference vers une instance englobante. Dans une classe, elle peut etre public,
protected, private ou d'acces package. Dans une interface, un type membre est
implicitement public static.

9.15 Types absents des interfaces J++ historiques

Visual J++ 6.0 ne prend pas en charge :

  default void methode() { ... }     methodes default Java 8
  static void methode() { ... }      methodes static d'interface Java 8
  private void methode() { ... }     methodes privees Java 9
  interface I<T> { ... }             generics Java 5
  @FunctionalInterface               annotations Java 5+
  interface I permits A, B           interfaces sealed modernes

Une interface a une seule methode peut servir de protocole de rappel, mais elle
n'est pas une interface fonctionnelle utilisable par lambda, car les lambdas
n'existent pas.

9.16 Classification des classes imbriquees

Une classe declaree dans le contexte d'une autre peut etre :

  classe membre static        class Externe { static class N { } }
  classe membre interne       class Externe { class I { } }
  classe locale               void f() { class L { } }
  classe anonyme              new Runnable() { ... }

Le terme "classe imbriquee" couvre toutes ces formes. Le terme "classe interne"
designe une classe imbriquee non static qui conserve un lien vers une instance
englobante : membre interne, locale ou anonyme creee en contexte d'instance.

Java 1.1 introduisit ces constructions. Visual J++ 6.0 les prend en charge dans
la mesure de son niveau de compatibilite Java 1.1.

9.17 Classe membre static

Syntaxe :

  class Externe {
      static class Imbriquee {
          ...
      }
  }

Creation :

  Externe.Imbriquee objet = new Externe.Imbriquee();

Elle n'a pas besoin d'une instance Externe et ne possede pas de Externe.this.
Elle peut acceder directement aux membres static de l'englobante, mais a besoin
d'une reference explicite pour ses membres d'instance.

Une classe membre static peut declarer champs static, blocs static et methodes
static comme une classe de premier niveau. Elle peut etre public, protected,
private ou d'acces package, ainsi que abstract ou final.

Seuls les types membres peuvent porter static; une classe locale ou anonyme ne
le peut pas.

9.18 Classe membre interne

Syntaxe :

  class Externe {
      private int valeur;

      class Interne {
          int lire() {
              return valeur;
          }
      }
  }

Chaque objet Interne est associe a une instance Externe. L'acces non qualifie a
valeur equivaut ici a Externe.this.valeur.

Depuis une methode d'instance d'Externe :

  Interne i = new Interne();

Depuis l'exterieur :

  Externe externe = new Externe();
  Externe.Interne interne = externe.new Interne();

La syntaxe externe.new Interne() fournit explicitement l'instance englobante.

9.19 Reference englobante

Dans une classe interne :

  this                    objet interne courant
  Externe.this            objet englobant de type Externe

Exemple :

  class Externe {
      int valeur;

      class Interne {
          int valeur;

          int somme() {
              return this.valeur + Externe.this.valeur;
          }
      }
  }

Une classe peut etre imbriquee dans plusieurs niveaux :

  TresExterne.this
  Externe.this
  this

Le qualificateur doit nommer une classe lexicalement englobante dont une
instance est disponible.

Une classe membre static coupe la chaine d'instances englobantes.

9.20 Constructeur d'une classe interne

Le source declare un constructeur ordinaire :

  class Externe {
      class Interne {
          Interne(int valeur) {
              ...
          }
      }
  }

Le compilateur ajoute conceptuellement la reference englobante aux donnees de
l'objet et a son mecanisme de construction. Ce parametre cache n'apparait pas
dans la syntaxe source :

  Externe e = new Externe();
  Externe.Interne i = e.new Interne(10);

Le constructeur ne peut choisir une autre instance englobante apres creation.
La reference participe a l'atteignabilite : conserver longtemps l'objet interne
peut aussi conserver l'objet externe.

Les details du nom et du parametre synthetique ne constituent pas une API
source stable.

9.21 Membres static d'une classe interne

Une classe interne non static ne peut normalement pas declarer de membre static
ou de bloc static, car son existence depend d'une instance englobante.

Exception historique : elle peut declarer une variable constante :

  class Externe {
      class Interne {
          static final int MAXIMUM = 10;
      }
  }

MAXIMUM doit etre une constante de compilation primitive ou String. Ceci serait
invalide :

  static int compteur;
  static { compteur = 1; }

Pour posseder un etat ou comportement static general, utiliser une classe
membre static.

9.22 Acces prive entre classes imbriquees

Au niveau du langage source, une classe englobante et ses classes imbriquees
peuvent acceder a leurs membres private respectifs :

  class Externe {
      private int secret;

      class Interne {
          private int valeurPrivee;

          int lire() {
              return secret;
          }
      }

      int lireInterne(Interne i) {
          return i.valeurPrivee;
      }
  }

Pour les anciens formats de classes et JVM, le compilateur peut generer des
methodes synthetiques d'acces afin de realiser cette permission. Ces methodes
ne doivent pas etre appelees directement par le source applicatif.

La reflection ou un desassembleur peut donc montrer des membres absents du
fichier .java d'origine.

9.23 Classe locale

Syntaxe :

  void methode() {
      class Locale {
          void executer() {
              ...
          }
      }

      Locale objet = new Locale();
  }

La portee du nom commence a la declaration et se termine a la fin du bloc. La
classe ne peut etre utilisee avant sa declaration par son nom simple.

Une classe locale ne peut etre public, protected, private ou static. Elle peut
etre abstract ou final selon les regles applicables.

Elle peut acceder aux membres de l'instance englobante. Elle peut capturer une
variable locale ou un parametre uniquement si celui-ci est final.

9.24 Capture des variables final

Java 1.1 exige final :

  final String message = "Bonjour";

  class Tache implements Runnable {
      public void run() {
          System.out.println(message);
      }
  }

La variable doit aussi etre definitivement affectee avant la creation de
l'objet interne. Le compilateur copie conceptuellement sa valeur dans l'objet
genere.

Forme invalide en J++ 6.0 :

  String message = "Bonjour";
  Runnable r = new Runnable() {
      public void run() {
          System.out.println(message);
      }
  };

Java moderne accepte une variable "effectivement finale", mais cette regle
n'existait pas en Java 1.1 : le mot final doit etre ecrit.

final protege la variable capturee, pas l'objet mutable qu'elle reference.

9.25 Classe anonyme et superclasse

Syntaxe :

  new SuperClasse(arguments) {
      declarationsDeMembres
  }

Exemple :

  Thread thread = new Thread("travail") {
      public void run() {
          travailler();
      }
  };

La classe anonyme etend exactement la superclasse indiquee. Les arguments sont
passes a son constructeur accessible.

Elle n'a aucun nom source et ne peut declarer de constructeur. Elle peut
declarer champs, methodes et initialisateurs d'instance. Le type de l'expression
new est compatible avec la superclasse.

Elle ne peut contenir les membres static interdits aux classes internes, sauf
constantes de compilation.

9.26 Classe anonyme et interface

Syntaxe :

  new Interface() {
      implementations
  }

Exemple de listener Java 1.1 :

  bouton.addActionListener(new ActionListener() {
      public void actionPerformed(ActionEvent evenement) {
          executer();
      }
  });

La classe anonyme implemente l'interface et etend Object. Les parentheses sont
obligatoires et restent vides, car une interface n'a pas de constructeur.

Toutes les methodes requises doivent etre implementees public, sauf si une
superclasse applicable les fournit; ici la classe anonyme est concrete.

Une seule interface figure apres new. Pour plusieurs contrats, declarer une
interface qui les etend ou une classe locale qui les implemente.

9.27 this dans une classe anonyme

Dans le corps anonyme, this designe l'objet anonyme, non l'objet englobant :

  class Fenetre {
      void installer() {
          bouton.addActionListener(new ActionListener() {
              public void actionPerformed(ActionEvent e) {
                  this.toString();       // listener anonyme
                  Fenetre.this.fermer(); // fenetre englobante
              }
          });
      }
  }

Une classe anonyme n'a pas de nom permettant NomAnonyme.this. Le nom de la
classe lexicale englobante permet d'atteindre celle-ci.

super dans la classe anonyme designe sa superclasse directe, Object lorsque la
forme new Interface() est utilisee.

9.28 Masquage des noms

Chaque niveau peut declarer un champ du meme nom :

  class Externe {
      int valeur;

      class Interne {
          int valeur;

          void f(int valeur) {
              utiliser(valeur);                // parametre
              utiliser(this.valeur);           // champ Interne
              utiliser(Externe.this.valeur);   // champ Externe
          }
      }
  }

Les qualifications rendent le choix explicite. Une classe locale peut aussi
masquer un nom de type d'une portee externe.

9.29 Heritage d'une classe interne

Une classe interne peut etendre une autre classe et implementer des interfaces :

  class Externe {
      class Tache extends Base implements Runnable {
          public void run() {
              ...
          }
      }
  }

Une classe externe peut aussi etendre un type interne accessible, mais la
construction doit fournir l'instance englobante requise. Ces formes avancees
sont difficiles a maintenir et dependent de regles precises de constructeur
qualifie.

Il est souvent plus clair d'utiliser composition ou une classe membre static
si l'objet n'a pas reellement besoin d'une instance englobante.

Les membres herites et redefinis suivent les memes regles que pour toute autre
classe.

9.30 Initialisation des classes imbriquees

Une classe membre static est initialisee comme toute classe lors de sa premiere
utilisation active. Charger Externe n'initialise pas automatiquement toutes ses
classes membres.

Une classe interne est une classe distincte au niveau de la JVM. Son objet
recoit d'abord ses valeurs par defaut, sa superclasse est construite, puis ses
initialisateurs et son constructeur s'executent.

La reference englobante est etablie dans le mecanisme de construction avant que
le corps utilisateur en ait besoin.

Les initialisateurs de la classe externe ne sont pas rejoues lors de chaque
creation d'une classe interne; l'instance externe existe deja.

9.31 Noms binaires et fichiers .class

Le compilateur emet souvent des fichiers dont le nom contient '$' :

  Externe.class
  Externe$Interne.class
  Externe$1.class

Les classes anonymes ou locales peuvent recevoir des numeros ou noms
synthetiques propres au compilateur.

Ces conventions ne doivent pas etre utilisees comme API stable :

  - la numerotation change si le source est reorganise;
  - un autre compilateur peut choisir une forme differente;
  - les membres synthetiques peuvent varier;
  - le nom source d'une classe anonyme n'existe pas.

Employer les noms et constructeurs du langage source plutot que de charger
directement un nom binaire genere.

9.32 Serialisation des classes internes

Une classe interne serialisable peut capturer implicitement son instance
englobante. La serialisation peut alors tenter d'enregistrer aussi cette
instance et tout son graphe d'objets.

Si l'englobante n'est pas Serializable, NotSerializableException peut survenir
meme si la classe interne declare Serializable.

Les champs synthetiques et noms generes peuvent aussi compliquer la
compatibilite entre versions.

Pour un objet serialisable independant, une classe membre static ou une classe
de premier niveau est souvent preferable.

9.33 Atteignabilite et fuites logiques

Une classe interne conserve l'instance englobante :

  Externe -> Interne
  Interne -> Externe

Le ramasse-miettes sait collecter les cycles devenus inatteignables. Le probleme
survient lorsqu'un service vivant longtemps conserve l'objet interne : la
reference retour peut garder toute l'instance externe en memoire.

Exemple typique :

  - une fenetre enregistre un listener anonyme dans un service global;
  - le service conserve le listener;
  - le listener conserve implicitement la fenetre.

Il faut retirer les listeners lorsqu'ils ne servent plus, ou utiliser une
classe membre static avec une reference geree explicitement.

9.34 Interfaces et modele evenementiel Java

Java 1.1 emploie des interfaces listener :

  public interface ActionListener extends EventListener {
      void actionPerformed(ActionEvent e);
  }

Une source d'evenements conserve les listeners et appelle leur methode :

  bouton.addActionListener(listener);
  bouton.removeActionListener(listener);

Une classe anonyme evite de creer un nom de classe distinct pour un petit
gestionnaire. Une classe nommee est preferable lorsque le comportement est
reutilise, teste separement ou doit etre retire facilement.

Les adapteurs, comme WindowAdapter, sont des classes abstraites fournissant des
implementations vides de plusieurs methodes listener; une classe anonyme peut
alors ne redefinir que la methode utile.

9.35 Interfaces Java et delegates J++

Le modele Java standard represente un rappel par une interface :

  interface Rappel {
      void appeler(int code);
  }

L'extension Microsoft peut representer une signature par un delegate :

  delegate void Rappel(int code);
  Rappel r = new Rappel(objet.traiter);

Differences conceptuelles :

  interface listener        objet avec une ou plusieurs methodes nommees
  delegate J++              reference typee vers une methode compatible
  multicast delegate        chaine possible de plusieurs methodes

Les delegates ne remplacent pas les interfaces pour le sous-typage general.
Ils sont surtout utiles aux rappels et au modele evenementiel Microsoft/WFC.
Leur syntaxe n'est pas portable vers javac.

9.36 Interfaces COM et classes generees

JActiveX traduit souvent une interface COM en interface Java et produit une
classe proxy :

  import com.ms.com.*;
  import paquet.genere.*;

  IApplication application = new Application();

L'interface Java decrite par le proxy sert au typage, tandis que l'objet reel
transmet les appels a COM. La liaison dynamique visible dans le source peut
donc franchir une frontiere native.

Ces interfaces generees peuvent contenir :

  - methodes get/set representant des proprietes COM;
  - parametres Variant;
  - interfaces derivant de IUnknown;
  - methodes correspondant aux dispinterfaces;
  - conventions de gestion d'erreurs et de references.

Modifier manuellement un fichier genere est risque, car JActiveX peut
l'ecraser. L'interface depend de la version de la bibliotheque de types.

9.37 Erreurs frequentes avec les interfaces

  Erreur                             Explication
  ---------------------------------  -------------------------------------------
  new MonInterface()                 interface non instanciable seule
  methode sans public                n'implemente pas le contrat public
  corps de methode dans interface    interdit en Java 1.1
  champ non initialise              constante d'interface incomplete
  champ mutable d'interface          tout champ est static final
  implements UneClasse              implements exige une interface
  extends I1, I2 sur une classe      une classe n'etend qu'une classe
  types retour incompatibles         conflit entre methodes heritees
  constante simple ambigue           qualifier avec le nom d'interface
  lambda                             absente de J++ historique

9.38 Erreurs frequentes avec les classes internes

  Erreur                             Explication
  ---------------------------------  -------------------------------------------
  new Externe.Interne()              instance englobante manquante
  capture d'une locale non final     interdit en Java 1.1
  static ordinaire dans Interne      seuls constants compilees sont permis
  constructeur de classe anonyme     impossible, aucun nom source
  this pris pour l'englobante        this designe l'objet interne
  oubli d'un listener enregistre     peut retenir l'objet englobant
  serialiser sans l'englobante       reference synthetique egalement capturee
  dependre de Externe$1              nom binaire instable

9.39 Exemple recapitulatif

  public interface SelectionListener {
      void selectionChangee(int index);
  }

  public class Liste {
      private SelectionListener listener;

      public void setSelectionListener(SelectionListener valeur) {
          listener = valeur;
      }

      public void selectionner(int index) {
          if (listener != null)
              listener.selectionChangee(index);
      }

      public static class Position {
          public int ligne;
          public int colonne;
      }

      public class Curseur {
          private int index;

          public void avancer() {
              index++;
              selectionner(index);
          }
      }

      public Runnable creerAction(final int index) {
          class ActionLocale implements Runnable {
              public void run() {
                  selectionner(index);
              }
          }
          return new ActionLocale();
      }
  }

Cet exemple comprend une interface, une classe membre static, une classe
interne associee a Liste, une classe locale, une capture final et une liaison
dynamique vers SelectionListener.

10. TABLEAUX
================================================================================

  int[] a = new int[5];
  int b[] = { 2, 3, 5, 7 };
  int[][] matrice = new int[2][3];
  int[][] irregulier = new int[2][];
  irregulier[0] = new int[3];
  irregulier[1] = new int[7];

Les indices valides vont de 0 a length - 1. Un indice invalide leve
ArrayIndexOutOfBoundsException. Les tableaux d'objets sont covariants : un
String[] est un Object[], mais une affectation incompatible par cette vue leve
ArrayStoreException.


================================================================================
11. EXCEPTIONS
================================================================================

  try {
      lire();
  } catch (IOException e) {
      System.err.println(e.getMessage());
  } finally {
      fermer();
  }

  void lire() throws IOException {
      if (erreur) throw new IOException("lecture impossible");
  }

Throwable est la racine. Error signale normalement une panne serieuse.
Exception regroupe les exceptions applicatives. RuntimeException et ses
sous-classes sont non verifiees; les autres exceptions doivent etre capturees
ou declarees par throws.

J++ ne possede ni multi-catch ni try-with-resources. Plusieurs catch sont
ecrits separement, du type le plus precis au plus general.


================================================================================
12. PAQUETS, IMPORTS ET VISIBILITE
================================================================================

  package societe.produit;
  import java.io.*;
  import java.util.Vector;

Un import n'inclut pas recursivement les sous-paquets. java.lang est importe
implicitement. Deux types de meme nom doivent etre distingues par leur nom
qualifie. Sans modificateur public/protected/private, un type ou membre a la
visibilite de package, parfois appelee default access.

Le CLASSPATH indique au compilateur et a la machine virtuelle ou chercher les
classes. Visual J++ pouvait aussi gerer son propre depot de classes et les
references du projet; un projet portable ne doit pas dependre implicitement de
ce depot.


================================================================================
13. ENTREES/SORTIES ET BIBLIOTHEQUE JAVA 1.1
================================================================================

Classes courantes disponibles selon la MSJVM installee :

  java.lang       Object, String, StringBuffer, Math, System, Thread
  java.io         InputStream, OutputStream, Reader, Writer, File
  java.util       Vector, Hashtable, Enumeration, Date, Random
  java.net        URL, Socket, ServerSocket
  java.awt        composants graphiques AWT et evenements
  java.applet     Applet

Exemple d'ecriture de fichier conforme a l'API de l'epoque :

  FileOutputStream f = new FileOutputStream("sortie.txt");
  PrintStream p = new PrintStream(f);
  try {
      p.println("Bonjour");
  } finally {
      p.close();
  }

Les collections historiques ne sont pas generiques. Iterator, ArrayList et le
Collections Framework appartiennent a Java 1.2 et ne doivent pas etre supposes
disponibles dans une cible J++ strictement Java 1.1.


================================================================================
14. THREADS ET SYNCHRONISATION
================================================================================

Un traitement concurrent derive de Thread ou implemente Runnable :

  class Travail implements Runnable {
      public void run() { effectuer(); }
  }
  new Thread(new Travail()).start();

Une methode synchronized verrouille l'objet receveur; une methode static
synchronized verrouille l'objet Class. Un bloc peut choisir son moniteur :

  synchronized (verrou) {
      compteur++;
  }

Object fournit wait(), notify() et notifyAll(). Ils doivent etre appeles par
un thread possedant le moniteur. Le modele memoire de la JVM Microsoft est
historique; le code nouveau ne doit pas lui attribuer retroactivement toutes
les garanties formalisees par les specifications Java modernes.


================================================================================
15. APPLETS, AWT ET WFC
================================================================================

Une applet Java classique derive de java.applet.Applet et utilise notamment
init(), start(), stop(), destroy() et paint(Graphics). Son execution depend
d'un navigateur ou d'un applet viewer compatible, aujourd'hui obsolete.

Visual J++ 6.0 introduisit les Windows Foundation Classes, bibliotheque
Microsoft de classes Windows. Les espaces de noms typiques commencent par :

  com.ms.wfc.app
  com.ms.wfc.core
  com.ms.wfc.ui
  com.ms.wfc.data

Une application WFC utilise en general une classe derivee de Form et appelle
Application.run(...). Les formulaires, controles, proprietes exposees par les
classes et le concepteur graphique relèvent de WFC, pas du noyau Java 1.1.
Un source qui importe com.ms.wfc.* est lie a Windows et a l'environnement
Microsoft.

15.1 Le paquet com.ms.ui

com.ms.ui appartient aux Application Foundation Classes (AFC), la bibliotheque
graphique Microsoft qui preceda les WFC. Il fournit un modele de composants,
de conteneurs et de gestionnaires de disposition comparable a AWT, avec des
controles legers (sans fenetre native propre) et des controles familiers aux
programmeurs Win32.

Les classes rencontrees dans les anciens programmes portent frequemment le
prefixe UI, par exemple UIPushButton ou UIRadioButton. Des sous-paquets
completent le noyau :

  com.ms.ui.event       gestion des evenements, notamment le modele Java 1.1
  com.ms.ui.resource    acces aux ressources Windows

Exemple schematique :

  import com.ms.ui.*;
  import com.ms.ui.event.*;

  public class FenetreAFC extends UIFrame {
      private UIPushButton bouton = new UIPushButton("Fermer");
      // construction et raccordement des evenements selon la version AFC
  }

Le nom exact des classes et de leurs constructeurs varie avec la version du
SDK. com.ms.ui est distinct de com.ms.wfc.ui : le premier designe les AFC des
premieres versions, le second la couche interface utilisateur des WFC de
Visual J++ 6.0. Une conversion ne doit pas remplacer aveuglement un import par
l'autre.

Une partie des AFC fut ecrite en Java, mais cela ne garantit pas la portabilite
de l'application complete. Certaines versions dependent d'extensions de la
MSJVM, de ressources Win32 ou de classes com.ms.* qu'une JVM standard ne
fournit pas. Les archives AFC appropriees doivent aussi etre distribuees quand
elles ne sont pas installees sur la machine cible.

15.2 Le paquet com.ms.awt

com.ms.awt contient l'implementation et les extensions Microsoft d'AWT. Il
fait l'interface entre les classes publiques java.awt et leurs pairs natifs
Windows. On y rencontre notamment des classes telles que WComponentPeer ainsi
que des implementations Microsoft de Toolkit, de composants ou de mesures de
polices, selon la version de la MSJVM.

Exemple rencontre dans du code fortement lie a Windows :

  import com.ms.awt.WComponentPeer;

Un programme AWT portable doit declarer ses variables avec les types java.awt
et ne jamais convertir un objet vers son implementation com.ms.awt. Ainsi, un
objet retourne comme java.awt.FontMetrics peut etre implemente par une classe
Microsoft interne, mais le programme ne doit pas dependre de cette classe.

com.ms.awt doit etre considere comme un paquet d'implementation :

  - il n'appartient pas a la specification Java 1.1;
  - ses classes peuvent changer entre versions de la MSJVM;
  - il n'est normalement pas present dans les JDK de Sun/Oracle ou OpenJDK;
  - son utilisation directe signale une dependance Windows/MSJVM a migrer.

Selon le besoin, une migration remplace cet acces par une API publique AWT,
Swing ou JavaFX, ou isole l'operation native derriere une couche specifique a
la plate-forme.

15.3 Proprietes : J++, JavaBeans, WFC et comparaison avec C#

Le mot "propriete" est frequent dans la documentation et dans l'interface de
Visual J++, mais il faut distinguer le concept expose par les composants de la
syntaxe effectivement acceptee par le langage.

Visual J++ sait afficher et manipuler des proprietes de composants JavaBeans,
AFC, WFC ou COM dans la fenetre Properties de l'IDE. Pour un JavaBean, une
propriete Age est normalement reconnue par convention lorsque la classe offre
les methodes publiques suivantes :

  public class Personne {
      private int age;

      public int getAge() {
          return age;
      }

      public void setAge(int valeur) {
          age = valeur;
      }
  }

Le code client Java ou J++ appelle ces methodes explicitement :

  Personne p = new Personne();
  p.setAge(42);
  int n = p.getAge();

Pour une propriete booleenne, le getter peut suivre la convention is... :

  public boolean isActif() { return actif; }
  public void setActif(boolean valeur) { actif = valeur; }

Les outils d'introspection JavaBeans deduisent le nom et le type de la
propriete a partir de getAge/setAge ou isActif/setActif. Une classe BeanInfo
peut fournir explicitement les memes metadonnees. Le fait que l'IDE presente
Age comme une propriete editable ne transforme pas ces methodes en une nouvelle
construction syntaxique Java.

Les classes WFC et les proxies COM presentent eux aussi des membres comme des
proprietes. Suivant la bibliotheque ou le generateur de proxy, le source J++
utilise des methodes get..., set..., get_..., set_... ou une methode de
dispatch prenant un nom et un Variant. Cette apparence uniforme dans le
concepteur ne signifie pas que J++ possede les accesseurs get/set de C#.

La declaration suivante ressemble a C#, mais n'est pas une syntaxe valide de
Visual J++ 6.0 :

  public int Age
  {
      get;
      set;
  }

Il s'agit plus precisement d'une propriete automatique de C# : le compilateur
C# cree le stockage et les accesseurs. Cette forme abregee apparut apres
Visual J++. Une propriete C# classique avec logique explicite s'ecrit par
exemple :

  private int age;

  public int Age
  {
      get { return age; }
      set { age = value; }
  }

Le code C# peut ensuite employer une syntaxe de champ tout en invoquant les
accesseurs :

  p.Age = 42;
  int n = p.Age;

Java ne possede toujours pas de declaration de propriete integree equivalente.
Dans du Java standard, il faut conserver les methodes getAge() et setAge(), ou
utiliser un outil externe qui genere ce code. Les records Java modernes
fournissent des composants accessibles par des methodes telles que age(), mais
ne creent pas une propriete mutable de style C#.

On peut donc dire que l'environnement J++ avait un modele de proprietes tres
visible et familier aux utilisateurs des outils Microsoft. Il serait toutefois
incorrect d'attribuer au langage Visual J++ la grammaire get/set qui sera celle
de C#. Les ressemblances conceptuelles viennent surtout du modele composant
Microsoft, de WFC/COM et de l'outillage Visual Studio.


================================================================================
16. DELEGATES ET EVENEMENTS MICROSOFT
================================================================================

Visual J++ fournit une extension de langage pour representer un appel de
methode type : le delegate. Un delegate associe une signature a une ou
plusieurs methodes compatibles et sert principalement au modele evenementiel
WFC. Sa forme usuelle est :

  delegate void Gestionnaire(Object source, Event e);

  Gestionnaire g = new Gestionnaire(objet.methode);
  g.invoke(source, evenement);

La declaration Microsoft "multicast delegate" cree une variante pouvant
contenir plusieurs destinataires. L'ajout, le retrait et le declenchement
precis dependent des classes et conventions du
runtime Microsoft utilisees par le projet. Les outils de conception WFC
generent normalement le raccordement des gestionnaires.

Cette syntaxe n'appartient pas a Java 1.1 et n'est pas acceptee par javac. Lors
d'une migration vers Java, on la remplace par les interfaces listener et les
classes anonymes; vers .NET, par les delegates et events C# ou J#.

ATTENTION : event, propriete graphique et delegate sont souvent presentes
ensemble dans l'IDE, mais tous les membres affiches comme "properties" ou
"events" par le concepteur ne correspondent pas necessairement a un nouveau
mot-cle J++. Ils peuvent etre des conventions de bibliotheque WFC.


================================================================================
17. DIRECTIVES MICROSOFT ET COMPILATION CONDITIONNELLE
================================================================================

J++ emploie des commentaires de documentation speciaux compris par JVC. Leur
aspect de commentaire permet a un compilateur Java standard de les ignorer,
mais leur effet est propre aux outils Microsoft.

Forme generale :

  /** @directive(parametres) */

La famille J/Direct permet de declarer l'import d'une fonction Win32 contenue
dans une DLL. Exemple schematique (la signature exacte depend de l'API) :

  /** @dll.import("USER32") */
  static native int MessageBox(int hwnd, String texte,
                               String titre, int type);

Ces declarations sont native, liees a Windows et a la MSJVM. Les conventions
d'appel, conversions de chaines, structures et pointeurs doivent etre verifiees
dans la documentation J/Direct correspondant a la version du SDK.

JVC prend egalement en charge des directives conditionnelles sous forme de
commentaires, notamment les familles @if, @else et @end. Elles permettent au
compilateur Microsoft d'inclure ou d'exclure du source selon des symboles de
compilation. Ce mecanisme n'est pas le preprocesseur C et n'est pas portable.

D'autres attributs en commentaire furent utilises pour COM, securite,
marshalling ou generation de metadonnees. Une directive inconnue ne doit pas
etre inventee : conserver son texte et consulter le SDK Microsoft d'origine.


================================================================================
18. INTEROPERABILITE COM ET WIN32
================================================================================

Visual J++ pouvait appeler :

  - des fonctions de DLL Windows par J/Direct;
  - des composants Automation/COM;
  - des bibliotheques de types converties en classes Java;
  - des objets ActiveX par les outils Java Type Library Wizard ou JActiveX.

JActiveX genere des classes proxy a partir d'une bibliotheque de types. Le
programme importe ensuite ces classes comme toute classe Java. Le code produit
reste cependant dependant de COM, de Windows, des GUID et du runtime Microsoft.

Le mot-cle native de Java indique seulement qu'une methode est implementee
hors Java. Il ne definit ni J/Direct ni COM. Ceux-ci appartiennent a la couche
Microsoft et doivent etre documentes separement dans une application.

18.1 Le paquet com.ms.win32

com.ms.win32 expose sous forme de classes Java une grande partie de l'API
Win32 deja declaree pour J/Direct. Il evite de recopier une declaration
@dll.import pour chaque fonction. Les classes principales correspondent aux
DLL et familles d'API Windows, par exemple :

  com.ms.win32.User32       fenetres, messages, clavier et interface utilisateur
  com.ms.win32.Gdi32        dessin et objets GDI
  com.ms.win32.Kernel32     processus, threads, fichiers et memoire
  com.ms.win32.Ole32        initialisation et activation OLE/COM
  com.ms.win32.Shell32      fonctions du shell Windows

Le nom Kernel ou Kernel32 et la couverture exacte peuvent varier selon la
version du SDK Microsoft. Toutes les fonctions Win32 ne sont pas necessairement
declarees.

Exemple :

  import com.ms.win32.*;

  public class MessageNatif {
      public static void main(String[] args) {
          User32.MessageBox(0, "Bonjour", "J++",
                            winm.MB_OK | winm.MB_ICONINFORMATION);
      }
  }

Les constantes Win32 sont regroupees dans des interfaces dont le nom commence
par win et se termine souvent par la premiere lettre de la constante. Par
exemple, plusieurs constantes MB_* se trouvent dans winm et SW_* dans wins.
Les structures C sont representees par des classes Java du paquet.

Le marshalling J/Direct convertit les types Java en representations natives.
Les points suivants doivent etre controles avant tout appel :

  - version ANSI ou Unicode de la fonction;
  - taille 32 bits des handles et pointeurs dans la MSJVM historique;
  - sens entree/sortie des tableaux utilises pour simuler un passage par
    reference;
  - duree de vie des buffers, structures et callbacks;
  - valeur retournee et GetLastError quand l'API l'exige.

Les callbacks Win32 peuvent utiliser les classes de rappel fournies par le SDK,
souvent en relation avec com.ms.dll.Callback. La signature doit correspondre
exactement au prototype natif. Une application utilisant com.ms.win32 ne peut
pas s'executer telle quelle sur une JVM standard ou sur une autre plate-forme.

18.2 Le paquet com.ms.com

com.ms.com fournit l'infrastructure Java/COM de Microsoft. Il est utilise avec
les classes proxy generees par JActiveX ou par le Java Type Library Wizard. Il
contient notamment, selon le SDK :

  IUnknown       representation de l'interface COM fondamentale
  Variant        valeur Automation pouvant contenir plusieurs types
  ComLib         operations utilitaires, dont la liberation explicite
  GUID et types  identifiants, interfaces et structures necessaires a COM

Exemple schematique avec un proxy genere :

  import com.ms.com.*;
  import excel8.*;                 // paquet produit depuis une type library

  public class Automatisation {
      public void executer() {
          _Application app = null;
          try {
              app = (_Application)new Application();
              app.setVisible(0, true);
          } finally {
              if (app != null)
                  ComLib.release(app);
          }
      }
  }

Les signatures reelles dependent entierement du proxy genere et de la version
de la bibliotheque de types. Les parametres optionnels sont souvent representes
par Variant; les proprietes COM deviennent en general des methodes get... et
set..., et les HRESULT sont convertis par la couche d'interoperabilite.

La MSJVM gere une partie du comptage de references COM, mais le ramasse-miettes
n'offre pas une liberation deterministe. ComLib.release permet de relacher un
objet au moment voulu, ce qui est important pour les serveurs COM lies a un
thread, les applications Office et les ressources natives couteuses. Il ne
faut plus utiliser un proxy apres l'avoir libere.

Les contraintes COM usuelles restent applicables : initialisation de
l'appartement du thread, enregistrement des CLSID/ProgID, disponibilite de la
bibliotheque de types, correspondance 32 bits et propagation des erreurs. Le
paquet com.ms.com est propre au runtime Microsoft et n'est pas equivalent a
une API Java standard.

18.3 Relations entre les quatre paquets

Ces paquets peuvent apparaitre ensemble mais remplissent des fonctions
distinctes :

  com.ms.win32   appels directs des fonctions et types de l'API Win32
  com.ms.com     objets COM/Automation et classes proxy de type libraries
  com.ms.ui      composants graphiques des anciennes AFC
  com.ms.awt     implementation Microsoft et pairs Windows de java.awt

Un formulaire com.ms.ui peut reposer indirectement sur com.ms.awt et Win32;
un composant ActiveX place dans une interface peut combiner com.ms.com et
com.ms.win32. Ces dependances indirectes expliquent pourquoi la simple
suppression des quatre lignes import ne rend pas un programme portable.

19. COMPILATION ET EXECUTION
================================================================================

Outils historiques typiques :

  jvc Bonjour.java             compilation avec le compilateur Microsoft
  jview Bonjour                execution console sur la MSJVM

Les options exactes varient selon Visual J++ 1.x, 6.0 et le SDK Microsoft Java.
L'IDE stocke aussi des options dans les fichiers de projet. Un diagnostic de
migration doit relever au minimum :

  - version de JVC et de la MSJVM;
  - options activant ou interdisant les extensions Microsoft;
  - CLASSPATH et references du projet;
  - imports com.ms.*;
  - directives /** @... */;
  - bibliotheques COM et DLL externes;
  - utilisation de WFC, AWT ou applets.

Un fichier .class produit pour la MSJVM n'est pas une preuve de portabilite :
il peut referencer des classes Microsoft ou des mecanismes non disponibles sur
une JVM standard.

20. GRAMMAIRE SYNTHETIQUE DU NOYAU
================================================================================

La notation suivante est volontairement resumee. [] signifie optionnel, {}
zero ou plusieurs repetitions et | une alternative.

  classDecl ::= { modClasse } class Ident [ extends type ]
                [ implements type { , type } ] corpsClasse

  interfaceDecl ::= { modInterface } interface Ident
                    [ extends type { , type } ] corpsInterface

  membre ::= champ | methode | constructeur | classeInterne |
             interfaceInterne | initialisateur | ;

  champ ::= { modChamp } type variables ;

  methode ::= { modMethode } ( type | void ) Ident
              ( [ parametres ] ) [ throws types ]
              ( bloc | ; )

  constructeur ::= { modConstructeur } Ident
                   ( [ parametres ] ) [ throws types ] bloc

  instruction ::= bloc | ; | expression ; | declarationLocale ; |
                  if (...) instruction [ else instruction ] |
                  switch (...) { ... } |
                  while (...) instruction |
                  do instruction while (...) ; |
                  for ( ... ; ... ; ... ) instruction |
                  break [ Ident ] ; | continue [ Ident ] ; |
                  return [ expression ] ; | throw expression ; |
                  synchronized ( expression ) bloc |
                  try bloc ( catch... [ finally bloc ] | finally bloc )

  primaire ::= litteral | this | super | Ident | ( expression ) |
               new creation | primaire . Ident | primaire [ expression ] |
               primaire ( [ arguments ] )

La grammaire normative complete du noyau est celle de Java 1.1. Les sections
16 et 17 ajoutent les extensions propres au compilateur Microsoft.

21. DIFFERENCES AVEC JAVA MODERNE
================================================================================

Construction moderne                 Equivalent ou solution J++ historique
-----------------------------------  ------------------------------------------
List<String>                         Vector contenant Object + casts
for (Type x : collection)            Enumeration ou boucle indexee
enum Couleur                         constantes int ou classe finale
@Annotation                          conventions/directives propres aux outils
lambda                               interface + classe anonyme, ou delegate MS
try (Ressource r = ...)              try/finally et fermeture explicite
String dans switch                   if/else ou comparaison equals
var                                  type local explicite
record                               classe avec champs, constructeur, methodes
module                               package + CLASSPATH
java.time                            Date/Calendar historiques ou bibliotheque

Ne pas ajouter une fonctionnalite recente au seul motif que le fichier porte
l'extension .java. Le niveau de langage est determine par JVC/MSJVM, non par
le suffixe du fichier.


================================================================================
22. PORTABILITE ET MIGRATION
================================================================================

Trois categories facilitent l'audit d'un projet :

  A. Java 1.1 portable
     java.lang, java.io, java.util historique, AWT, syntaxe standard.

  B. API Microsoft remplacable
     com.ms.wfc.*, utilitaires com.ms.*, modele evenementiel WFC.

  C. liaison native
     @dll.import, COM/ActiveX, bibliotheques de types, appels Win32.

Migration vers Java : conserver A, remplacer B par AWT/Swing/JavaFX ou une
autre bibliotheque, et remplacer C par JNI/JNA/FFM ou un service externe. Les
delegates deviennent en general des listener interfaces.

Migration vers .NET : convertir la syntaxe vers C# (ou historiquement J#),
remplacer WFC par Windows Forms/WPF et convertir J/Direct vers P/Invoke. Les
details de marshalling doivent etre testes, surtout pour char, boolean, tableaux
et chaines.

La migration doit aussi traiter les ecarts d'encodage, le modele de securite
des applets, les dependances 32 bits, l'enregistrement COM et les API Windows
disparues. Une recompilation reussie ne suffit pas : des tests fonctionnels
sont indispensables.

23. EXEMPLE RECAPITULATIF PORTABLE
================================================================================

  import java.util.Vector;
  import java.util.Enumeration;

  public class Catalogue {
      private Vector articles = new Vector();

      public synchronized void ajouter(String nom) {
          if (nom == null)
              throw new IllegalArgumentException("nom nul");
          articles.addElement(nom);
      }

      public void afficher() {
          Enumeration e = articles.elements();
          while (e.hasMoreElements()) {
              String nom = (String)e.nextElement();
              System.out.println(nom);
          }
      }

      public static void main(String[] args) {
          Catalogue c = new Catalogue();
          c.ajouter("clavier");
          c.ajouter("souris");
          c.afficher();
      }
  }

Cet exemple evite les extensions Microsoft et respecte le style Java 1.1 :
Vector sans generics, Enumeration sans foreach et casts explicites.
