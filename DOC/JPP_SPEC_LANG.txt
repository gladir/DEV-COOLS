================================================================================
                  SPECIFICATIONS DU LANGAGE DE PROGRAMMATION J++
================================================================================

Document de reference pour l'integration dans le systeme d'aide DEVENV.HLP.
Dans ce document, J++ designe Microsoft Visual J++ 1.0/1.1 et 6.0, et non
Java moderne, J#, C++/CLI, ni un langage nomme "Java++" par un autre editeur.

Visual J++ est une implementation historique du langage Java accompagnee du
compilateur JVC, de la Microsoft Virtual Machine for Java (MSJVM), de
bibliotheques Microsoft et, dans Visual J++ 6.0, des Windows Foundation
Classes (WFC). Son noyau syntaxique correspond essentiellement a Java 1.1.

IMPORTANT : J++ est abandonne. La MSJVM, les WFC et les outils Visual J++ ne
doivent pas servir a de nouveaux logiciels. Cette specification documente le
langage pour la lecture, la maintenance et la migration de sources anciens.


================================================================================
1. PRESENTATION ET PERIMETRE
================================================================================

Visual J++ fut livre par Microsoft avec Visual Studio 97 puis Visual Studio
6.0. Il permettait de produire du bytecode Java execute par la MSJVM, et de
creer notamment des applications console, des applets, des applications
Windows WFC et des composants interoperables avec COM/Win32.

Le terme J++ recouvre trois couches qu'il faut distinguer :

  1. le langage Java 1.1 : classes, interfaces, exceptions, threads, etc.;
  2. les extensions du compilateur Microsoft : delegates, directives en
     commentaire, compilation conditionnelle et services d'interoperabilite;
  3. les bibliotheques Microsoft : com.ms.*, WFC, J/Direct et acces COM.

Une classe WFC n'est donc pas une construction syntaxique du langage. De meme,
un programme peut etre syntaxiquement Java tout en etant non portable parce
qu'il importe com.ms.wfc.* ou appelle Win32.

J++ reste anterieur a Java 2 et ne comprend pas les ajouts ulterieurs :
assert, strictfp, generics, annotations, enum, foreach, varargs, autoboxing,
imports statiques, lambdas, methodes default, modules, records, text blocks,
classes sealed, pattern matching et expressions switch.

1.1 Resume comparatif Java / J++

Le tableau suivant compare Java standard a Microsoft Visual J++ dans son
environnement historique. "Java" designe ici le langage et les API standard,
sans bibliotheque native ajoutee par un fournisseur.

  Fonctionnalite                    Java                  J++
  --------------------------------  --------------------  ----------------------
  Portable                          Oui                   Non
  Conforme aux specifications Java Oui                   Partiellement
  API Windows integree              Non                   Oui
  COM / ActiveX integre             Non                   Oui
  Registre Windows integre          Non                   Oui
  Delegates                         Non (a l'epoque)      Oui
  Events integres au modele MS      Non                   Oui
  Proprietes de composants          Non (*)               Oui (*)
  Plates-formes                     Toutes                Windows
  Etat actuel                       Developpement actif    Abandonne

(*) La ligne "Proprietes" demande une precision. Visual J++ et son IDE
exposent les proprietes de JavaBeans, AFC, WFC et COM comme des membres de
composants. J++ ne reconnait toutefois pas la declaration automatique C# :

  public int Age { get; set; }      // C#, pas Visual J++ 6.0

Dans une classe Java ou J++, une propriete JavaBean est implementee par des
methodes telles que getAge() et setAge(). Voir la section 15.3. Ainsi, "Oui"
dans la colonne J++ qualifie le modele composant et son integration aux outils,
pas une grammaire get/set identique a celle de C#.

Java peut appeler Windows, COM, ActiveX ou le registre par JNI, JNA ou une
autre bibliotheque externe. La valeur "Non" signifie que ces services ne font
pas partie du langage ni de l'API Java standard. A l'inverse, Visual J++ les
integre directement par la MSJVM, J/Direct et les paquets com.ms.*.

"Toutes" signifie que les implementations Java existent sur de nombreuses
plates-formes et que le code respectant les API standard est concu pour etre
portable. Cela ne garantit pas qu'un programme Java utilisant du code natif ou
une API propre a un systeme fonctionne sans adaptation partout.


================================================================================
2. UNITES DE COMPILATION ET PROGRAMME MINIMAL
================================================================================

Un fichier source porte normalement l'extension .java. Une unite de
compilation contient, dans cet ordre :

  [ declaration de package ]
  { declaration d'import }
  { declaration de classe ou d'interface }

Exemple console :

  public class Bonjour {
      public static void main(String[] args) {
          System.out.println("Bonjour depuis J++");
      }
  }

Le point d'entree est une methode public static void main(String[] args).
Le nom du fichier d'une classe publique correspond normalement au nom de la
classe : Bonjour.java.

Exemple avec package :

  package exemple.outils;
  import java.util.Vector;

  public class Demo {
      public static void main(String[] args) {
          Vector v = new Vector();
          v.addElement("un");
      }
  }

Syntaxe abstraite :

  unite          ::= [ package nomQualifie ; ] { importDecl } { typeDecl }
  importDecl     ::= import nomQualifie [ .* ] ;
  typeDecl       ::= classDecl | interfaceDecl | ;
  nomQualifie    ::= identificateur { . identificateur }


================================================================================
3. ELEMENTS LEXICAUX
================================================================================

3.1 Identificateurs

Les identificateurs sont sensibles a la casse. Une lettre, '_' ou '$' peut
commencer un identificateur; les positions suivantes acceptent aussi les
chiffres. Les lettres Unicode reconnues dependent de la version de la MSJVM et
du compilateur. Les noms commencant par '$' restent possibles mais sont a
eviter dans le code applicatif.

3.2 Mots reserves du noyau Java 1.1

  abstract    boolean      break       byte        case
  catch       char         class       const       continue
  default     do           double      else        extends
  final       finally      float       for         goto
  if          implements   import      instanceof  int
  interface   long         native      new         package
  private     protected    public      return      short
  static      super        switch      synchronized this
  throw       throws       transient   try         void
  volatile    while

const et goto sont reserves mais n'ont pas d'instruction associee. true, false
et null sont des litteraux et non des identificateurs.

Les extensions Microsoft peuvent en outre reconnaitre des mots contextuels,
notamment delegate et multicast, lorsque les extensions sont activees. Il est
prudent de ne pas les employer comme identificateurs.

3.3 Commentaires

  // commentaire jusqu'a la fin de ligne
  /* commentaire sur une ou plusieurs lignes */
  /** commentaire de documentation */

Les commentaires de bloc ne s'imbriquent pas. Certaines lignes commencant par
/** @... */ ont un sens special pour les outils Microsoft; voir section 17.

3.4 Litteraux

  entiers decimaux       0  12  2147483647  12L
  entiers octaux         077
  entiers hexadecimaux   0xFF  0XCAFE
  reels                  1.0  .5  6.02e23  3.14F  2.0D
  caracteres             'A'  '\n'  '\u00e9'
  chaines                "texte"  "ligne 1\nligne 2"
  booleens               true  false
  reference nulle        null

Echappements : \b, \t, \n, \f, \r, \", \', \\, sequences octales et
Unicode \uXXXX. Une chaine ne traverse pas directement une fin de ligne.

3.5 Formation des tokens et separateurs

Avant l'analyse grammaticale, le compilateur transforme le flux de caracteres
en tokens. Les espaces, tabulations, fins de ligne et commentaires separent les
tokens mais n'ont autrement pas de signification. Une fin de ligne ne termine
pas une instruction : le point-virgule ou la structure grammaticale le fait.

Le compilateur choisit normalement le token valide le plus long. Ainsi, >>= est
un seul operateur, tandis que > > = represente trois tokens. Deux mots doivent
etre separes s'ils formeraient autrement un identificateur unique.

  separateurs     (  )  {  }  [  ]  ;  ,  .
  operateurs      =  >  <  !  ~  ?  :  ==  <=  >=  !=  &&  ||
                   ++  --  +  -  *  /  &  |  ^  %  <<  >>  >>>
                   +=  -=  *=  /=  &=  |=  ^=  %=  <<=  >>=  >>>=

Le caractere '.' sert a la fois dans un reel, dans un nom qualifie et dans un
acces a un membre. Le contexte permet par exemple de distinguer 1.0, java.io et
objet.champ. Les caracteres # et ` n'ont aucun role lexical standard en J++.

3.6 Identificateurs en detail

Forme simplifiee :

  identificateur ::= lettreJava { lettreJava | chiffreJava }

Une lettreJava comprend les lettres Unicode reconnues, '_' et '$'. La valeur
Unicode exacte acceptee depend des tables de caracteres du vieux compilateur
JVC. Aucun mot reserve, ni true, false ou null, ne peut etre un identificateur.

  age, Age, _age, $temp, nombre2     valides
  2nombre, mon-nom, class, true      invalides

Age et age sont deux noms differents. Les conventions, non imposees par la
grammaire, emploient NomDeClasse pour les types, nomDeMethode pour les methodes
et CHAMP_CONSTANT pour les constantes static final.

3.7 Litteraux entiers en detail

Forme simplifiee :

  decimal     ::= 0 | chiffreNonNul { chiffre }
  octal       ::= 0 chiffreOctal { chiffreOctal }
  hexadecimal ::= 0x chiffreHexa { chiffreHexa }
  entier      ::= ( decimal | octal | hexadecimal ) [ l | L ]

Sans suffixe, le type est int si la valeur est representable; avec L ou l, le
type est long. Le suffixe L majuscule est preferable car l minuscule ressemble
au chiffre 1. Java 1.1/J++ n'accepte ni prefixe binaire 0b ni separateurs '_'.

Un zero initial rend un entier octal : 010 vaut huit, et 08 est invalide. Le
signe moins n'appartient pas au litteral; -1 est l'operateur unaire '-' suivi
du litteral 1. Les cas limites -2147483648 et -9223372036854775808L suivent des
regles speciales de compilation liees a cette representation.

3.8 Litteraux flottants en detail

Forme simplifiee :

  flottant ::= chiffres . [ chiffres ] [ exposant ] [ suffixe ] |
               . chiffres [ exposant ] [ suffixe ] |
               chiffres exposant [ suffixe ] |
               chiffres [ exposant ] suffixe
  exposant ::= ( e | E ) [ + | - ] chiffres
  suffixe  ::= f | F | d | D

Sans F/f, un litteral flottant est de type double; D/d le confirme. F/f donne
un float. Les formes hexadecimales flottantes et les separateurs '_' sont
posterieurs a J++. Un litteral non nul trop petit ou trop grand pour son type
est une erreur de compilation plutot qu'une conversion silencieuse en zero ou
en infini.

3.9 Caracteres, chaines et Unicode

Un litteral char contient exactement un caractere ou une sequence
d'echappement entre apostrophes. Un litteral String contient zero ou plusieurs
caracteres/echappements entre guillemets. Les echappements octaux utilisent de
un a trois chiffres avec les limites imposees par la premiere valeur octale.

Les echappements Unicode \uXXXX sont traites tres tot, avant le decoupage des
tokens. Ils peuvent donc introduire un caractere syntaxique, y compris une fin
de ligne; cette particularite exige de la prudence dans les commentaires. Deux
unites char peuvent etre necessaires pour representer un caractere Unicode hors
du plan multilingue de base.

Les litteraux String identiques sont internes par la JVM et peuvent partager
la meme reference. Le programme doit cependant utiliser equals() pour comparer
le contenu et non ==. Il n'existe ni chaine brute, ni interpolation, ni bloc de
texte multilignes dans J++ 6.0.

3.10 Reference de chacun des mots reserves

Les formes ci-dessous utilisent les conventions suivantes :

  [ element ]       element facultatif
  { element }       zero ou plusieurs repetitions
  A | B             choix entre A et B
  expression        expression du type exige par le contexte
  instruction       instruction simple ou bloc { ... }

Un mot reserve n'a pas toujours des "parametres" au sens d'une methode. Les
elements places apres le mot sont ses operandes, noms, types, expressions ou
blocs grammaticaux. Leur role est precise pour chaque entree.

abstract

  abstract class Nom { ... }
  [modificateurs] abstract Type methode(parametres) [throws types] ;

Marque une classe incomplete ou une methode sans corps. Une classe abstract ne
peut pas etre instanciee avec new. Une sous-classe concrete doit implementer
les methodes abstraites accessibles. abstract est incompatible avec final et,
pour une methode, avec private, static, native ou synchronized.

boolean

  boolean nom [= expressionBooleenne] ;
  boolean methode(parametres) { ... }

Declare le type primitif dont les seules valeurs sont true et false. Il n'est
pas implicitement convertible en entier; une condition if/while/for exige une
expression boolean.

break

  break ;
  break etiquette ;

Sans etiquette, termine le switch ou la boucle englobante la plus proche. Avec
une etiquette, termine l'instruction etiquetee correspondante. Il ne prend pas
de valeur et ne quitte pas une methode; return remplit ce role.

byte

  byte nom [= expressionEntiereCompatible] ;

Type entier signe sur 8 bits (-128 a 127). Les calculs promeuvent normalement
byte en int; un cast peut donc etre requis pour reutiliser le resultat comme
byte. Une constante int representable peut etre affectee sans cast.

case

  case expressionConstante : instructions

Introduit une branche de switch. L'operande doit etre une constante entiere
compatible avec le type du selecteur et unique dans ce switch. L'execution
continue dans le case suivant tant qu'un break, return ou throw ne l'arrete.

catch

  catch (TypeException nom) { instructions }

Suit un bloc try et capture une exception assignable au type indique. Le
parametre entre parentheses declare une variable locale contenant l'objet
exception. Plusieurs catch sont permis, du type le plus specifique au plus
general; un catch inaccessible est une erreur.

char

  char nom [= caractereOuEntierCompatible] ;

Type primitif non signe sur 16 bits, representant une unite UTF-16. Ses valeurs
vont de '\u0000' a '\uffff'. Les calculs promeuvent char en int.

class

  [modificateurs] class Nom [extends Super]
      [implements Interface {, Interface}] { membres }

Declare une classe. Nom cree le nouveau type; extends choisit au plus une
superclasse; implements donne zero ou plusieurs contrats d'interface. Le corps
contient champs, methodes, constructeurs, initialisateurs et types imbriques.

const

  aucune syntaxe executable

Mot reserve inutilise. J++ le refuse comme identificateur mais ne permet pas de
declarer une constante avec lui. Employer final, souvent static final.

continue

  continue ;
  continue etiquette ;

Passe a l'iteration suivante d'une boucle. Dans for, l'expression de mise a
jour est executee avant le prochain test. L'etiquette doit designer une boucle
englobante, contrairement a break qui peut viser une instruction etiquetee.

default

  default : instructions

Introduit la branche d'un switch utilisee lorsqu'aucun case ne correspond. Un
seul default est permis; sa position dans le bloc switch n'est pas imposee.
J++ historique ne connait pas les methodes default des interfaces modernes.

do

  do instruction while (expressionBooleenne) ;

Execute d'abord l'instruction, puis repete tant que la condition vaut true. Le
corps s'execute donc au moins une fois. Le point-virgule final est obligatoire.

double

  double nom [= expressionNumerique] ;

Type flottant IEEE 754 double precision sur 64 bits. C'est le type par defaut
des litteraux reels. Il accepte NaN, les infinis et les zeros signes produits
par les calculs, selon les regles flottantes de la JVM.

else

  if (condition) instruction else instruction

Fournit la branche executee quand la condition if vaut false. Un else est
associe au if sans else le plus proche, sauf regroupement explicite par blocs.

extends

  class Fille extends Mere { ... }
  interface Fille extends I1, I2 { ... }

Indique l'heritage. Une classe a une seule superclasse directe; une interface
peut etendre plusieurs interfaces. Les types cites doivent etre accessibles et
ne doivent pas produire un cycle d'heritage.

final

  final class Nom { ... }
  final Type variable = valeur ;
  final Type methode(parametres) { ... }

Interdit respectivement la derivation d'une classe, une seconde affectation a
une variable, ou la redefinition d'une methode. final sur une reference ne rend
pas l'objet immuable. Une variable final doit etre definitivement affectee une
fois avant utilisation.

finally

  try { ... } [catch (...) { ... }] finally { nettoyage }

Declare le bloc execute lors de la sortie normale ou exceptionnelle du try et
de ses catch, y compris apres return, break ou continue. Une sortie brutale de
la JVM peut l'empecher. Un return/throw dans finally masque la sortie initiale
et doit etre evite.

float

  float nom [= expressionCompatible] ;

Type flottant IEEE 754 simple precision sur 32 bits. Un litteral reel doit en
general porter F, par exemple 1.5F; 1.5 est un double et exige un cast.

for

  for ([initialisation] ; [condition] ; [miseAJour]) instruction

initialisation est une declaration locale ou une liste d'expressions;
condition doit etre boolean et vaut true si elle est omise; miseAJour est une
liste d'expressions executees apres chaque iteration. Chacune des trois zones
peut etre vide. Il n'existe pas de for-each en J++ 6.0.

goto

  aucune syntaxe executable

Mot reserve inutilise. Aucune instruction goto n'existe. Les transferts
structures disponibles sont break, continue, return et throw.

if

  if (expressionBooleenne) instruction [else instruction]

Execute la premiere instruction si la condition vaut true, sinon la branche
else eventuelle. La condition doit etre boolean : if (1) est invalide.

implements

  class Nom implements I1 [, I2 ...] { ... }

Liste les interfaces promises par une classe. Une classe concrete doit fournir
les methodes d'interface non deja implementees par une superclasse. implements
n'accepte que des types interface.

import

  import paquet.Type ;
  import paquet.* ;

Rend un type utilisable par son nom simple dans l'unite de compilation. La
forme * importe les types directs d'un paquet, pas ses sous-paquets. Elle ne
charge pas la classe et n'importe ni membres ni paquet recursivement.

instanceof

  expressionReference instanceof TypeReference

Produit true si la valeur non nulle peut etre convertie vers le type indique
sans ClassCastException; null produit false. Les types des deux operandes
doivent etre compatibles a la compilation.

int

  int nom [= expressionEntiere] ;

Type entier signe sur 32 bits. C'est le type usuel des calculs entiers, indices
de tableaux et selecteurs switch. Le debordement utilise les 32 bits de poids
faible sans lever d'exception.

interface

  [public | protected | private | static | abstract]
  interface Nom [extends I1, I2 ...] { membres }

Declare un contrat. Les methodes sont implicitement public abstract et sans
corps; les champs sont public static final et doivent etre initialises. Les
modificateurs possibles dependent du fait que l'interface soit de premier
niveau ou imbriquee.

long

  long nom [= expressionEntiereCompatible] ;

Type entier signe sur 64 bits. Un litteral long utilise L, par exemple 10L.
Les decalages d'un long n'emploient que les 6 bits faibles de leur distance.

native

  [modificateurs] native Type methode(parametres) [throws types] ;

Declare une methode implementee hors du bytecode Java. Elle se termine par ';'
et n'a pas de corps. Dans J++, J/Direct, RNI ou la MSJVM assurent la liaison;
les parametres doivent pouvoir etre convertis selon les regles de marshalling.

new

  new TypeClasse(arguments)
  new TypeElement[dimension] { [dimension] } [initialiseur]
  new TypeInterface(arguments) { corpsClasseAnonyme }

Cree un objet ou un tableau et retourne sa reference. Les arguments choisissent
un constructeur compatible. Les dimensions de tableau sont des int non
negatifs; une taille negative leve NegativeArraySizeException.

package

  package nom.qualifie ;

Doit etre la premiere declaration non commentee de l'unite. Il place les types
du fichier dans le paquet indique et influence leur nom complet et leur acces.
En son absence, les types appartiennent au paquet sans nom.

private

  private declarationDeMembre

Limite l'acces a la classe qui declare le membre. Il s'applique aux membres et
types imbriques, pas a une classe de premier niveau Java 1.1. Une methode
private n'est pas redefinie polymorphiquement dans une sous-classe.

protected

  protected declarationDeMembre

Autorise l'acces depuis le meme paquet et, sous certaines regles de receveur,
depuis les sous-classes situees ailleurs. Il s'applique aux membres et types
imbriques, pas a un type de premier niveau.

public

  public declaration

Rend la declaration accessible partout ou son type englobant est accessible.
Une classe ou interface public de premier niveau reside normalement dans un
fichier portant son nom. public ne contourne pas l'inaccessibilite du contenant.

return

  return ;
  return expression ;

Termine la methode ou le constructeur courant. Une methode void et un
constructeur utilisent la forme sans valeur; une methode non void doit fournir
une expression convertible vers son type retour. finally s'execute avant le
transfert effectif.

short

  short nom [= expressionEntiereCompatible] ;

Type entier signe sur 16 bits (-32768 a 32767). Comme byte et char, il est
promu en int pour la plupart des operations arithmetiques.

static

  static Type champ ;
  static Type methode(parametres) { ... }
  static { initialisationDeClasse }
  static class TypeImbrique { ... }

Associe le membre a la classe et non a chaque objet. Une methode static n'a pas
de this et ne peut acceder directement aux membres d'instance. Un bloc static
s'execute une fois lors de l'initialisation de la classe.

super

  super(arguments)              // premier appel d'un constructeur
  super.membre
  super.methode(arguments)

Designe la partie superclasse de l'objet courant. super(...) appelle un
constructeur direct et doit etre la premiere instruction du constructeur.
super.methode force l'implementation de la superclasse immediate.

switch

  switch (expressionEntiere) {
      { case constante : instructions }
      [ default : instructions ]
  }

Evalue une fois son selecteur char, byte, short ou int, puis saute vers le case
correspondant. L'execution continue ensuite sequentiellement. Ni String, ni
enum, ni expression switch moderne ne sont disponibles en J++ 6.0.

synchronized

  synchronized Type methode(parametres) { ... }
  synchronized (expressionReferenceNonNulle) { instructions }

Acquiert un moniteur avant d'executer le corps et le libere a la sortie. Une
methode d'instance verrouille this; une methode static verrouille l'objet Class.
Le bloc verrouille l'objet calcule et leve NullPointerException s'il vaut null.

this

  this
  this.membre
  this(arguments)

Designe l'instance courante. this(...) appelle un autre constructeur de la
meme classe et doit etre la premiere instruction. this est indisponible dans un
contexte static. Il peut lever une ambiguite entre champ et parametre.

throw

  throw expressionThrowable ;

Evalue puis lance un objet assignable a Throwable. Lancer null provoque
NullPointerException. Une exception verifiee doit etre capturee ou annoncee
par throws. L'instruction ne revient pas normalement.

throws

  Type methode(parametres) throws E1 [, E2 ...] { ... }

Annonce les types d'exceptions verifiees qu'une methode ou un constructeur
peut laisser sortir. La liste ne cree ni ne lance une exception et n'est pas
une liste de parametres; throw effectue le lancement concret.

transient

  transient Type champ ;

Indique au mecanisme standard de serialisation de ne pas sauvegarder ce champ.
Il ne s'applique qu'aux champs. Apres deserialisation standard, le champ reprend
sa valeur par defaut, sauf restauration explicite par la classe.

try

  try { instructions }
      catch (Type nom) { instructions } { catch ... }
      [finally { instructions }]
  try { instructions } finally { instructions }

Delimite une region dont les exceptions sont traitees par catch et/ou dont la
sortie est protegee par finally. Au moins un catch ou un finally est requis.
J++ ne connait ni try-with-resources ni multi-catch.

void

  void methode(parametres) { ... }

Indique qu'une methode ne retourne aucune valeur. void n'est pas un type de
variable, d'argument ou d'element de tableau. Une methode void peut employer
return; mais pas return expression;.

volatile

  volatile Type champ ;

Demande que les acces au champ partage respectent la semantique volatile de la
JVM Microsoft de l'epoque. Le modificateur ne s'applique pas aux variables
locales ni methodes. Il ne rend pas atomique une operation composee comme i++.

while

  while (expressionBooleenne) instruction

Teste la condition avant chaque iteration et execute le corps tant qu'elle
vaut true. Si elle est fausse au premier test, le corps ne s'execute jamais.

3.11 Litteraux reserves true, false et null

true et false sont les deux litteraux boolean. Ils s'emploient comme valeurs et
conditions, sans guillemets ni majuscule : True et FALSE seraient de simples
identificateurs. null est la reference nulle et peut etre converti vers tout
type reference, mais vers aucun type primitif. Dereferencer null leve
NullPointerException.

  boolean termine = false;
  String texte = null;
  if (!termine && texte == null) { ... }

3.12 Extensions lexicales Microsoft : delegate et multicast

Lorsque les extensions J++ sont actives, delegate introduit un type d'appel
fortement type. Sa liste entre parentheses decrit les parametres que la methode
cible doit accepter; le type place avant son nom est le type retourne :

  [modificateurs] delegate TypeRetour Nom(Type p1, Type p2) ;

  delegate void Notification(Object source, int code);

Une instance lie une methode de signature compatible :

  Notification n = new Notification(objet.traiter);
  n.invoke(source, 10);

multicast modifie une declaration de delegate afin que l'instance puisse
representer une chaine de plusieurs methodes compatibles :

  [modificateurs] multicast delegate TypeRetour Nom(parametres) ;

Ces deux mots et la syntaxe de reference liee objet.methode ne font pas partie
de Java 1.1. Leur disponibilite depend des options de JVC et du runtime
Microsoft. Les details du modele evenementiel figurent a la section 16.


================================================================================
4. TYPES ET VALEURS
================================================================================

4.1 Classification generale

J++ reprend le systeme de types de Java 1.1. Il est statique : le type de toute
expression est determine a la compilation. Il est aussi fortement type : une
valeur ne peut etre employee comme un autre type que si une conversion definie
par le langage existe.

Deux familles de types existent :

  types primitifs    boolean, byte, short, int, long, char, float, double
  types reference    classes, interfaces et tableaux

Une variable primitive contient directement une valeur. Une variable reference
contient null ou une reference vers un objet. Les objets eux-memes sont crees
sur le tas gere par la machine virtuelle et leur memoire est recuperee par le
ramasse-miettes.

void n'est ni un type primitif de variable ni un type reference. Il indique
seulement qu'une methode ne retourne aucune valeur :

  void afficher() {
      System.out.println("Bonjour");
  }

J++ 6.0 ne possede ni type unsigned general, ni pointeur Java arithmetique, ni
structure valeur, ni enum, ni generics, ni type nullable primitif. Les types
propres a Win32 et COM sont representes par les classes et conventions des
paquets com.ms.*; ils ne modifient pas le systeme de types du noyau.

4.2 Type boolean

  Domaine             true ou false
  Litteraux           true, false
  Valeur par defaut   false
  Classe enveloppe    java.lang.Boolean

La taille de stockage de boolean n'est pas fixee par le langage. boolean n'est
pas un entier : aucune conversion implicite ou explicite standard n'existe
entre boolean et byte/int. Ces exemples sont invalides :

  boolean b = 1;          // invalide
  int n = (int)true;      // invalide

Les conditions de if, while, do et for exigent une expression boolean :

  boolean pret = true;
  if (pret) { ... }

Les operateurs disponibles sont !, &, |, ^, ==, !=, && et ||. Avec boolean,
& et | evaluent toujours leurs deux operandes; && et || peuvent ne pas evaluer
le second operande.

4.3 Types entiers byte, short, int et long

  Type    Bits  Domaine                                  Defaut  Suffixe
  ------  ----  ---------------------------------------  ------  -------
  byte       8  -128 a 127                                   0  aucun
  short     16  -32768 a 32767                               0  aucun
  int       32  -2147483648 a 2147483647                     0  aucun
  long      64  -9223372036854775808 a 9223372036854775807   0L L ou l

Tous ces types sont signes et utilisent une representation binaire en
complement a deux pour les operations definies par la JVM. int est le type
entier usuel : les litteraux sans suffixe et la plupart des calculs sur byte,
short et char produisent un int.

Exemples :

  byte niveau = 100;
  short port = 8080;
  int population = 1000000;
  long distance = 9876543210L;

Un litteral int constant peut etre affecte directement a byte ou short si sa
valeur est representable :

  byte a = 127;           // valide
  byte b = 128;           // invalide sans cast
  byte c = (byte)128;     // valide, resultat -128

Le debordement entier ne leve pas d'exception. Le resultat conserve les bits
de poids faible :

  int maximum = 2147483647;
  int suivant = maximum + 1;       // -2147483648

La division entiere tronque vers zero. Le reste % a le signe du dividende :

  7 / 3     vaut  2
 -7 / 3     vaut -2
 -7 % 3     vaut -1

Diviser un entier par zero leve ArithmeticException. Le cas
-2147483648 / -1 deborde et conserve -2147483648 sans exception, conformement
aux regles entieres Java.

Classes enveloppes correspondantes :

  byte  -> java.lang.Byte
  short -> java.lang.Short
  int   -> java.lang.Integer
  long  -> java.lang.Long

Il n'existe pas d'autoboxing : Integer i = 1; est invalide en J++ historique.
Il faut ecrire new Integer(1), puis i.intValue() pour recuperer le int.

4.4 Type char

  Taille              16 bits non signes
  Domaine             '\u0000' a '\uffff', soit 0 a 65535
  Valeur par defaut   '\u0000'
  Classe enveloppe    java.lang.Character

char contient une unite de codage Unicode sur 16 bits. Une valeur du plan
multilingue de base tient dans un char. Les caracteres Unicode hors de ce plan,
formalises plus tard sous forme de paires de substituts UTF-16, demandent deux
char et ne doivent pas etre assimiles a une seule valeur char.

  char lettre = 'A';
  char accent = '\u00e9';
  int code = lettre;             // conversion char vers int : 65

char participe aux calculs entiers et est alors promu en int :

  char c = 'A';
  c = (char)(c + 1);             // 'B'; cast necessaire

Une constante int representable peut etre affectee directement :

  char nul = 0;
  char max = 65535;

char n'est pas un entier signe. Le cast (char)-1 produit '\uffff'.

4.5 Types flottants float et double

  Type    Format IEEE 754       Precision approx.  Defaut  Suffixe
  ------  --------------------  ------------------  ------  -------
  float   binaire 32 bits       6 a 9 chiffres      0.0F   F ou f
  double  binaire 64 bits       15 a 17 chiffres    0.0D   D/d ou aucun

float et double representent les nombres finis, les deux zeros signes, les
infinis positif et negatif et NaN (Not a Number). Les constantes utiles sont
notamment Float.NaN, Float.POSITIVE_INFINITY, Double.NaN et
Double.NEGATIVE_INFINITY.

  float taux = 1.5F;
  double pi = 3.141592653589793;
  double infini = 1.0 / 0.0;       // Infinity, aucune exception
  double nan = 0.0 / 0.0;          // NaN

Contrairement a la division entiere, la division flottante par zero ne leve pas
ArithmeticException. NaN n'est egal a aucune valeur, pas meme a lui-meme :

  nan == nan                       // false
  nan != nan                       // true

Les comparaisons <, <=, > et >= avec NaN sont toutes false. Pour tester NaN,
utiliser Float.isNaN(x) ou Double.isNaN(x).

Les calculs flottants sont arrondis vers la valeur representable la plus proche
selon les regles de la JVM. Visual J++ est anterieur au mot strictfp. Selon le
processeur et la version de la MSJVM, des valeurs intermediaires peuvent avoir
une plage d'exposant plus large que le type declare. Le stockage dans un champ,
un tableau ou certains points de conversion force la valeur au format attendu.

4.6 Valeurs par defaut et initialisation

Les champs d'instance, champs static et elements de tableaux recoivent
automatiquement une valeur par defaut :

  Type                         Valeur par defaut
  ---------------------------  -----------------
  boolean                      false
  byte, short, int             0
  long                         0L
  char                         '\u0000'
  float                        0.0F
  double                       0.0D
  tout type reference          null

Exemple :

  class Etat {
      int compteur;             // 0
      boolean termine;          // false
      String message;           // null
  }

Les variables locales ne recoivent pas de valeur par defaut. Le compilateur
verifie leur affectation certaine :

  int x;
  System.out.println(x);        // erreur : x non initialise

Une variable locale peut toutefois ne pas etre affectee sur un chemin qui ne la
lit jamais. L'analyse est faite pour chaque chemin de controle.

4.7 Promotions numeriques

Les promotions determinent le type utilise pendant un calcul.

Promotion unaire :

  byte, short ou char -> int
  int                 -> int
  long                -> long
  float               -> float
  double              -> double

Promotion binaire, appliquee aux deux operandes :

  si l'un est double  -> les deux deviennent double
  sinon, si float     -> les deux deviennent float
  sinon, si long      -> les deux deviennent long
  sinon               -> les deux deviennent int

Exemples :

  byte a = 10, b = 20;
  int somme = a + b;             // resultat int

  int n = 3;
  double d = 2.0;
  double produit = n * d;        // n est converti en double

  long grand = 1L;
  float f = grand + 1.0F;        // grand est converti en float

Les operateurs de decalage suivent une regle differente : chaque operande subit
une promotion unaire; le type du resultat est celui de l'operande gauche. Seuls
les 5 bits faibles de la distance sont utilises pour int, et les 6 bits faibles
pour long.

4.8 Conversions numeriques elargissantes

Une conversion elargissante est implicite :

  byte  -> short -> int -> long -> float -> double
  char  -> int   -> long -> float -> double

Les chemins supplementaires compatibles avec ce graphe sont egalement permis,
par exemple byte vers long ou int vers double.

  byte petit = 12;
  int entier = petit;
  long grand = entier;
  double reel = grand;

Une conversion elargissante ne deborde pas au sens de la plage, mais peut perdre
de la precision lors du passage d'un entier vers float ou double :

  int exact = 16777217;
  float arrondi = exact;         // peut devenir 16777216

boolean ne participe a aucune conversion numerique. Il n'existe pas de
conversion implicite entre char et byte/short, car leurs domaines ne sont pas
inclus l'un dans l'autre.

4.9 Conversions numeriques reductrices et casts

Une conversion reductrice exige normalement un cast :

  TypeCible nom = (TypeCible)expression;

Conversions reductrices possibles :

  double -> float -> long -> int -> short -> byte
  types entiers entre eux, y compris vers et depuis char

Pour une conversion entier vers entier, les bits de poids fort sont supprimes
si necessaire; le resultat est ensuite interprete selon le type cible :

  int n = 300;
  byte b = (byte)n;              // 44
  char c = (char)-1;             // '\uffff'

Pour float/double vers un entier :

  - la partie fractionnaire est tronquee vers zero;
  - NaN devient zero;
  - une valeur trop grande est ramenee au maximum du type intermediaire;
  - une valeur trop petite est ramenee au minimum;
  - une reduction finale vers byte, short ou char conserve les bits utiles.

  int a = (int)3.9;              // 3
  int b = (int)-3.9;             // -3
  int c = (int)Double.NaN;       // 0

Un cast indique que le programmeur accepte une perte possible; il ne controle
pas la plage et ne leve pas d'exception pour un debordement numerique.

4.10 Conversions constantes lors d'une affectation

Une expression constante de type int peut etre affectee sans cast a byte,
short ou char si sa valeur est representable :

  final int DIX = 10;
  byte a = DIX;                  // valide
  short b = 30000;               // valide
  char c = 65;                   // valide, 'A'

  int variable = 10;
  byte d = variable;             // invalide : expression non constante
  byte e = 200;                  // invalide : hors du domaine

Cette facilite s'applique aux constantes calculees a la compilation :

  byte masque = 1 << 3;          // 8, representable

Elle ne transforme pas le type de l'expression arithmetique generale : a + b
reste int lorsque a et b sont byte.

4.11 Types reference : classes

Une variable de type classe contient null ou une reference vers une instance de
cette classe ou d'une sous-classe :

  Point p = new Point(1, 2);
  Object o = p;                  // Point derive de Object
  Point absent = null;

Toutes les classes derivent directement ou indirectement de java.lang.Object.
Object declare notamment equals(), hashCode(), toString(), getClass(), clone(),
finalize(), wait(), notify() et notifyAll(), avec les visibilites et contrats de
Java 1.1.

Le type compile de la variable determine quels membres peuvent etre nommes. Le
type reel de l'objet determine quelle implementation d'une methode d'instance
redefinie est appelee :

  Animal a = new Chien();
  a.crier();                     // appelle Chien.crier()

Les champs et methodes static sont resolus principalement a partir du type
compile; ils ne suivent pas le meme polymorphisme que les methodes d'instance.

4.12 Types reference : interfaces

Une variable interface contient null ou une reference vers un objet dont la
classe implemente cette interface :

  Runnable travail = new MonTravail();
  travail.run();

Une classe peut implementer plusieurs interfaces. Une interface peut etendre
plusieurs interfaces. Le type interface limite les operations visibles a ses
membres et a ceux de ses superinterfaces; un cast est necessaire pour retrouver
des membres propres a la classe concrete.

  if (travail instanceof MonTravail) {
      MonTravail m = (MonTravail)travail;
      m.methodeSpeciale();
  }

Une interface n'est pas instanciable directement. Une classe anonyme peut
toutefois l'implementer dans une expression new.

4.13 Types reference : tableaux

Un type tableau s'ecrit avec une paire [] par dimension :

  int[] vecteur;
  String[][] matrice;
  int ancienStyle[];             // valide mais moins recommande

Creation et initialisation :

  vecteur = new int[10];
  int[] premiers = { 2, 3, 5, 7 };
  String[][] noms = new String[2][3];
  int[][] irregulier = new int[2][];
  irregulier[0] = new int[3];
  irregulier[1] = new int[8];

Chaque tableau est un objet :

  - sa taille fixe est accessible par le champ final length;
  - ses indices vont de 0 a length - 1;
  - un mauvais indice leve ArrayIndexOutOfBoundsException;
  - un acces par une reference null leve NullPointerException;
  - ses elements recoivent leur valeur par defaut;
  - il derive de Object et implemente Cloneable et Serializable selon le
    runtime/bibliotheque disponible.

Les tableaux de types reference sont covariants :

  String[] textes = new String[2];
  Object[] objets = textes;      // conversion permise
  objets[0] = new Integer(1);    // ArrayStoreException a l'execution

Les tableaux primitifs ne sont pas covariants entre eux : int[] n'est ni un
long[] ni un Object[]. Il reste cependant un Object.

Un tableau multidimensionnel est un tableau de tableaux. Ses lignes peuvent
avoir des longueurs differentes; new int[3][4] cree un tableau externe et trois
tableaux internes distincts.

4.14 Chaines String et StringBuffer

java.lang.String est une classe reference speciale dont le langage reconnait
les litteraux et la concatenation par + :

  String nom = "Ada";
  String message = "Bonjour " + nom + " numero " + 1;

Un objet String est immuable. Une operation qui semble le modifier cree une
nouvelle chaine ou emploie un tampon interne :

  String s = "a";
  s = s + "b";                   // nouvelle valeur String "ab"

Les litteraux et expressions constantes String peuvent etre internes. == teste
l'identite des references, pas le contenu :

  String a = new String("abc");
  String b = new String("abc");
  a == b                         // false en general
  a.equals(b)                    // true

StringBuffer est la classe mutable historique pour construire efficacement une
chaine :

  StringBuffer buf = new StringBuffer();
  buf.append("Age=").append(42);
  String resultat = buf.toString();

StringBuilder n'existe pas dans Java 1.1. Une concatenation String peut etre
traduite par le compilateur en operations StringBuffer.

4.15 Classes enveloppes et absence d'autoboxing

Chaque type primitif possede une classe enveloppe :

  boolean -> Boolean       char   -> Character
  byte    -> Byte          short  -> Short
  int     -> Integer       long   -> Long
  float   -> Float         double -> Double

Les enveloppes sont des objets immuables. En J++ historique, les conversions
entre primitif et enveloppe sont toujours explicites :

  Integer objet = new Integer(42);  // boxing manuel
  int valeur = objet.intValue();    // unboxing manuel

  Vector liste = new Vector();
  liste.addElement(new Integer(42));
  int n = ((Integer)liste.elementAt(0)).intValue();

Les methodes parseInt, valueOf et toString disponibles dependent de la version
exacte de java.lang livree avec la MSJVM, mais le principe reste identique.
Integer ne derive pas de int et un cast (Integer) ne peut pas convertir
directement une valeur primitive.

4.16 null et dereferencement

null est le litteral de reference nulle. Il est convertible vers tout type
classe, interface ou tableau :

  String texte = null;
  Runnable action = null;
  int[] valeurs = null;

Il n'est convertible vers aucun type primitif. Toute reference peut etre
comparee a null par == ou !=. L'acces a un membre d'instance, a un element de
tableau ou a length par null leve NullPointerException :

  if (texte != null)
      System.out.println(texte.length());

Appeler une methode static au moyen d'une expression reference est autorise
par l'ancien langage mais deconseille : la resolution est statique et peut ne
pas dereferencer l'objet comme le lecteur s'y attend.

4.17 Egalite, identite et equivalence

Pour les primitifs, == et != comparent les valeurs apres promotion numerique.
boolean ne se compare qu'a boolean. Pour les references, ils comparent
l'identite : les deux operandes designent-ils le meme objet ou sont-ils tous
deux null ?

  Point p = new Point(1, 2);
  Point q = p;
  p == q                         // true

equals() est une methode virtuelle definie par Object. Une classe peut la
redefinir pour comparer un contenu logique. Lorsqu'elle le fait, elle doit
normalement redefinir hashCode() de facon coherente.

Les tableaux ne redefinissent pas equals() pour leur contenu dans l'API
historique : deux tableaux distincts restent inegaux selon equals(), meme si
leurs elements sont identiques. Une comparaison element par element est alors
necessaire.

4.18 Conversions de references

Conversions elargissantes implicites :

  sous-classe -> superclasse
  classe      -> interface qu'elle implemente
  sous-interface -> superinterface
  tableau reference -> Object, Cloneable ou Serializable
  tableau S[] -> tableau T[] si S -> T est elargissante
  null -> tout type reference

Exemple :

  Chien chien = new Chien();
  Animal animal = chien;
  Object objet = animal;

Une conversion reductrice exige un cast :

  Animal animal = obtenirAnimal();
  Chien chien = (Chien)animal;

Le cast est autorise a la compilation lorsque les types peuvent plausiblement
etre lies. A l'execution :

  - null caste vers un type reference reste null;
  - une instance compatible est acceptee;
  - une instance incompatible leve ClassCastException.

instanceof permet le test prealable :

  if (animal instanceof Chien) {
      Chien chien = (Chien)animal;
      chien.aboyer();
  }

Il n'existe pas de conversion definie par l'utilisateur ni de surcharge des
operateurs de cast en Java/J++.

4.19 Conversions String

L'operateur + convertit un operande en String lorsqu'un autre operande est une
String. Les primitifs utilisent leur representation textuelle; une reference
non nulle utilise String.valueOf, qui appelle normalement toString(); null
devient le texte "null".

  String s1 = "valeur=" + 42;        // "valeur=42"
  String s2 = "actif=" + true;       // "actif=true"
  String s3 = "objet=" + (Object)null; // "objet=null"

L'association de gauche a droite est importante :

  "total=" + 1 + 2                // "total=12"
  "total=" + (1 + 2)              // "total=3"

Il n'existe aucune conversion generale implicite de String vers nombre. Il faut
employer les methodes des classes enveloppes, par exemple :

  int n = Integer.parseInt("42");
  double d = Double.valueOf("3.5").doubleValue();

Une chaine invalide leve NumberFormatException.

4.20 Passage des arguments et affectation

Java et J++ passent toujours les arguments par valeur.

Pour un primitif, la valeur numerique ou booleenne est copiee :

  void incrementer(int x) { x++; }   // ne modifie pas l'argument appelant

Pour un objet, la reference est copiee. La methode peut modifier l'objet vise,
mais remplacer sa copie locale de la reference ne remplace pas la variable de
l'appelant :

  void modifier(Point p) {
      p.x = 10;                       // visible par l'appelant
      p = new Point(0, 0);            // seulement la copie locale
  }

Les tableaux suivent la meme regle puisqu'ils sont des objets. Les conventions
J/Direct peuvent utiliser un tableau a un element pour simuler un parametre
natif de sortie ou passe par adresse; cela reste une convention de marshalling,
pas un passage par reference du langage J++.

4.21 Type compile, type reel et liaison dynamique

Toute expression a un type compile connu du compilateur. Une reference peut
designer a l'execution une instance d'un sous-type :

  Forme f = new Cercle();

Ici, Forme est le type compile de f et Cercle son type reel. Les membres
accessibles sont controles avec Forme. Une methode d'instance redefinie est
selectionnee dynamiquement d'apres Cercle. Les champs, methodes static et
methodes private ne suivent pas cette selection dynamique.

getClass() retourne l'objet java.lang.Class representant le type reel :

  Class typeReel = f.getClass();
  String nom = typeReel.getName();

Deux classes de meme nom chargees par deux chargeurs de classes differents
peuvent etre considerees comme deux types distincts par la JVM.

4.22 Compatibilite d'affectation et d'appel

Une expression est compatible avec une variable, un parametre ou un retour si
au moins une des operations suivantes suffit :

  - identite de type;
  - conversion primitive elargissante;
  - conversion de reference elargissante;
  - conversion constante int vers byte, short ou char dans les cas permis;
  - conversion de null vers un type reference.

Le compilateur n'insere ni cast reducteur, ni boxing, ni conversion String vers
nombre. Pour une surcharge, il cherche une methode applicable aux arguments et
choisit la plus specifique. Java 1.1/J++ ne dispose pas de parametres
optionnels, nommes ou varargs :

  void f(int x) { ... }
  void f(long x) { ... }

  f(1);                           // choisit f(int)
  f(1L);                          // choisit f(long)

Le type retour seul ne participe jamais au choix d'une surcharge.

4.23 Types et extensions Microsoft

Les extensions Microsoft reposent toujours sur les types precedents :

  - un delegate est un type reference genere par le compilateur;
  - un handle Win32 est souvent represente par int dans la MSJVM 32 bits;
  - un pointeur ou parametre de sortie peut etre represente par une classe,
    un tableau ou un type de com.ms.dll/com.ms.win32;
  - une valeur Automation utilise souvent com.ms.com.Variant;
  - une interface COM est exposee comme une interface reference Java;
  - les structures Win32 deviennent des classes de donnees Java.

Exemple conceptuel :

  Variant valeur = new Variant(42);
  int hwnd = 0;
  User32.MessageBox(hwnd, "Texte", "Titre", winm.MB_OK);

Ces correspondances ne sont pas des equivalences universelles. Leur taille,
leur sens entree/sortie, leur encodage et leur duree de vie sont fixes par les
declarations J/Direct et le runtime Microsoft. Il faut notamment eviter de
supposer qu'un int reste un type de pointeur correct hors de la MSJVM Win32.

4.24 Resume des erreurs liees aux types

  Erreur/exception                    Cause typique
  ----------------------------------  ------------------------------------------
  erreur de compilation               types incompatibles sans conversion
  ClassCastException                  cast de reference incompatible
  NullPointerException                dereferencement de null
  ArrayStoreException                 objet incompatible stocke dans un tableau
  ArrayIndexOutOfBoundsException      indice hors de 0..length-1
  NegativeArraySizeException          dimension negative a la creation
  ArithmeticException                 division ou reste entier par zero
  NumberFormatException               texte numerique invalide

Un debordement entier, une perte de precision flottante ou un cast numerique
reducteur ne leve normalement aucune exception. La verification de plage doit
etre programmee explicitement lorsque ces cas sont interdits par l'application.


================================================================================
5. VARIABLES, CHAMPS ET CONSTANTES
================================================================================

5.1 Categories de variables

Le langage distingue sept emplacements contenant une valeur :

  1. champs d'instance;
  2. champs de classe, declares static;
  3. variables locales;
  4. parametres de methodes et de constructeurs;
  5. parametres de gestionnaires catch;
  6. composantes de tableaux;
  7. variables cachees creees par le compilateur ou la machine virtuelle.

Les six premieres categories sont visibles dans le langage source. La derniere
comprend par exemple la reference vers l'instance englobante d'une classe
interne ou des temporaires de compilation; leurs noms et leur representation ne
font pas partie de la specification.

Une variable possede :

  - un type fixe a la compilation;
  - un nom, sauf pour certaines variables cachees;
  - une valeur courante;
  - une portee lexicale, qui indique ou son nom est utilisable;
  - une duree de vie, qui indique combien de temps son stockage existe;
  - eventuellement des modificateurs tels que final, static ou volatile.

Le contenu d'une variable primitive est une valeur primitive. Le contenu d'une
variable reference est null ou une reference; l'objet vise reste distinct de la
variable qui contient cette reference.

5.2 Syntaxe generale des declarations

Declaration simple :

  [modificateurs] Type nom [= initialiseur] ;

Declaration de plusieurs variables du meme type :

  [modificateurs] Type nom1 [= init1], nom2 [= init2], nom3 ;

Exemples :

  int compteur;
  int x = 1, y = 2;
  String message = "Bonjour";
  final int maximum = 100;

La virgule separe des declarateurs, pas des instructions. Chaque initialiseur
est evalue de gauche a droite :

  int a = 1, b = a + 1, c = b + 1;

Le type apparait une seule fois et s'applique a tous les declarateurs. J++ 6.0
n'a ni inference locale var, ni declaration multiple de types differents, ni
deconstruction.

La notation de tableau peut suivre le type ou chaque nom :

  int[] a, b;              // a et b sont int[]
  int c[], d;              // c est int[], d est int
  int[] e[];               // e est int[][]

La premiere forme est recommandee, car elle rend le type commun plus visible.

5.3 Champs d'instance

Syntaxe :

  [public | protected | private]
  [final] [transient] [volatile] Type nom [= initialiseur] ;

Un champ d'instance appartient a un objet. Chaque instance possede son propre
emplacement :

  class Compteur {
      private int valeur;

      public void incrementer() {
          valeur++;
      }
  }

  Compteur a = new Compteur();
  Compteur b = new Compteur();
  a.incrementer();         // ne modifie pas b.valeur

Dans une methode d'instance, un champ peut etre nomme directement ou avec this :

  valeur = 10;
  this.valeur = 10;

Depuis une autre classe, l'acces passe par une expression reference :

  objet.champ

L'expression est evaluee avant l'acces. Si elle vaut null, l'acces a un champ
d'instance leve NullPointerException.

Un champ est selectionne statiquement selon le type compile de la reference; il
n'est pas polymorphe comme une methode redefinie :

  class A { int valeur = 1; }
  class B extends A { int valeur = 2; }

  A objet = new B();
  objet.valeur             // 1, champ declare par A

Le champ de B masque celui de A mais ne le remplace pas. Dans B, super.valeur
accede au champ de A.

5.4 Champs de classe static

Syntaxe :

  [visibilite] static [final] [transient] [volatile]
  Type nom [= initialiseur] ;

Un champ static appartient a la classe. Il existe en un seul exemplaire par
classe chargee, et non un exemplaire par objet :

  class Compteur {
      private static int nombreInstances;

      Compteur() {
          nombreInstances++;
      }

      public static int getNombreInstances() {
          return nombreInstances;
      }
  }

Acces recommande :

  Compteur.getNombreInstances();
  Compteur.nombreInstances;       // seulement si le champ est accessible

Java 1.1 autorise egalement objet.champStatic, mais cette forme est trompeuse :
le champ est choisi selon le type compile de l'expression, pas selon l'objet.
Utiliser le nom de classe rend la liaison statique explicite.

Une methode static peut acceder directement aux champs static de sa classe.
Elle ne peut acceder a un champ d'instance sans posseder une reference vers un
objet, car elle n'a pas de this.

Chaque chargeur de classes peut charger sa propre copie d'une classe; chacune
possede alors ses propres champs static. "Une seule copie" signifie donc une
copie par classe telle que definie par la JVM, pas necessairement une copie pour
tout le processus.

5.5 Variables locales

Syntaxe dans un bloc :

  [final] Type nom [= initialiseur] ;

Exemple :

  void calculer() {
      int total = 0;
      final int facteur = 2;

      for (int i = 0; i < 10; i++) {
          int produit = i * facteur;
          total += produit;
      }
  }

Les modificateurs public, protected, private, static, transient et volatile ne
s'appliquent pas aux variables locales. final est permis.

Une variable locale est creee lorsque l'execution atteint son bloc ou son
declarateur et cesse d'etre accessible a la sortie de sa portee. Son stockage
peut etre place dans une pile, un registre ou optimise; cette representation
n'est pas garantie par le langage.

Contrairement aux champs, une locale n'a aucune valeur par defaut. Elle doit
etre definitivement affectee avant toute lecture. Le ramasse-miettes peut
recuperer un objet des qu'aucune reference encore utilisable ne le rend
atteignable, meme si le bloc source n'est pas encore termine.

5.6 Parametres

Syntaxe d'une methode ou d'un constructeur :

  TypeRetour methode([final] Type p1, [final] Type p2) { ... }
  NomClasse([final] Type p1, [final] Type p2) { ... }

Les parametres sont des variables locales initialisees avec les valeurs des
arguments lors de l'appel :

  int additionner(int gauche, int droite) {
      return gauche + droite;
  }

J++ passe toujours les arguments par valeur. Affecter un parametre primitif ne
modifie pas la variable de l'appelant. Pour un objet, la reference est copiee :
la methode peut modifier l'objet vise, mais remplacer le parametre ne remplace
pas la reference de l'appelant.

  void changer(Point p) {
      p.x = 10;                   // modifie l'objet de l'appelant
      p = new Point(0, 0);        // modifie seulement le parametre local
  }

Un parametre final ne peut pas recevoir une nouvelle valeur dans la methode,
mais l'objet qu'il designe peut rester mutable.

Java 1.1/J++ ne possede pas de parametres optionnels, nommes, par defaut ou
varargs. Une methode voulant accepter plusieurs combinaisons emploie des
surcharges ou un tableau explicite.

5.7 Parametre de catch

Syntaxe :

  catch ([final] TypeException nom) {
      instructions
  }

La machine virtuelle initialise ce parametre avec l'objet Throwable capture.
Sa portee est le bloc catch correspondant :

  try {
      lire();
  } catch (IOException erreur) {
      System.err.println(erreur.getMessage());
  }

Il ne peut etre lu hors de ce bloc. Son type determine les exceptions capturees
et doit etre Throwable ou une sous-classe. Un parametre final ne peut etre
remplace par une autre reference.

5.8 Composantes de tableaux

Chaque element d'un tableau se comporte comme une variable anonyme selectionnee
par un indice :

  int[] valeurs = new int[3];
  valeurs[0] = 10;
  valeurs[1] = valeurs[0] + 5;

Le type de valeurs[i] est le type element du tableau. L'expression tableau puis
l'indice sont evalues avant l'acces. Une reference null leve
NullPointerException; un indice hors de 0..length-1 leve
ArrayIndexOutOfBoundsException.

Les composantes recoivent automatiquement leur valeur par defaut a la creation
du tableau. Elles ne peuvent pas etre declarees individuellement final,
volatile ou transient. Une variable final contenant un tableau empeche de
remplacer la reference, mais pas de modifier ses elements :

  final int[] nombres = new int[3];
  nombres[0] = 42;              // valide
  nombres = new int[4];         // invalide

Pour un tableau covariant de references, une affectation incompatible leve
ArrayStoreException a l'execution.

5.9 Portee lexicale

La portee indique la region du source dans laquelle un nom simple designe une
variable.

  Type de variable       Debut de la portee            Fin
  ---------------------  -----------------------------  ------------------------
  champ                  corps de la classe            corps de la classe
  parametre de methode   declaration terminee          corps de la methode
  parametre constructeur declaration terminee          corps du constructeur
  parametre catch        apres sa declaration          fin du bloc catch
  locale de bloc         apres son declarateur         fin du bloc
  locale de for          apres son declarateur         instruction for incluse

Exemple :

  {
      int a = 1;
      {
          int b = a + 1;         // a et b visibles
      }
      // b n'est plus visible
  }

La variable declaree dans l'initialisation d'un for est visible dans la
condition, la mise a jour et le corps, mais pas apres la boucle :

  for (int i = 0; i < 10; i++) {
      System.out.println(i);
  }
  // i n'est pas visible ici

La portee n'est pas la duree de vie. Un objet reference par une locale peut
survivre au bloc s'il est stocke ailleurs; inversement, il peut devenir
inatteignable avant la fin lexicale de la portee.

5.10 Masquage et redeclaration des noms

Un parametre ou une variable locale peut masquer un champ :

  class Personne {
      private int age;

      Personne(int age) {
          this.age = age;
      }
  }

Dans le constructeur, age designe le parametre et this.age le champ.

Une classe derivee peut masquer un champ herite; Classe.nom ou super.nom permet
de preciser le champ souhaite selon le contexte. Un champ static peut aussi
etre masque par un champ static d'une sous-classe.

Le langage interdit de redeclarer une variable locale ou un parametre dans une
portee locale englobante qui se chevauche :

  void f(int x) {
      int x = 1;                 // invalide : parametre deja nomme x
      {
          int x = 2;             // invalide dans la portee de ce x
      }
  }

Deux blocs successifs sans chevauchement peuvent reutiliser un nom. Une locale
peut aussi masquer un champ, puisque les espaces de declaration concernes sont
differents.

5.11 Affectation certaine des variables locales

Le compilateur effectue une analyse de controle avant de permettre la lecture
d'une locale. Il faut que toutes les routes pouvant atteindre la lecture aient
affecte la variable :

  int resultat;
  if (condition)
      resultat = 1;
  else
      resultat = 2;
  System.out.println(resultat);  // valide

Exemple invalide :

  int resultat;
  if (condition)
      resultat = 1;
  System.out.println(resultat);  // condition peut etre false

Une boucle n'est generalement pas supposee s'executer au moins une fois, sauf
si le compilateur peut l'etablir avec les regles formelles. Une affectation
situee uniquement dans un while peut donc ne pas suffire apres la boucle.

L'operateur += lit l'ancienne valeur avant d'ecrire la nouvelle; la variable
doit deja etre initialisee :

  int n;
  n += 1;                        // invalide

Les champs ne sont pas soumis a cette lecture certaine ordinaire, car ils ont
une valeur par defaut. Les champs final suivent toutefois des regles
d'affectation certaine particulieres.

5.12 Modificateur final

final signifie qu'une variable ne peut etre affectee qu'une fois :

  final int maximum = 100;
  final String nom = "J++";

Une seconde affectation est une erreur de compilation :

  maximum = 200;                 // invalide

Pour une reference, final protege la variable et non l'objet :

  final Vector liste = new Vector();
  liste.addElement("A");         // valide : objet modifie
  liste = new Vector();          // invalide : reference remplacee

Une variable final peut etre initialisee dans son declarateur ou, dans certains
cas, plus tard lorsqu'elle est definitivement non affectee. On parle alors de
blank final.

  class Connexion {
      private final String serveur;

      Connexion(String serveur) {
          this.serveur = serveur;
      }
  }

Chaque constructeur doit affecter exactement une fois tout champ final
d'instance non initialise, directement ou par un autre constructeur. Un champ
static final vide doit etre affecte dans un initialisateur static.

  class Configuration {
      static final String RACINE;

      static {
          RACINE = chargerRacine();
      }
  }

final n'implique ni constante de compilation, ni immuabilite profonde, ni
synchronisation entre threads.

5.13 Constantes de compilation

Une variable constante au sens de la grammaire Java est une variable final de
type primitif ou String, initialisee par une expression constante :

  static final int TAILLE = 80;
  static final int DOUBLE = TAILLE * 2;
  static final String VERSION = "6." + "0";

Une expression constante n'utilise que des litteraux, certaines variables
constantes, des operateurs autorises et des conversions primitives ne demandant
aucun calcul a l'execution. Un appel de methode ou new exclut la constante :

  static final int A = 1 + 2;           // constante de compilation
  static final int B = Math.abs(-3);    // final, mais pas constante compilee
  static final Integer C = new Integer(3); // final, pas type primitif/String

Une constante de compilation peut etre utilisee dans case, dans certaines
initialisations reductrices et par le compilateur pour simplifier des
expressions.

Les clients peuvent incorporer directement la valeur d'un champ public static
final constant dans leur propre bytecode. Changer ensuite la constante sans
recompiler les clients peut leur laisser l'ancienne valeur. Pour une valeur
susceptible de changer entre versions, une methode d'acces peut etre preferable.

5.14 const n'est pas le mot-cle des constantes

const est reserve mais inutilise :

  const int MAX = 100;            // invalide en J++/Java

Il faut ecrire :

  final int max = 100;
  static final int MAX = 100;     // constante partagee par la classe

static indique l'appartenance a la classe; final interdit la reaffectation. La
combinaison static final est la forme habituelle d'une constante publique, mais
elle n'est une constante de compilation que si les conditions exposees dans la
section 5.13 sont satisfaites.

Les symboles de compilation conditionnelle de JVC ne sont pas des variables
const Java. Ils sont interpretes par les directives Microsoft avant ou pendant
la compilation; voir section 17.

5.15 Champs d'interface

Tout champ declare dans une interface est implicitement :

  public static final

Il doit posseder un initialiseur :

  interface Limites {
      int MINIMUM = 0;
      int MAXIMUM = 100;
  }

La declaration equivaut a :

  public static final int MINIMUM = 0;

Une classe qui implemente l'interface peut utiliser le nom si aucune ambiguite
n'existe, mais Limites.MINIMUM est plus clair. Un champ d'interface ne constitue
jamais un etat d'instance et ne peut etre modifie.

Si deux interfaces heritees fournissent le meme nom, un acces simple peut etre
ambigu; qualifier avec le nom de l'interface resout l'ambiguite.

5.16 Valeurs par defaut des champs

Avant tout initialiseur Java, la JVM attribue a chaque champ sa valeur par
defaut :

  boolean              false
  byte, short, int     0
  long                 0L
  char                 '\u0000'
  float                0.0F
  double               0.0D
  reference            null

Exemple :

  class Etat {
      int nombre;                // 0
      boolean actif;             // false
      String texte;              // null
  }

L'initialiseur explicite remplace ensuite cette valeur :

  int nombre = 10;
  String texte = "pret";

Il est legal mais rarement utile de recopier explicitement la valeur par
defaut. La valeur par defaut existe avant l'execution des constructeurs et peut
etre observee dans des situations d'initialisation mal concues; un constructeur
ne doit pas publier this avant d'avoir etabli ses invariants.

5.17 Ordre d'initialisation static

Une classe est initialisee lors de sa premiere utilisation active, par exemple
avant la creation de sa premiere instance, l'appel d'une methode static ou
l'acces a un champ static non constant. Avant cela, ses champs static ont deja
leur valeur par defaut.

Les initialisateurs de champs static et blocs static s'executent dans l'ordre
textuel :

  class Exemple {
      static int a = 1;

      static {
          a = a + 1;             // a vaut 2
          b = 5;                 // ecriture permise
      }

      static int b = 3;          // remplace ensuite 5 par 3
  }

La superclasse est initialisee avant la sous-classe. Une classe n'est
normalement initialisee qu'une fois par son chargeur de classes. Si
l'initialisation echoue par exception, l'utilisation ulterieure peut produire
ExceptionInInitializerError ou NoClassDefFoundError selon le moment.

Les references simples illegales vers un champ declare plus bas sont soumises
aux regles de reference avant declaration. Ecrire un champ peut etre permis la
ou le lire directement ne l'est pas. Il est preferable de declarer les champs
avant les blocs qui les utilisent.

5.18 Ordre d'initialisation d'une instance

Lors de new Classe(arguments), l'ordre conceptuel est :

  1. allocation de l'objet;
  2. mise a la valeur par defaut de tous ses champs;
  3. appel du constructeur de la superclasse;
  4. initialisateurs de champs d'instance, dans l'ordre textuel;
  5. blocs d'initialisation d'instance, dans l'ordre textuel;
  6. reste du corps du constructeur.

Exemple :

  class Exemple extends Base {
      int a = calculerA();

      {
          preparer();
      }

      Exemple() {
          terminer();
      }
  }

La premiere instruction d'un constructeur peut etre this(...) ou super(...).
Si elle est absente, super() est insere. Lorsqu'un constructeur appelle
this(...), l'initialisation d'instance est effectuee une seule fois par le
constructeur qui finit par appeler super(...), et non a chaque maillon.

Pendant l'execution du constructeur de la superclasse, les champs propres a la
sous-classe n'ont encore que leur valeur par defaut. Appeler une methode
redefinissable depuis un constructeur est donc dangereux : l'implementation de
la sous-classe peut observer un objet partiellement initialise.

5.19 Initialiseur de champ

Un initialiseur de champ est une expression :

  private int largeur = 80;
  private Vector elements = new Vector();
  private static String version = calculerVersion();

Un initialiseur d'instance peut utiliser this implicitement et appeler des
methodes d'instance, sous reserve des regles de reference avant declaration. Un
initialiseur static n'a pas de this et n'accede directement qu'au contexte
static.

L'expression doit etre compatible par affectation avec le type du champ. Elle
peut lever une exception non verifiee. Une exception verifiee dans un
initialiseur de champ est fortement contrainte : elle doit etre geree ou etre
compatible avec les constructeurs selon les regles du langage.

Creer un objet mutable dans un champ static le partage entre toutes les
instances. Creer l'objet dans un champ d'instance donne normalement une
instance distincte a chaque construction.

5.20 Blocs d'initialisation

Bloc d'instance :

  {
      instructions
  }

Bloc de classe :

  static {
      instructions
  }

Les blocs d'instance factorisent du code execute par tous les constructeurs.
Ils peuvent completer l'affectation de champs final d'instance sous les regles
d'affectation certaine. Ils ne peuvent contenir return.

Les blocs static initialisent les ressources partagees et les champs static
final vides. Ils ne peuvent utiliser this ou super comme reference d'instance.
Une exception verifiee ne peut pas simplement en sortir; elle doit etre
capturee. Une exception non verifiee rend l'initialisation de classe fautive.

Un bloc d'initialisation n'est pas une methode : il n'a ni nom, ni parametres,
ni type retour et ne peut etre appele explicitement.

5.21 transient

Syntaxe :

  transient Type champ;

transient ne s'applique qu'aux champs. Il indique au mecanisme standard de
serialisation que le champ ne fait pas partie de l'etat persistant ordinaire :

  class Session implements Serializable {
      String utilisateur;
      transient String motDePasse;
  }

Apres une deserialisation standard, motDePasse vaut null, sauf si la classe le
restaure par un mecanisme personnalise. transient ne masque pas le champ,
n'efface pas sa memoire et ne constitue pas une protection cryptographique.

Les champs static ne sont pas l'etat d'une instance serialisee; transient y est
donc sans utilite pratique meme si certaines grammaires anciennes acceptent la
combinaison.

5.22 volatile

Syntaxe :

  volatile Type champ;

volatile ne s'applique qu'aux champs. Il demande que les lectures et ecritures
passent par la semantique de memoire partagee de la JVM, au lieu de conserver
indefiniment une copie invisible aux autres threads :

  private volatile boolean arreter;

  public void demanderArret() {
      arreter = true;
  }

  public void executer() {
      while (!arreter) {
          travailler();
      }
  }

volatile ne rend pas atomique une suite lecture-calcul-ecriture :

  volatile int compteur;
  compteur++;                     // operation composee, pas atomique

Pour proteger un invariant ou une operation composee, utiliser synchronized.
Le modele memoire de Java 1.1/MSJVM est historique et moins precis que celui
formalise par Java 5; ne pas lui attribuer retroactivement toutes les garanties
modernes. Les anciennes JVM peuvent aussi traiter long/double volatile selon
leurs propres contraintes d'atomicite.

5.23 Visibilite des champs

  Modificateur  Acces principal
  ------------  --------------------------------------------------------------
  private       classe qui declare le champ
  aucun         classes du meme package
  protected     meme package et sous-classes selon les regles de receveur
  public        partout ou la classe englobante est accessible

La visibilite est controlee a la compilation. Elle ne change ni le type ni la
duree de vie du champ. Un champ public d'une classe non publique reste limite
par l'accessibilite de cette classe.

Les champs publics mutables exposent directement la representation interne :

  public int age;                 // possible, mais peu encapsule

La forme habituelle emploie un champ private et des methodes :

  private int age;
  public int getAge() { return age; }
  public void setAge(int valeur) { age = valeur; }

Comme explique a la section 15.3, l'IDE J++ peut presenter ces methodes comme
une propriete de composant; cela ne change pas les regles d'acces du langage.

5.24 Variables locales et classes internes

Java 1.1 autorise une classe locale ou anonyme a utiliser une variable locale
englobante seulement si cette variable est final :

  final String message = "Bonjour";

  Runnable action = new Runnable() {
      public void run() {
          System.out.println(message);
      }
  };

La variable capturee doit etre definitivement affectee avant la creation de
l'objet interne. Le compilateur peut copier sa valeur dans un champ cache de
la classe generee.

La notion moderne de variable "effectivement finale" n'existe pas encore :
omettre final rend la capture invalide en Java 1.1/J++ 6.0, meme si le source
ne reaffecte jamais la variable.

Les champs de l'instance englobante sont accessibles par la reference cachee et
n'ont pas besoin d'etre final. Une classe imbriquee static ne possede pas cette
reference d'instance englobante.

5.25 Duree de vie et ramasse-miettes

La duree de vie du stockage depend de la categorie :

  champ d'instance   aussi longtemps que l'objet est atteignable
  champ static       aussi longtemps que la classe reste chargee
  variable locale    pendant l'activation utile de son bloc/methode
  parametre          pendant l'appel
  element tableau    aussi longtemps que le tableau est atteignable

La fin de portee d'un nom ne detruit pas explicitement l'objet. Le
ramasse-miettes recupere les objets devenus inatteignables a un moment non
determine. Affecter null peut supprimer une reference, mais n'oblige pas une
collecte immediate :

  objet = null;
  System.gc();                    // simple demande, aucune garantie

Les ressources externes telles que fichiers, handles Win32 et objets COM ne
doivent pas dependre seulement du ramasse-miettes. Il faut les fermer ou les
liberer explicitement, notamment avec close() ou ComLib.release selon l'API.

5.26 Variables et interoperabilite Microsoft

Les paquets J/Direct et COM utilisent des variables Java pour representer des
valeurs natives :

  int hwnd;                       // handle Win32 dans la MSJVM 32 bits
  int[] resultat = new int[1];    // parametre de sortie simule
  Variant valeur;                 // valeur COM Automation
  IUnknown objetCom;              // interface COM

Un tableau a un element peut servir a simuler un pointeur de sortie, mais la
variable Java contient toujours une reference vers le tableau; le langage
n'acquiert pas un passage par reference.

Les declarations J/Direct determinent le marshalling, le sens des parametres et
la taille native. Un int convenable pour un handle dans Visual J++ 6.0/MSJVM
32 bits ne doit pas etre suppose portable vers un processus 64 bits.

Un objet COM stocke dans une variable reference peut demander une liberation
deterministe. Affecter null ne garantit pas que les references COM soient
relachees immediatement; voir com.ms.com.ComLib a la section 18.2.

5.27 Conventions de nommage

Les conventions n'ont aucun effet grammatical, mais ameliorent la lecture :

  compteur, nombreElements       variables et champs
  MAXIMUM, TAILLE_BLOC           constantes static final
  nomDeClasse.compteurGlobal     acces explicite a un champ static
  this.valeur                    distinction champ/parametre

Le prefixe hongrois lie au type, courant dans certains programmes Windows
anciens, est autorise mais non exige :

  int nCompteur;
  String strNom;

Comme J++ est sensible a la casse, MAXIMUM, Maximum et maximum sont trois noms
distincts. Eviter les noms commencant par $ destines aux outils et les noms qui
ressemblent aux mots contextuels Microsoft delegate et multicast.

5.28 Erreurs frequentes

  Erreur                               Explication
  -----------------------------------  -----------------------------------------
  lire une locale non initialisee      affectation certaine absente
  reaffecter une variable final        une seule affectation est permise
  modifier l'objet d'une final         permis : final protege la reference
  employer const                       mot reserve sans syntaxe executable
  acceder a l'instance depuis static   aucune reference this disponible
  croire un champ polymorphe           les champs sont resolus statiquement
  capturer une locale non final        interdit aux classes internes Java 1.1
  croire compteur++ atomique           volatile ne suffit pas
  attendre la liberation par x=null    collecte et finalisation non garanties
  confondre portee et duree de vie     notions distinctes

Exemple recapitulatif :

  public class Mesure {
      public static final int LIMITE = 100;
      private static int nombre;

      private final int identifiant;
      private volatile boolean active;
      private transient String cache;
      private int valeur;

      public Mesure(int valeur) {
          identifiant = ++nombre;
          this.valeur = valeur;
          active = true;
      }

      public int getValeur() {
          return valeur;
      }

      public void setValeur(int valeur) {
          if (valeur < 0 || valeur > LIMITE)
              throw new IllegalArgumentException("hors limites");
          this.valeur = valeur;
      }
  }

LIMITE est une constante de compilation partagee; nombre est un etat de
classe; identifiant est un blank final initialise par le constructeur; active
est un indicateur volatile; cache est exclu de la serialisation ordinaire; et
valeur est un champ d'instance encapsule par des methodes.


================================================================================
6. OPERATEURS ET EXPRESSIONS
================================================================================

  postfixes       expr++ expr-- () [] .
  unaires         ++expr --expr + - ~ ! (type)
  multiplicatifs  * / %
  additifs        + -                 (+ concatene aussi les String)
  decalages       << >> >>>
  relationnels    < <= > >= instanceof
  egalite         == !=
  bits/logique    & puis ^ puis |
  conditionnels   && puis ||
  ternaire        condition ? valeur1 : valeur2
  affectations    = *= /= %= += -= <<= >>= >>>= &= ^= |=

Les operateurs && et || sont a evaluation conditionnelle. == compare les
valeurs primitives, mais l'identite des references; equals compare en general
le contenu selon le contrat de la classe. La division entiere par zero leve
ArithmeticException. Les calculs entiers debordent sans exception.

L'evaluation des operandes se fait de gauche a droite. Les operateurs de meme
niveau sont generalement associatifs a gauche; le ternaire et les affectations
sont associatifs a droite.


================================================================================
7. INSTRUCTIONS
================================================================================

Bloc et instruction vide :

  { instruction1; instruction2; }
  ;

Condition :

  if (x >= 0) {
      signe = 1;
  } else {
      signe = -1;
  }

Selection :

  switch (code) {
      case 1:
          nom = "un";
          break;
      case 2:
          nom = "deux";
          break;
      default:
          nom = "autre";
  }

Les etiquettes case sont des constantes entieres compatibles avec char, byte,
short ou int. J++ historique n'accepte pas String dans switch.

Boucles :

  while (condition) { ... }
  do { ... } while (condition);
  for (int i = 0; i < 10; i++) { ... }

Il n'existe pas de boucle foreach dans le langage J++ historique.

Transferts :

  break;             break etiquette;
  continue;          continue etiquette;
  return;            return expression;
  throw expression;

Une etiquette precede une instruction :

  externe:
  for (int i = 0; i < n; i++) {
      for (int j = 0; j < m; j++)
          if (fini(i, j)) break externe;
  }


================================================================================
8. CLASSES ET OBJETS
================================================================================

Forme generale :

  [modificateurs] class Nom [extends SuperClasse]
      [implements Interface1, Interface2] {
      membres
  }

Exemple :

  public class Point {
      private int x;
      private int y;

      public Point(int x, int y) {
          this.x = x;
          this.y = y;
      }

      public int getX() { return x; }
      public void deplacer(int dx, int dy) { x += dx; y += dy; }
      public String toString() { return "(" + x + "," + y + ")"; }
  }

Une classe n'etend qu'une classe, mais peut implementer plusieurs interfaces.
Sans extends explicite, elle derive de Object. Les constructeurs portent le
nom de la classe et n'ont aucun type retour. La premiere instruction peut etre
this(...) ou super(...); sinon super() est insere implicitement.

Modificateurs principaux :

  public       visible partout ou le type est accessible
  protected    meme package et sous-classes
  private      classe declarante seulement
  static       membre de classe
  final        non redefinissable; une classe final n'est pas derivable
  abstract     implementation incomplete
  synchronized verrouillage associe a l'appel d'une methode
  native       implementation externe a la machine virtuelle
  transient    champ ignore par la serialisation standard
  volatile     acces direct partage entre threads selon le modele ancien

Les methodes acceptent la surcharge par liste de parametres. Le type retour
seul ne suffit pas a distinguer deux surcharges. Une methode d'instance peut
etre redefinie dans une sous-classe; une methode static est masquee, non
redefinie dynamiquement.

Initialisation : les champs static et blocs static s'executent a
l'initialisation de la classe. Les initialisateurs d'instance s'executent avec
les initialisations de champs lors de la construction.

  static { chargerConfiguration(); }
  { compteurInstances++; }


================================================================================
9. INTERFACES ET CLASSES INTERNES
================================================================================

  public interface Dessinable {
      void dessiner();
  }

Les methodes d'une interface sont implicitement public et abstract. Ses champs
sont implicitement public static final. Une interface peut etendre plusieurs
interfaces. Les methodes default et static d'interface n'existent pas encore.

Java 1.1, donc J++ 6.0, prend en charge les classes membres, locales et
anonymes :

  bouton.addActionListener(new ActionListener() {
      public void actionPerformed(ActionEvent e) {
          executer();
      }
  });

Une classe interne non static conserve une reference vers son instance
englobante. Une classe imbriquee static n'en conserve pas.


================================================================================
10. TABLEAUX
================================================================================

  int[] a = new int[5];
  int b[] = { 2, 3, 5, 7 };
  int[][] matrice = new int[2][3];
  int[][] irregulier = new int[2][];
  irregulier[0] = new int[3];
  irregulier[1] = new int[7];

Les indices valides vont de 0 a length - 1. Un indice invalide leve
ArrayIndexOutOfBoundsException. Les tableaux d'objets sont covariants : un
String[] est un Object[], mais une affectation incompatible par cette vue leve
ArrayStoreException.


================================================================================
11. EXCEPTIONS
================================================================================

  try {
      lire();
  } catch (IOException e) {
      System.err.println(e.getMessage());
  } finally {
      fermer();
  }

  void lire() throws IOException {
      if (erreur) throw new IOException("lecture impossible");
  }

Throwable est la racine. Error signale normalement une panne serieuse.
Exception regroupe les exceptions applicatives. RuntimeException et ses
sous-classes sont non verifiees; les autres exceptions doivent etre capturees
ou declarees par throws.

J++ ne possede ni multi-catch ni try-with-resources. Plusieurs catch sont
ecrits separement, du type le plus precis au plus general.


================================================================================
12. PAQUETS, IMPORTS ET VISIBILITE
================================================================================

  package societe.produit;
  import java.io.*;
  import java.util.Vector;

Un import n'inclut pas recursivement les sous-paquets. java.lang est importe
implicitement. Deux types de meme nom doivent etre distingues par leur nom
qualifie. Sans modificateur public/protected/private, un type ou membre a la
visibilite de package, parfois appelee default access.

Le CLASSPATH indique au compilateur et a la machine virtuelle ou chercher les
classes. Visual J++ pouvait aussi gerer son propre depot de classes et les
references du projet; un projet portable ne doit pas dependre implicitement de
ce depot.


================================================================================
13. ENTREES/SORTIES ET BIBLIOTHEQUE JAVA 1.1
================================================================================

Classes courantes disponibles selon la MSJVM installee :

  java.lang       Object, String, StringBuffer, Math, System, Thread
  java.io         InputStream, OutputStream, Reader, Writer, File
  java.util       Vector, Hashtable, Enumeration, Date, Random
  java.net        URL, Socket, ServerSocket
  java.awt        composants graphiques AWT et evenements
  java.applet     Applet

Exemple d'ecriture de fichier conforme a l'API de l'epoque :

  FileOutputStream f = new FileOutputStream("sortie.txt");
  PrintStream p = new PrintStream(f);
  try {
      p.println("Bonjour");
  } finally {
      p.close();
  }

Les collections historiques ne sont pas generiques. Iterator, ArrayList et le
Collections Framework appartiennent a Java 1.2 et ne doivent pas etre supposes
disponibles dans une cible J++ strictement Java 1.1.


================================================================================
14. THREADS ET SYNCHRONISATION
================================================================================

Un traitement concurrent derive de Thread ou implemente Runnable :

  class Travail implements Runnable {
      public void run() { effectuer(); }
  }
  new Thread(new Travail()).start();

Une methode synchronized verrouille l'objet receveur; une methode static
synchronized verrouille l'objet Class. Un bloc peut choisir son moniteur :

  synchronized (verrou) {
      compteur++;
  }

Object fournit wait(), notify() et notifyAll(). Ils doivent etre appeles par
un thread possedant le moniteur. Le modele memoire de la JVM Microsoft est
historique; le code nouveau ne doit pas lui attribuer retroactivement toutes
les garanties formalisees par les specifications Java modernes.


================================================================================
15. APPLETS, AWT ET WFC
================================================================================

Une applet Java classique derive de java.applet.Applet et utilise notamment
init(), start(), stop(), destroy() et paint(Graphics). Son execution depend
d'un navigateur ou d'un applet viewer compatible, aujourd'hui obsolete.

Visual J++ 6.0 introduisit les Windows Foundation Classes, bibliotheque
Microsoft de classes Windows. Les espaces de noms typiques commencent par :

  com.ms.wfc.app
  com.ms.wfc.core
  com.ms.wfc.ui
  com.ms.wfc.data

Une application WFC utilise en general une classe derivee de Form et appelle
Application.run(...). Les formulaires, controles, proprietes exposees par les
classes et le concepteur graphique relèvent de WFC, pas du noyau Java 1.1.
Un source qui importe com.ms.wfc.* est lie a Windows et a l'environnement
Microsoft.

15.1 Le paquet com.ms.ui

com.ms.ui appartient aux Application Foundation Classes (AFC), la bibliotheque
graphique Microsoft qui preceda les WFC. Il fournit un modele de composants,
de conteneurs et de gestionnaires de disposition comparable a AWT, avec des
controles legers (sans fenetre native propre) et des controles familiers aux
programmeurs Win32.

Les classes rencontrees dans les anciens programmes portent frequemment le
prefixe UI, par exemple UIPushButton ou UIRadioButton. Des sous-paquets
completent le noyau :

  com.ms.ui.event       gestion des evenements, notamment le modele Java 1.1
  com.ms.ui.resource    acces aux ressources Windows

Exemple schematique :

  import com.ms.ui.*;
  import com.ms.ui.event.*;

  public class FenetreAFC extends UIFrame {
      private UIPushButton bouton = new UIPushButton("Fermer");
      // construction et raccordement des evenements selon la version AFC
  }

Le nom exact des classes et de leurs constructeurs varie avec la version du
SDK. com.ms.ui est distinct de com.ms.wfc.ui : le premier designe les AFC des
premieres versions, le second la couche interface utilisateur des WFC de
Visual J++ 6.0. Une conversion ne doit pas remplacer aveuglement un import par
l'autre.

Une partie des AFC fut ecrite en Java, mais cela ne garantit pas la portabilite
de l'application complete. Certaines versions dependent d'extensions de la
MSJVM, de ressources Win32 ou de classes com.ms.* qu'une JVM standard ne
fournit pas. Les archives AFC appropriees doivent aussi etre distribuees quand
elles ne sont pas installees sur la machine cible.

15.2 Le paquet com.ms.awt

com.ms.awt contient l'implementation et les extensions Microsoft d'AWT. Il
fait l'interface entre les classes publiques java.awt et leurs pairs natifs
Windows. On y rencontre notamment des classes telles que WComponentPeer ainsi
que des implementations Microsoft de Toolkit, de composants ou de mesures de
polices, selon la version de la MSJVM.

Exemple rencontre dans du code fortement lie a Windows :

  import com.ms.awt.WComponentPeer;

Un programme AWT portable doit declarer ses variables avec les types java.awt
et ne jamais convertir un objet vers son implementation com.ms.awt. Ainsi, un
objet retourne comme java.awt.FontMetrics peut etre implemente par une classe
Microsoft interne, mais le programme ne doit pas dependre de cette classe.

com.ms.awt doit etre considere comme un paquet d'implementation :

  - il n'appartient pas a la specification Java 1.1;
  - ses classes peuvent changer entre versions de la MSJVM;
  - il n'est normalement pas present dans les JDK de Sun/Oracle ou OpenJDK;
  - son utilisation directe signale une dependance Windows/MSJVM a migrer.

Selon le besoin, une migration remplace cet acces par une API publique AWT,
Swing ou JavaFX, ou isole l'operation native derriere une couche specifique a
la plate-forme.

15.3 Proprietes : J++, JavaBeans, WFC et comparaison avec C#

Le mot "propriete" est frequent dans la documentation et dans l'interface de
Visual J++, mais il faut distinguer le concept expose par les composants de la
syntaxe effectivement acceptee par le langage.

Visual J++ sait afficher et manipuler des proprietes de composants JavaBeans,
AFC, WFC ou COM dans la fenetre Properties de l'IDE. Pour un JavaBean, une
propriete Age est normalement reconnue par convention lorsque la classe offre
les methodes publiques suivantes :

  public class Personne {
      private int age;

      public int getAge() {
          return age;
      }

      public void setAge(int valeur) {
          age = valeur;
      }
  }

Le code client Java ou J++ appelle ces methodes explicitement :

  Personne p = new Personne();
  p.setAge(42);
  int n = p.getAge();

Pour une propriete booleenne, le getter peut suivre la convention is... :

  public boolean isActif() { return actif; }
  public void setActif(boolean valeur) { actif = valeur; }

Les outils d'introspection JavaBeans deduisent le nom et le type de la
propriete a partir de getAge/setAge ou isActif/setActif. Une classe BeanInfo
peut fournir explicitement les memes metadonnees. Le fait que l'IDE presente
Age comme une propriete editable ne transforme pas ces methodes en une nouvelle
construction syntaxique Java.

Les classes WFC et les proxies COM presentent eux aussi des membres comme des
proprietes. Suivant la bibliotheque ou le generateur de proxy, le source J++
utilise des methodes get..., set..., get_..., set_... ou une methode de
dispatch prenant un nom et un Variant. Cette apparence uniforme dans le
concepteur ne signifie pas que J++ possede les accesseurs get/set de C#.

La declaration suivante ressemble a C#, mais n'est pas une syntaxe valide de
Visual J++ 6.0 :

  public int Age
  {
      get;
      set;
  }

Il s'agit plus precisement d'une propriete automatique de C# : le compilateur
C# cree le stockage et les accesseurs. Cette forme abregee apparut apres
Visual J++. Une propriete C# classique avec logique explicite s'ecrit par
exemple :

  private int age;

  public int Age
  {
      get { return age; }
      set { age = value; }
  }

Le code C# peut ensuite employer une syntaxe de champ tout en invoquant les
accesseurs :

  p.Age = 42;
  int n = p.Age;

Java ne possede toujours pas de declaration de propriete integree equivalente.
Dans du Java standard, il faut conserver les methodes getAge() et setAge(), ou
utiliser un outil externe qui genere ce code. Les records Java modernes
fournissent des composants accessibles par des methodes telles que age(), mais
ne creent pas une propriete mutable de style C#.

On peut donc dire que l'environnement J++ avait un modele de proprietes tres
visible et familier aux utilisateurs des outils Microsoft. Il serait toutefois
incorrect d'attribuer au langage Visual J++ la grammaire get/set qui sera celle
de C#. Les ressemblances conceptuelles viennent surtout du modele composant
Microsoft, de WFC/COM et de l'outillage Visual Studio.


================================================================================
16. DELEGATES ET EVENEMENTS MICROSOFT
================================================================================

Visual J++ fournit une extension de langage pour representer un appel de
methode type : le delegate. Un delegate associe une signature a une ou
plusieurs methodes compatibles et sert principalement au modele evenementiel
WFC. Sa forme usuelle est :

  delegate void Gestionnaire(Object source, Event e);

  Gestionnaire g = new Gestionnaire(objet.methode);
  g.invoke(source, evenement);

La declaration Microsoft "multicast delegate" cree une variante pouvant
contenir plusieurs destinataires. L'ajout, le retrait et le declenchement
precis dependent des classes et conventions du
runtime Microsoft utilisees par le projet. Les outils de conception WFC
generent normalement le raccordement des gestionnaires.

Cette syntaxe n'appartient pas a Java 1.1 et n'est pas acceptee par javac. Lors
d'une migration vers Java, on la remplace par les interfaces listener et les
classes anonymes; vers .NET, par les delegates et events C# ou J#.

ATTENTION : event, propriete graphique et delegate sont souvent presentes
ensemble dans l'IDE, mais tous les membres affiches comme "properties" ou
"events" par le concepteur ne correspondent pas necessairement a un nouveau
mot-cle J++. Ils peuvent etre des conventions de bibliotheque WFC.


================================================================================
17. DIRECTIVES MICROSOFT ET COMPILATION CONDITIONNELLE
================================================================================

J++ emploie des commentaires de documentation speciaux compris par JVC. Leur
aspect de commentaire permet a un compilateur Java standard de les ignorer,
mais leur effet est propre aux outils Microsoft.

Forme generale :

  /** @directive(parametres) */

La famille J/Direct permet de declarer l'import d'une fonction Win32 contenue
dans une DLL. Exemple schematique (la signature exacte depend de l'API) :

  /** @dll.import("USER32") */
  static native int MessageBox(int hwnd, String texte,
                               String titre, int type);

Ces declarations sont native, liees a Windows et a la MSJVM. Les conventions
d'appel, conversions de chaines, structures et pointeurs doivent etre verifiees
dans la documentation J/Direct correspondant a la version du SDK.

JVC prend egalement en charge des directives conditionnelles sous forme de
commentaires, notamment les familles @if, @else et @end. Elles permettent au
compilateur Microsoft d'inclure ou d'exclure du source selon des symboles de
compilation. Ce mecanisme n'est pas le preprocesseur C et n'est pas portable.

D'autres attributs en commentaire furent utilises pour COM, securite,
marshalling ou generation de metadonnees. Une directive inconnue ne doit pas
etre inventee : conserver son texte et consulter le SDK Microsoft d'origine.


================================================================================
18. INTEROPERABILITE COM ET WIN32
================================================================================

Visual J++ pouvait appeler :

  - des fonctions de DLL Windows par J/Direct;
  - des composants Automation/COM;
  - des bibliotheques de types converties en classes Java;
  - des objets ActiveX par les outils Java Type Library Wizard ou JActiveX.

JActiveX genere des classes proxy a partir d'une bibliotheque de types. Le
programme importe ensuite ces classes comme toute classe Java. Le code produit
reste cependant dependant de COM, de Windows, des GUID et du runtime Microsoft.

Le mot-cle native de Java indique seulement qu'une methode est implementee
hors Java. Il ne definit ni J/Direct ni COM. Ceux-ci appartiennent a la couche
Microsoft et doivent etre documentes separement dans une application.

18.1 Le paquet com.ms.win32

com.ms.win32 expose sous forme de classes Java une grande partie de l'API
Win32 deja declaree pour J/Direct. Il evite de recopier une declaration
@dll.import pour chaque fonction. Les classes principales correspondent aux
DLL et familles d'API Windows, par exemple :

  com.ms.win32.User32       fenetres, messages, clavier et interface utilisateur
  com.ms.win32.Gdi32        dessin et objets GDI
  com.ms.win32.Kernel32     processus, threads, fichiers et memoire
  com.ms.win32.Ole32        initialisation et activation OLE/COM
  com.ms.win32.Shell32      fonctions du shell Windows

Le nom Kernel ou Kernel32 et la couverture exacte peuvent varier selon la
version du SDK Microsoft. Toutes les fonctions Win32 ne sont pas necessairement
declarees.

Exemple :

  import com.ms.win32.*;

  public class MessageNatif {
      public static void main(String[] args) {
          User32.MessageBox(0, "Bonjour", "J++",
                            winm.MB_OK | winm.MB_ICONINFORMATION);
      }
  }

Les constantes Win32 sont regroupees dans des interfaces dont le nom commence
par win et se termine souvent par la premiere lettre de la constante. Par
exemple, plusieurs constantes MB_* se trouvent dans winm et SW_* dans wins.
Les structures C sont representees par des classes Java du paquet.

Le marshalling J/Direct convertit les types Java en representations natives.
Les points suivants doivent etre controles avant tout appel :

  - version ANSI ou Unicode de la fonction;
  - taille 32 bits des handles et pointeurs dans la MSJVM historique;
  - sens entree/sortie des tableaux utilises pour simuler un passage par
    reference;
  - duree de vie des buffers, structures et callbacks;
  - valeur retournee et GetLastError quand l'API l'exige.

Les callbacks Win32 peuvent utiliser les classes de rappel fournies par le SDK,
souvent en relation avec com.ms.dll.Callback. La signature doit correspondre
exactement au prototype natif. Une application utilisant com.ms.win32 ne peut
pas s'executer telle quelle sur une JVM standard ou sur une autre plate-forme.

18.2 Le paquet com.ms.com

com.ms.com fournit l'infrastructure Java/COM de Microsoft. Il est utilise avec
les classes proxy generees par JActiveX ou par le Java Type Library Wizard. Il
contient notamment, selon le SDK :

  IUnknown       representation de l'interface COM fondamentale
  Variant        valeur Automation pouvant contenir plusieurs types
  ComLib         operations utilitaires, dont la liberation explicite
  GUID et types  identifiants, interfaces et structures necessaires a COM

Exemple schematique avec un proxy genere :

  import com.ms.com.*;
  import excel8.*;                 // paquet produit depuis une type library

  public class Automatisation {
      public void executer() {
          _Application app = null;
          try {
              app = (_Application)new Application();
              app.setVisible(0, true);
          } finally {
              if (app != null)
                  ComLib.release(app);
          }
      }
  }

Les signatures reelles dependent entierement du proxy genere et de la version
de la bibliotheque de types. Les parametres optionnels sont souvent representes
par Variant; les proprietes COM deviennent en general des methodes get... et
set..., et les HRESULT sont convertis par la couche d'interoperabilite.

La MSJVM gere une partie du comptage de references COM, mais le ramasse-miettes
n'offre pas une liberation deterministe. ComLib.release permet de relacher un
objet au moment voulu, ce qui est important pour les serveurs COM lies a un
thread, les applications Office et les ressources natives couteuses. Il ne
faut plus utiliser un proxy apres l'avoir libere.

Les contraintes COM usuelles restent applicables : initialisation de
l'appartement du thread, enregistrement des CLSID/ProgID, disponibilite de la
bibliotheque de types, correspondance 32 bits et propagation des erreurs. Le
paquet com.ms.com est propre au runtime Microsoft et n'est pas equivalent a
une API Java standard.

18.3 Relations entre les quatre paquets

Ces paquets peuvent apparaitre ensemble mais remplissent des fonctions
distinctes :

  com.ms.win32   appels directs des fonctions et types de l'API Win32
  com.ms.com     objets COM/Automation et classes proxy de type libraries
  com.ms.ui      composants graphiques des anciennes AFC
  com.ms.awt     implementation Microsoft et pairs Windows de java.awt

Un formulaire com.ms.ui peut reposer indirectement sur com.ms.awt et Win32;
un composant ActiveX place dans une interface peut combiner com.ms.com et
com.ms.win32. Ces dependances indirectes expliquent pourquoi la simple
suppression des quatre lignes import ne rend pas un programme portable.

19. COMPILATION ET EXECUTION
================================================================================

Outils historiques typiques :

  jvc Bonjour.java             compilation avec le compilateur Microsoft
  jview Bonjour                execution console sur la MSJVM

Les options exactes varient selon Visual J++ 1.x, 6.0 et le SDK Microsoft Java.
L'IDE stocke aussi des options dans les fichiers de projet. Un diagnostic de
migration doit relever au minimum :

  - version de JVC et de la MSJVM;
  - options activant ou interdisant les extensions Microsoft;
  - CLASSPATH et references du projet;
  - imports com.ms.*;
  - directives /** @... */;
  - bibliotheques COM et DLL externes;
  - utilisation de WFC, AWT ou applets.

Un fichier .class produit pour la MSJVM n'est pas une preuve de portabilite :
il peut referencer des classes Microsoft ou des mecanismes non disponibles sur
une JVM standard.

20. GRAMMAIRE SYNTHETIQUE DU NOYAU
================================================================================

La notation suivante est volontairement resumee. [] signifie optionnel, {}
zero ou plusieurs repetitions et | une alternative.

  classDecl ::= { modClasse } class Ident [ extends type ]
                [ implements type { , type } ] corpsClasse

  interfaceDecl ::= { modInterface } interface Ident
                    [ extends type { , type } ] corpsInterface

  membre ::= champ | methode | constructeur | classeInterne |
             interfaceInterne | initialisateur | ;

  champ ::= { modChamp } type variables ;

  methode ::= { modMethode } ( type | void ) Ident
              ( [ parametres ] ) [ throws types ]
              ( bloc | ; )

  constructeur ::= { modConstructeur } Ident
                   ( [ parametres ] ) [ throws types ] bloc

  instruction ::= bloc | ; | expression ; | declarationLocale ; |
                  if (...) instruction [ else instruction ] |
                  switch (...) { ... } |
                  while (...) instruction |
                  do instruction while (...) ; |
                  for ( ... ; ... ; ... ) instruction |
                  break [ Ident ] ; | continue [ Ident ] ; |
                  return [ expression ] ; | throw expression ; |
                  synchronized ( expression ) bloc |
                  try bloc ( catch... [ finally bloc ] | finally bloc )

  primaire ::= litteral | this | super | Ident | ( expression ) |
               new creation | primaire . Ident | primaire [ expression ] |
               primaire ( [ arguments ] )

La grammaire normative complete du noyau est celle de Java 1.1. Les sections
16 et 17 ajoutent les extensions propres au compilateur Microsoft.

Construction moderne                 Equivalent ou solution J++ historique
-----------------------------------  ------------------------------------------
List<String>                         Vector contenant Object + casts
for (Type x : collection)            Enumeration ou boucle indexee
enum Couleur                         constantes int ou classe finale
@Annotation                          conventions/directives propres aux outils
lambda                               interface + classe anonyme, ou delegate MS

================================================================================
22. PORTABILITE ET MIGRATION
================================================================================

Trois categories facilitent l'audit d'un projet :

  A. Java 1.1 portable
     java.lang, java.io, java.util historique, AWT, syntaxe standard.

  B. API Microsoft remplacable
     com.ms.wfc.*, utilitaires com.ms.*, modele evenementiel WFC.

  C. liaison native
     @dll.import, COM/ActiveX, bibliotheques de types, appels Win32.

Migration vers Java : conserver A, remplacer B par AWT/Swing/JavaFX ou une
autre bibliotheque, et remplacer C par JNI/JNA/FFM ou un service externe. Les
delegates deviennent en general des listener interfaces.

Migration vers .NET : convertir la syntaxe vers C# (ou historiquement J#),
remplacer WFC par Windows Forms/WPF et convertir J/Direct vers P/Invoke. Les
details de marshalling doivent etre testes, surtout pour char, boolean, tableaux
et chaines.

La migration doit aussi traiter les ecarts d'encodage, le modele de securite
des applets, les dependances 32 bits, l'enregistrement COM et les API Windows
disparues. Une recompilation reussie ne suffit pas : des tests fonctionnels
sont indispensables.


================================================================================
23. EXEMPLE RECAPITULATIF PORTABLE
================================================================================

  import java.util.Vector;
  import java.util.Enumeration;

  public class Catalogue {
      private Vector articles = new Vector();

      public synchronized void ajouter(String nom) {
          if (nom == null)
              throw new IllegalArgumentException("nom nul");
          articles.addElement(nom);
      }

      public void afficher() {
          Enumeration e = articles.elements();
          while (e.hasMoreElements()) {
              String nom = (String)e.nextElement();
              System.out.println(nom);
          }
      }

      public static void main(String[] args) {
          Catalogue c = new Catalogue();
          c.ajouter("clavier");
          c.ajouter("souris");
          c.afficher();
      }
  }

Cet exemple evite les extensions Microsoft et respecte le style Java 1.1 :
Vector sans generics, Enumeration sans foreach et casts explicites.
